# Tour

A code-review tool that pairs an ephemeral, GitHub-style diff (split or unified) with persisted AI annotations. Drives the same data from a TUI and a webapp, both consuming local files written by agents through a CLI.

## Language

**Tour**:
A single guided traversal of a pinned git diff, with zero or more annotations attached. Lives in `.tour/<id>/`.
_Avoid_: review, PR, pull request, code review session, changeset

**Diff**:
The set of file changes shown in a Tour, recomputed from git on every open. Never persisted.
_Avoid_: patch, changes, delta

**Head**:
The git ref the Tour's diff ends at. Resolved to a SHA at create time. May be a real commit or a synthetic snapshot of the working tree.
_Avoid_: tip, target

**Base**:
The git ref the Tour's diff starts from. Resolved to a SHA at create time. Defaults to `head^` for single-commit tours.
_Avoid_: parent, ancestor

**Annotation**:
A note anchored to `(file, side, line_start, line_end)` inside a Tour's diff. `side` is `additions` or `deletions` (matching Pierre's vocabulary); line numbers are file-line-numbers in the file as it exists on that side at the pinned SHA. Authored by agents (via CLI) or humans (via UI or CLI). Persisted in the Tour's folder. The body is **GitHub Flavored Markdown** (no raw HTML); the webapp renders it as rich markdown — including ` ```mermaid ` fences as diagrams — while the TUI shows the raw source. No `kind` enum — structure lives in the markdown itself. Carries an `author_kind ∈ {agent, human}` so the system can distinguish the two without parsing the free-form `author` display string.
_Avoid_: comment, review comment, note

**Reply**:
An Annotation whose `replies_to` field points at another Annotation in the same Tour. Inherits the parent's `(file, side, line_start, line_end)` anchor — replies don't have independent positions. Renders inside the parent's card as part of a Thread.
_Avoid_: comment-on-comment, sub-annotation, response

**Thread**:
A top-level Annotation plus the chain of Replies attached to it. Lives entirely inside one Tour — Threads do not carry across Tours in v1 (preserves ADR 0003's "no re-anchoring" stance). When a Tour closes, its Threads close with it. Continuity across Tours is the agent's job (it re-references prior concerns in prose when reviewing the new SHA), not Tour's data model.
_Avoid_: conversation, discussion, exchange

**Reply-agent**:
An ephemeral, capability-bounded agent process spawned by the renderer's watcher whenever a new Annotation with `author_kind = "human"` lands. The agent has **no tools** — it runs in non-interactive print mode and emits the reply body to stdout; Tour captures stdout and writes the Reply Annotation itself. The capability boundary is "no tools," strictly stronger than the prior single-allowed-tool design: the agent literally cannot read, write, edit, or shell out. A Tour-canonical system prompt instructs the model to emit only the reply body — no preamble, no narration. Distinct from the **main-agent** — the user's existing Claude Code / Codex / etc. session — which is unrestricted and consumes the conversation later via `tour pickup`.
_Avoid_: bot, listener, autoreplier

**Side**:
The diff-half an Annotation belongs to: `deletions` for the base-file half, `additions` for the head-file half. Required, not derived. Layout-independent: in `split` layout, `deletions` renders on the left column and `additions` on the right; in `unified` layout, `deletions` attaches to the `-` row and `additions` to the `+` row. Annotations on unchanged context lines pick `additions` by convention. Naming matches Pierre's `AnnotationSide` so `core/` ↔ Pierre is a no-op.
_Avoid_: LEFT/RIGHT, before/after, old/new, column

**Layout**:
How a Tour's Diff is rendered: `split` (deletions left, additions right) or `unified` (single column with `-`/`+` rows). Per-session UI state in each surface — not persisted on the Tour, not synced between TUI and webapp. Annotations are layout-independent: the same `(file, side, line_start, line_end)` anchor resolves in both. Maps to Pierre's `diffStyle` and OpenTUI's `view` props (both already accept `"unified" | "split"`).
_Avoid_: view mode, diff style, view

**Working-tree snapshot**:
A synthetic commit object capturing uncommitted changes at the moment a Tour is created, so the Diff stays pinned even as the working tree keeps moving.
_Avoid_: stash, WIP commit

**Line cursor** (TUI only):
The TUI's persistent diff-pane position used as the anchor for new top-level Annotations. Walks the full concatenated diff stream across files (cross-file scope, mirroring how the diff pane scrolls), lands only on rendered diff lines, and carries an explicit `preferredSide` so split layout's two columns are both addressable. Tracked separately from the **annotation cursor** (`currentAnnotationId`); navigating annotations with `n`/`p` moves both, but moving the line cursor with `j`/`k` does not move the annotation cursor — line motion is for code reading, annotation motion is for engaging with conversation. Webapp has no equivalent — it relies on click-to-position via Pierre's `FileDiff`. See ADR 0011.
_Avoid_: caret, point, position, selection

## Relationships

- A **Tour** has exactly one **Head** and one **Base**, both stored as SHAs.
- A **Tour** has zero or more **Annotations**.
- An **Annotation** belongs to exactly one **Tour** and anchors to one file + line-range inside that Tour's **Diff**.
- A **Reply** is an Annotation that points at another Annotation in the same Tour via `replies_to`.
- A **Thread** is a top-level Annotation plus its chain of Replies; lives entirely inside one Tour.
- A **Working-tree snapshot** acts as a synthetic **Head** when an agent creates a Tour of uncommitted work.

## Example dialogue

> **Dev:** "If the agent amends the commit after creating a **Tour**, do the **Annotations** still line up?"
> **User:** "The **Tour** is pinned to the original SHA. Amending creates a new SHA — the old one is still in git's object store, so the **Diff** and **Annotations** still resolve. If the agent wants to tour the amended version, they create a new **Tour**."

## Resolved decisions

- **Annotation lifetime**: per-tour-pass. Annotations are not re-anchored across rebases or amendments. Stale Tours are abandoned, not migrated.
- **Diff source**: commit-pinned only. Working-tree tours are supported by snapshotting to a synthetic commit at create time, so the rest of the system only ever sees SHAs.
- **Multiplicity**: many Tours coexist, ordered by creation time. Default open behavior surfaces the most recent unfinished one.
- **Tool surface**: CLI-first. Agents drive the system by shelling out (`tour create`, `tour annotate`, …) with `--json` output. MCP is deferred and would be a thin wrapper if added later.
- **Annotation anchor**: `(file, side, line_start, line_end)` with `side ∈ {additions, deletions}` (matching Pierre's `AnnotationSide`) and line numbers being file-line-numbers at the pinned SHA. View-mode independent; both split and unified consume the same shape. No `kind` enum in v1 — the body is plain text.
- **Authoring**: both agents and humans author Annotations. Top-level Annotations and Replies use the same primitive (`tour annotate`, optionally with `--reply-to <ann-id>`). Each Annotation carries `author_kind ∈ {agent, human}`; the free-form `author` string is kept for display. Both surfaces (TUI + webapp) gain a composer for top-level notes and replies. The CLI is canonical and remains the agent path; the UI is the human path. Originally v1 said "agents only, UI is read-only"; that was reversed once Threads landed (see ADR 0010). Resolution state is **not** modeled — readers (the main-agent at `tour pickup` time, or another human) infer outcomes from natural language. No `agree:`/`dismiss:` prefix conventions, no status enum.
- **Sidebar (tree)**: folder tree of changed files. Path compression folds any non-leaf node with exactly one folder child into its child (lazygit-style), so sparse tours don't waste rows on empty intermediate folders. Folders sort before files at each level; alphabetical within each group. File rows render an Octicon-based file-status icon (added/modified/deleted/renamed, sourced from `@primer/octicons-react`'s diff icon set via the `web/client/icons.ts` shim and the `web/client/file-icon.ts` `type → { Icon, statusClass }` mapping) and a per-file annotation count badge. Folder rows render a chevron Octicon (`ChevronRightIcon` when collapsed, `ChevronDownIcon` when expanded) followed by `FileDirectoryFillIcon` and the compressed path — GitHub-PR-tree-shaped, with one folder icon in both states (the chevron carries the open/closed signal). Per-depth indent is `16px` for both row kinds, giving the chevron + folder-icon group enough horizontal breathing room. The TUI sidebar mirrors this noise-reduction (no folder rollup, no inline classification-reason tag) and keeps its single-letter status glyph and per-file count. The classification reason still surfaces in the main-pane file header on both surfaces — only the sidebar duplicate is removed. The pure shared module `core/file-tree.ts` is unchanged: `flatten` keeps producing `VisibleRow[]` with `annotationCount` on every row; the surfaces simply ignore it on folder rows. Default state on first render: every folder expanded. Clicking a folder toggles its expand/collapse; clicking a file selects it and scrolls the diff stream to it. `n`/`p` annotation navigation reveals (expands) the ancestor folder chain of the target file. Expand/collapse state is React state, in-memory only, per renderer instance, per tour — page reload resets to "all expanded" (no localStorage, no on-disk state). The tree algorithm lives in the pure shared module `core/file-tree.ts`; both the webapp and TUI consume `VisibleRow[]` from `flatten`. Originally deferred from v1 ("Real folder-tree rendering is deferred until file counts justify it") to ship the smallest viable Tour; reinstated now that the v1 surface is shipped and real tours expose the flat list's limitations on deeply-nested paths. Main pane is unchanged: a single top-to-bottom stream of all file diffs; clicking a sidebar file entry scrolls to that file.
- **Watch behavior**: TUI and webapp watch the open Tour's `.tour/<id>/` folder for changes and re-render. The watcher fires on `*.jsonl` (annotation log changes) and `.reply-lock.json` (in-flight reply state, see Reply-agent dispatch below). They do not watch git refs or working-tree files — the diff is pinned to `head_sha`.

- **Reply-agent dispatch**: when the watcher detects a new Annotation with `author_kind = "human"` AND the renderer was launched with `--reply-agent <name>`, it spawns the configured adapter from `~/.config/tour/agents/<name>.sh` with the triggering Annotation's thread context as JSON on stdin and `TOUR_*` vars in env. Agent invocation is single-flight per Tour, gated by `.tour/<id>/.reply-lock.json` (which doubles as the renderer's status-pill source). System prompt is owned by Tour and exposed via `tour reply-system-prompt`; adapters concatenate it with their agent's framing. Capability-bounding to `tour annotate --as-agent --reply-to` is enforced by each adapter through its agent's native allow-list (claude `--allowedTools`, codex sandbox, etc.). MVP ships adapters for claude / codex / gemini / opencode / pi (Copilot CLI cut from MVP — its capability-bounding surface isn't designed for single-tool-only enforcement; see ADR 0010); users supply their own at the same path for unsupported agents. Without `--reply-agent` the watcher is inert and Tour functions as a static review tool — humans can still author, the conversation just sits until consumed via `tour pickup`. See ADR 0010.

- **Reply streaming**: while a reply-agent is in flight, the renderer paints a status pill below the targeted Annotation ("✏️ `<agent>` is replying… (Ns)") sourced from `.reply-lock.json`. When the agent's `tour annotate --reply-to` write lands and the lock releases, the pill is replaced atomically by the rendered Reply. Token-by-token streaming was rejected in favour of this lockfile-based status because it composes with the single-flight lock we already need (zero new IPC channel) and captures the actual user need ("is the agent on it?") for ~5% of the implementation cost. Stale locks (>2min) flip the pill to a warning and offer `tour reply-cancel <ann-id>` to recover.

- **Conversation pickup**: `tour pickup <id> --json` emits the full conversation tree as JSON for consumption by the user's main-agent session. The schema is `annotations.jsonl` reshaped into a tree (each top-level Annotation carries its `replies` array). No status field, no actionable-only filter, no markdown digest in v1 — Tour emits raw conversation; the main-agent does all reasoning. Triggered by the human (or asked of the main-agent) when they want code changes acted on; Tour itself never moves the SHA mid-conversation.
- **Handoff**: agent prints the tour ID to stdout (and a one-line "Open with: …" hint). No auto-open. Humans open the UI explicitly via `tour` (TUI) or `tour serve` (webapp). `tour create --open` opt-in for users who do want auto-launch.
- **Tour ID**: `YYYY-MM-DD-HHMMSS-xxxx` (UTC timestamp + 4-char random suffix). Sortable by creation time; collision-safe under parallel creates; prefix-matchable (`tour show 2026-05-08-1`).
- **CLI surface**: `tour create --head <ref> [--base <ref>] [--title <s>]` resolves both ends to SHAs at create time. `--head WIP` snapshots uncommitted changes. `--base` default is context-sensitive: `<head>^` for commit tours, `HEAD` for `WIP` tours. Verbs: `create`, `annotate`, `list`, `show`, `close`, `delete`, `prune`, `tui`, `serve`, `pickup`, `reply-cancel`, `reply-system-prompt`. `tour annotate` accepts `--reply-to <ann-id>` for replies and an `--as-agent` / `--as-human` pair that sets `author_kind` (default: `agent`). `tour tui` and `tour serve` accept `--reply-agent <name>` to enable live agent replies; without it the watcher is inert. Bare `tour` opens TUI on the most recent open Tour.
- **On-disk schema**: `tour.toml` holds `id`, `title?`, `status`, `created_at`, `closed_at?`, `head_sha`, `base_sha`, `head_source`, `base_source`, `wip_snapshot`. `annotations.jsonl` is one annotation per line: `{id, file, side, line_start, line_end, body, author, author_kind, created_at, replies_to?}`. `author` is a free-form display string (e.g. `"agent"`, `"claude-code"`, `"almas"`); `author_kind` is the structured discriminator (`"agent"` | `"human"`) used by the reply-agent dispatch predicate, required on every Annotation. `replies_to` (optional) carries the bare `id` of the parent Annotation when this entry is a Reply. Pre-bidirectional `.tour/` data is not migrated — Tour isn't in production yet, so the schema break is taken cleanly rather than papered over with reader-side defaults. `.reply-lock.json` (transient) holds `{agent, responding_to, started_at, pid}` while a reply-agent is in flight.
- **Webapp lifecycle**: `tour serve` is long-running, single-user, binds to `127.0.0.1:7777` (configurable), no auth. `Ctrl-C` to stop. Browser auto-open only with `--open`.
- **Snapshot loss**: if a Tour's `head_sha` no longer resolves (e.g. user manually ran `git update-ref -d refs/tour/<id>`), the UI shows a banner "Snapshot lost — annotations preserved but diff cannot be displayed." Read-only, can be deleted.
- **Packaging**: single binary, multi-subcommand (same shape as Backlog.md). One `tour` package containing a shared `core/` (git, tour IO, annotations, watcher) and two renderers (`tui/`, `web/`). Stack: Bun + TypeScript + OpenTUI for TUI; Bun HTTP server + React for web.
- **Distribution**: npm package is `tourdiff`; installed binary is `tour`. Per-platform sub-packages `tourdiff-darwin-arm64`, `tourdiff-darwin-x64`, `tourdiff-linux-arm64`, `tourdiff-linux-x64`, `tourdiff-windows-x64` are pulled in via pinned `optionalDependencies` (only the matching one installs on a given OS/arch). The parent ships a `scripts/cli.cjs` shim that resolves the right binary at runtime — exact mirror of `backlog.md`'s install model. GitHub Releases also attach the same 5 binaries as a second distribution path. No telemetry, no update-check, no outbound network calls. Release pipeline is a GitHub Actions workflow on `vX.Y.Z` tag push: matrix-builds 5 binaries with `bun build --production --compile`, smoke-tests `./tour --version` per-platform, then publishes to npm with `--provenance` (granular token, publish-only, scoped to `tourdiff` + `tourdiff-*`). Pre-1.0 semver: minor=breaking, patch=compat. Repo `github.com/a9a4k/tour`, license MIT.
- **Diff engine (v1)**: `@pierre/diffs` parses git diffs in both surfaces. Webapp renders via Pierre's `FileDiff` (uses its built-in annotation framework). TUI walks Pierre's parsed `FileDiffMetadata` row-by-row through a Tour-owned row planner in `core/` (see ADR 0007); `@opentui/core`'s `<diff>` and `hunkdiff/opentui`'s `HunkDiffView` were both rejected because neither exposes a slot for inline annotation content between rows; `anomalyco/opentui#621` (`SlottableDiffRenderable`) does, but was rejected to stay on stock OpenTUI rather than couple to an unmerged fork PR. The TUI render layer is built per-cell with row-flex columns and OpenTUI's `<code>` renderable for `tree-sitter` syntax highlighting (see ADR 0009). Accepted coupling: Pierre is a young package; if it churns, `core/` is insulated and we eat migration cost on the parser only.
- **Lifecycle**: two states, `open` and `closed`. Human transitions via `tour close` or UI button. Closed tours stay in `.tour/<id>/` with `status = "closed"`. Cleanup is explicit (`tour prune --older-than 30d`).
- **Folder layout**: `.tour/<id>/` flat at top level; each contains `tour.toml` (front-matter) and `annotations.jsonl` (append-only annotation log). `.tour/` is gitignored by default (auto-added on first `tour create`); teams can opt-in to commit by removing the line.
- **Working-tree snapshot mechanics**: `git stash create` produces a synthetic commit SHA without touching the working tree; `git update-ref refs/tour/<id> <sha>` keeps it alive past gc. Stored as `head_sha` in `tour.toml`. Released on `tour delete` / `tour prune` via `git update-ref -d refs/tour/<id>`.
- **Annotation body format**: GitHub Flavored Markdown, no raw HTML. The webapp renders it rich (including ` ```mermaid ` fences as SVG diagrams); the TUI shows the raw source unchanged. Single string field — no `kind` enum, no separate `format` field. Existing annotations were already authored with markdown syntax; this decision honours that intent rather than introducing a new format.
- **Layout (v1)**: per-session, in-memory toggle between `split` and `unified`. Both surfaces default to `split` on first paint, matching the project's GitHub-style identity. Pressing `l` flips the layout in either surface (the webapp also exposes a segmented `[ Split | Unified ]` control in the top bar). State lives in `useState` on each renderer's `App`; not persisted (no localStorage, no `tour.toml` field, no on-disk state) and not synced across surfaces — page reload (webapp) or relaunch (TUI) returns to `split`. A single layout value applies to every Tour switched to in the same webapp session; layout is not stored on the Tour. Pierre's `diffStyle` and OpenTUI's `view` props consume the same `"split" | "unified"` vocabulary, so no translation layer.
- **Webapp annotation rendering**: annotations render inline via Pierre's `FileDiff` with `lineAnnotations` + `renderAnnotation`. Each row in `[line_start, line_end]` carries two visual cues: a subtle blue background tint (`rgba(88, 166, 255, 0.12)`) and a 3px accent-coloured gutter stripe at the left edge of the matching column (side-aware via Pierre's `[data-line-type]` selectors — `addition` rows for `additions` annotations, `deletion` rows for `deletions` annotations). The annotation card is bordered in the same `#58a6ff` accent — connecting card and range via shared colour. The card's left margin is `0` so its 3px `border-left` shares an x-column with the gutter stripe painted on the annotated lines above it (per Pierre's grid, `[data-line]` and `[data-annotation-content]` both sit at the start of the matching content column — left margin offsets either away). Both visual rules are injected into Pierre's shadow root via `unsafeCSS`. Mirrors the TUI's two-cue pattern (see ADR 0008); the surfaces share the accent colour and the side-aware placement.
- **TUI annotation rendering**: annotations render inline in the diff stream, directly below their `line_end` row, mirroring the webapp's "below the highlighted area" pattern. Each row in `[line_start, line_end]` carries two visual cues: a subtle blue background tint (matching the webapp's `RANGE_TINT`, applied full-width on context rows and to the line-number column only on `+`/`-` rows so the diff bg survives) and a single accent-colored gutter character (`▎`) in column 0. The annotation card is a bordered `<box>` rendered after the `line_end` row, with `borderColor` set to the same accent — connecting card and range via shared colour rather than literal box-drawing brackets. **Placement** is side-anchored in split layout (the card occupies only the half matching `Annotation.side` — `deletions` → left column, `additions` → right column — with the opposite half empty so the card sits under the range it discusses, mirroring `@pierre/diffs`'s `<FileDiff>` placement on the webapp) and full-width in unified layout. Hunk's docking-bracket apparatus was rejected as bespoke and high-cost; tint-only was rejected as single-cue and accessibility-weak. See ADR 0008.
- **TUI annotation navigation**: `n` / `p` scroll the annotation card itself into view (target id `annotation-${ann.id}`), not the `line_end` row or `line_start`. Mirrors the webapp's `scrollIntoView({ block: "center" })` pattern: the annotation is the navigation focus, and surrounding diff context appears above/below for free because the card is sandwiched between diff rows in the planner's output. The bottom annotation panel from the pre-inline TUI is removed — all annotation reading happens in the diff stream. Files with annotations are auto-expanded (existing rule in `core/file-classifier.ts`); a manually-collapsed file hides its annotations along with its diff body. `n` / `p` also move the **Line cursor** to the target annotation's anchor `(file, side, line_start)` so a subsequent `a` (top-level annotate) / `r` (reply) operates on the annotation the user just navigated to. The reverse coupling is asymmetric: `j` / `k` line motion does not update `currentAnnotationId` — code reading and annotation engagement are two distinct modes (β-coupling).

- **TUI line cursor**: persistent diff-pane cursor used to seed the top-level annotation composer with an arbitrary `(file, side, line_start, line_end)` anchor. Walks the full concatenated diff stream across files (skips folded files, hunk-header rows, annotation cards). Carries an explicit `preferredSide` (`additions` | `deletions`) that survives row moves — so commenting on three deletion-side lines in a row is `h` once + `a` three times, not `h+a` × 3; on rows where the preferred side isn't available, the effective side is forced to the only valid one (e.g. an addition-only row is always `additions` regardless of preference). Initial position seeds from the first top-level Annotation if any exist (matching today's "land on first annotation" UX), else the first annotatable row of the first non-folded file. Sidebar file-click moves the cursor to that file's first annotatable row (clicking a file expresses "show me from the top"); bundle reload (watcher fires) preserves cursor when its row still resolves. Visual treatment is two-cue and gutter-confined: a `▶` glyph in the line-number column on the active side plus a soft bg highlight on the same gutter strip — content area visuals (annotation tint, +/- bg, syntax highlight) are untouched. Keys: `j`/`k`/`down`/`up` move; `h`/`l`/`left`/`right` toggle side on paired rows in split layout (no-op elsewhere); click-to-position with side derived from clicked column. The previous global `l` binding for layout toggle is rebound to `L` (Shift-L) to free the lowercase pair for cursor side. See ADR 0011.
- **Row planner**: the TUI's diff renderer consumes a pure-data row planner in `core/diff-rows.ts` (no React, no OpenTUI dependency, fully unit-testable). The planner takes Pierre's `FileDiffMetadata` plus the file's annotations plus a `layout` and returns a flat `PlannedRow[]` of three kinds: `diff-row` (one diff line, with optional `tint` and `gutterMark` flags per side), `hunk-header`, and `annotation` (a card to render). Annotation rows are inserted directly after the row whose line number matches `line_end` on the matching `side`; multiple annotations with the same `(side, line_end)` stack in `created_at` ascending order. Side-awareness in split layout: tint paints only the column matching the annotation's `side` (`deletions` → left column, `additions` → right column); the gutter mark sits at the left edge of the matching column (column 0 for `deletions`, just right of the split divider for `additions`). In unified layout, both collapse to column 0 / single-column row tint. The TUI renders each kind with a thin React switch; the planner stays framework-agnostic to keep `core/`'s posture (cf. `file-tree.ts`).
- **Top header**: both surfaces share a single-line GitHub-style header. **Left cluster** (anchored to the left edge): bordered hamburger button (`☰`) that opens the Tour picker, then the title, then `<base_source> ← <head_source>` muted (arrow direction matches `git diff base..head`; refs come from `tour.toml`, not SHAs). **Right cluster** (anchored to the right edge): `[← N/M →]` annotation-navigation pill (hidden when `M === 0`), then `[Split | Unified]` layout toggle. The header has no border; only the hamburger has a border — the title sits naked next to it as a page heading, not inside a chip. The hamburger is icon-only (no inline label; webapp uses `title`/`aria-label`, TUI relies on the existing footer `t: tour picker` hint). When the row can't fit on one line, the right cluster wraps to its own row (still right-aligned), recovering the previous 2-row shape on narrow viewports / 80-col terminals; the title and sources clip with ellipsis when individually too long. Tour short-id (`#abc1`) is **omitted** from the header — disambiguation lives in the Tour picker (status glyph + age + count badge) and the URL / `tour list` carry the full id. Status pill (open/closed) is omitted — lifecycle isn't surfaced in v1's read-only flow. Created-at and total-files counts are omitted — recoverable from the tour picker (relative age) and sidebar (file list). Title is plain text, not clickable; the hamburger is the only picker trigger (mirrors GitHub PR pattern where `All commits ▾` is a separate widget from the PR title). Empty title falls back to `(untitled)` muted, matching the picker. Snapshot-lost banner stays directly below the header (unchanged), not inside it. The webapp's pre-existing floating `SequencePill` is relocated into the header (no longer `position: fixed`). The TUI gains a visible navigation counter for the first time — `n`/`p` keys still work in addition to clicking the pill arrows. Layout toggle in TUI gains a visible state indicator alongside the existing `l` key (previously keyboard-only with no on-screen indicator of current mode).
- **Thread rendering on existing card UI**: top-level Annotations are the unit of the sidebar count badge and the `[← N/M →]` header navigation pill — Replies are *inside* a Thread, not their own navigation targets (`n`/`p` walks topics, not utterances). The TUI's `c` collapse (commit 2a389e5) collapses Replies only, leaving the parent Annotation visible — whole-Thread collapse is reachable via the existing file-level collapse. The TUI's side-anchored card placement (commit c415152) accepts vertical bloat as Threads grow rather than re-flowing across columns; same trade-off the webapp accepts via Pierre's inline placement. Intra-Thread navigation (jumping between Replies inside one Thread) is not bound in v1; revisit if Threads routinely exceed ~5 Replies in real use.

- **Tour picker (v1)**: both surfaces open a Tour-switcher modal on `t` (top-level binding, focus-agnostic; webapp also opens via clicking the hamburger button in the top header; webapp ignores `t` when an editable element has focus). The modal lists every Tour from `.tour/` (open + closed) sorted newest-first; each row shows the status glyph (`●` open / `○` closed), relative age (`just now` / `Nm ago` / `Nh ago` / `Nd ago` / `Nw ago` / `Nmo ago` / `Ny ago`), title (`(untitled)` fallback), and an annotation-count badge when non-zero. Tour ID is hidden from the row. The current Tour is highlighted with a coloured background (webapp: `#1f6feb22`; TUI: `bg="blue"`, since `bg="cyan"` is taken by selection); the initial cursor lands on the first non-current row, or index 0 if the list is all-current. Keys inside the modal: j / ArrowDown / down, k / ArrowUp / up, Enter to commit, click row to commit (webapp), `t` and Esc to close, click scrim to close (webapp). While the picker is open, the App's input dispatch routes everything to the picker first; regular surface keys (`n` / `p` / `l`, j/k for files, Tab, Space, arrows, etc.) are inert (webapp: `n`/`p`/`l` are blocked; TUI: picker absorbs all input, no fall-through to `dispatchKey`). The modal opens even when only one Tour exists. Snapshot-lost Tours appear in the list with no special marker; selecting one falls through to the existing in-Tour banner. **TUI switch**: in-process — `cli/tui.ts` injects `loadTour(id)` and `loadTours()` callbacks; `loadTour` re-runs the existing pipeline (`getTour` + `readAnnotations` + `isShaResolvable` + `getDiff` + `parseDiff` + `classifyFile`) and the App swaps the bundle into local state without exiting; `loadTours` walks `readAnnotations` per Tour for accurate badge counts on first picker open. State reset on TUI switch: `selectedRowIdx` resets to 0, `currentAnnotationId` clears, `collapsedOverrides` clears, `collapsedFolders` clears; `sidebarFocused` and `layout` are preserved. **Webapp URL**: on user-initiated switch, `history.pushState({ tourId }, "", "/?tour=<id>")` (no `ann=` — the cursor restorer writes it once the new Tour anchors); on mount, `?tour=<id>` takes precedence over `window.__INITIAL_TOUR_ID__`; `popstate` syncs `tourId` back to the query param. The current top-level Annotation also persists as `?tour=<id>&ann=<ann-id>` via `replaceState` — chosen over `pushState` so the browser back button steps over Tour switches, not over every `n`/`p` keystroke. On mount/refresh, when the URL `ann=` resolves to a top-level Annotation in the loaded Tour, the surface restores it as if the user had just navigated there: file uncollapsed, folder ancestors expanded, card scrolled into view. A missing `ann=` (deleted, hand-edited, or pointing at a Reply — Replies aren't navigation targets) silently falls back to the first top-level Annotation and the URL is rewritten to drop the stale param, so shared links can't propagate broken state. State reset on webapp switch: `selectedFile`, `currentAnnotationId`, `collapsedOverrides`, and `collapsedFolders` are cleared; `layout` is preserved (per PRD #36). No search / filter v1 — deferred until tour counts justify it (~30+).
