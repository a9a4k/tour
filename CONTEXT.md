# Tour

A code-review tool that pairs an ephemeral, GitHub-style diff (split or unified) with persisted AI annotations. Drives the same data from a TUI and a webapp, both consuming local files written by agents through a CLI.

## Language

**Tour**:
A single guided traversal of a pinned git diff, with zero or more annotations attached. Lives in `.tour/<id>/`.
_Avoid_: review, PR, pull request, code review session, changeset

**Diff**:
The set of file changes shown in a Tour, recomputed from git on every open. Never persisted.
_Avoid_: patch, changes, delta

**Head**:
The git ref the Tour's diff ends at. Resolved to a SHA at create time. May be a real commit or a synthetic snapshot of the working tree.
_Avoid_: tip, target

**Base**:
The git ref the Tour's diff starts from. Resolved to a SHA at create time. Defaults to `head^` for single-commit tours.
_Avoid_: parent, ancestor

**Annotation**:
A note anchored to `(file, side, line_start, line_end)` inside a Tour's diff. `side` is `additions` or `deletions` (matching Pierre's vocabulary); line numbers are file-line-numbers in the file as it exists on that side at the pinned SHA. Authored by agents (via CLI) or humans (via UI or CLI). Persisted in the Tour's folder. The body is **GitHub Flavored Markdown** (no raw HTML); the webapp renders it as rich markdown — including ` ```mermaid ` fences as diagrams — while the TUI shows the raw source. No `kind` enum — structure lives in the markdown itself. Carries an `author_kind ∈ {agent, human}` so the system can distinguish the two without parsing the free-form `author` display string.
_Avoid_: comment, review comment, note

**Reply**:
An Annotation whose `replies_to` field points at another Annotation in the same Tour. Inherits the parent's `(file, side, line_start, line_end)` anchor — replies don't have independent positions. Renders inside the parent's card as part of a Thread.
_Avoid_: comment-on-comment, sub-annotation, response

**Thread**:
A top-level Annotation plus the chain of Replies attached to it. Lives entirely inside one Tour — Threads do not carry across Tours in v1 (preserves ADR 0003's "no re-anchoring" stance). When a Tour closes, its Threads close with it. Continuity across Tours is the agent's job (it re-references prior concerns in prose when reviewing the new SHA), not Tour's data model.
_Avoid_: conversation, discussion, exchange

**Reply-agent**:
An ephemeral, capability-bounded agent process spawned by the renderer's watcher whenever a new Annotation with `author_kind = "human"` lands. The agent has **no tools** — it runs in non-interactive print mode and emits the reply body to stdout; Tour captures stdout and writes the Reply Annotation itself. The capability boundary is "no tools," strictly stronger than the prior single-allowed-tool design: the agent literally cannot read, write, edit, or shell out. A Tour-canonical system prompt instructs the model to emit only the reply body — no preamble, no narration. Distinct from the **main-agent** — the user's existing Claude Code / Codex / etc. session — which is unrestricted and consumes the conversation later via `tour pickup`.
_Avoid_: bot, listener, autoreplier

**Side**:
The diff-half an Annotation belongs to: `deletions` for the base-file half, `additions` for the head-file half. Required, not derived. Layout-independent: in `split` layout, `deletions` renders on the left column and `additions` on the right; in `unified` layout, `deletions` attaches to the `-` row and `additions` to the `+` row. Context rows are addressable on **both** columns — `side` follows the column the user clicked (or the cursor's `preferredSide` for keyboard-driven `a`); the prior "context → additions by convention" rule was reversed in ADR 0012's Revisions to match GitHub's split-view comment model. Naming matches Pierre's `AnnotationSide` so `core/` ↔ Pierre is a no-op.
_Avoid_: LEFT/RIGHT, before/after, old/new, column

**Layout**:
How a Tour's Diff is rendered: `split` (deletions left, additions right) or `unified` (single column with `-`/`+` rows). Per-session UI state in each surface — not persisted on the Tour, not synced between TUI and webapp. Annotations are layout-independent: the same `(file, side, line_start, line_end)` anchor resolves in both. Maps to Pierre's `diffStyle` and OpenTUI's `view` props (both already accept `"unified" | "split"`).
_Avoid_: view mode, diff style, view

**Working-tree snapshot**:
A synthetic commit object capturing uncommitted changes at the moment a Tour is created, so the Diff stays pinned even as the working tree keeps moving.
_Avoid_: stash, WIP commit

**Tour bundle**:
Everything required to render a pinned Tour at open time — the Tour TOML, its Annotations, the parsed Diff, per-side file contents, file classifications, and orphan-annotation auto-windows — packaged as one value computed fresh on every open. Never persisted. Discriminated by `kind`: `ok` carries the full payload; `snapshot-lost` carries just the Tour + Annotations so the surface can render the snapshot-lost banner without per-file machinery. The Reply lock is **not** part of the bundle (lock changes are O(read one file) and fetched on their own watcher event without re-running the full hydrate pipeline).
_Avoid_: payload, snapshot (collides with Working-tree snapshot), state

**Hidden context**:
The unchanged lines between hunks in a Tour's Diff that aren't part of any hunk's visible context window (git's `-U3` default). Surfaces in three places: between consecutive hunks, before the first hunk (file-top gap, absorbed into the first hunk's `hunk-header row`), after the last hunk (file-bottom gap, addressed via a standalone `boundary-bottom` row). The `hunk-header row` visualises the gap and is the keyboard target for revealing it; for mid-file gaps larger than `2N` (N = 20 lines per direction press), a second `gap-mid-top` row appears above the `hunk-header` so each end of the gap has its own affordance. Anchor of an Annotation may live inside Hidden context; revealing it is a render-time concern, not a data concern (the SHA, file, and line numbers are unchanged).
_Avoid_: hidden lines, collapsed lines, gap, missing context

**Hunk-header row**:
The synthetic row that carries a hunk's `@@ -a,b +c,d @@` metadata. Always emitted — one per hunk in the planner's output (`HunkHeaderRow` in `core/diff-rows.ts`). **Interactive iff `gapAbove > 0`**: when the row sits above a non-zero stretch of Hidden context, the cursor walks to it and `Enter` expands the gap; when `gapAbove === 0` (first hunk starts at line 1, or two consecutive hunks abut with no hidden gap between them) the row stays inert text. Direction semantics derive from `(hunkIndex, gapAbove)`: for the first hunk, `gapAbove` is the file-top gap (lines 1 to first-hunk-start) and expansion is single-direction toward the file's start — the asymmetric merge that replaces the standalone file-top boundary row, structural because git's hunk format has headers but no footers; for mid-file hunks with `0 < gapAbove ≤ 2N` the row carries symmetric expansion (single keystroke / chevron reveals from both ends of the gap); for mid-file hunks with `gapAbove > 2N` the row carries bottom-direction expansion and pairs with a `gap-mid-top` row above. N = 20 lines per direction press across both surfaces.
_Avoid_: hunk separator, hunk-separator row, `@@` row (as a row kind)

**Gap-mid-top row**:
A synthetic interactive row emitted **only** for mid-file gaps where `gapAbove > 2N` (with N = 20, so `gapAbove > 40` lines). Sits immediately above the bottom-direction `hunk-header` of the next hunk. Enter / click expands the **top** end of the gap — newly-revealed lines appear above this row, closest to the previous hunk's end. Paired with its `hunk-header` neighbour the row gives a large mid-file gap two cursor stops and two `Enter` actions, one per end of the gap; small mid-file gaps (`0 < gapAbove ≤ 2N`) need no `gap-mid-top` because the `hunk-header`'s symmetric expand covers the whole gap in one press.
_Avoid_: gap top, top-of-gap row, mid-top boundary

**Boundary-bottom row**:
The synthetic interactive row emitted after the last hunk's content when the file's hidden tail is non-zero. Stays a standalone row — there is no merge target on the file-bottom side, structurally asymmetric with file-top because git's hunk format has hunk-headers but no hunk-footers. Enter / click expands toward EOF; `Shift+Enter` / shift-click expands all of the remaining file. Cursor-walkable on both surfaces.
_Avoid_: file-bottom row, EOF row, end boundary

**Expand context**:
The act of revealing some or all of a stretch of Hidden context. Mechanism is per-surface but the row stream is shared — both surfaces consume the same planner output (`core/diff-rows.ts`) and emit the same gap-row family (`hunk-header` when interactive, `gap-mid-top`, `boundary-bottom`). **Webapp**: Tour-owned gap-row affordances overlay-injected onto Pierre's DOM; click calls Pierre's `expandHunk(hunkIndex, direction, expansionLineCount)` via a per-file `FileDiff` ref. Tour does not mirror Pierre's expansion state — Pierre stays the source of truth for which lines are currently revealed, and Tour re-renders so the planner recomputes gap counts from Pierre's updated `FileDiffMetadata`. **TUI**: cursor walks the same gap-row family; `Enter` routes through `core/expansion-state.ts` reducers to reveal lines, `Shift+Enter` expands the whole gap. The cursor's annotation-anchor scope is unchanged — `a` (top-level annotate) is a no-op when the cursor sits on any interactive row. The same `Enter` primitive also expands collapsed-by-classifier files (lockfiles / generated / vendored): the file-collapsed indicator becomes a cursor-walkable interactive row, unifying "expand hidden context" and "expand hidden file" under one gesture. **D1 directional convention** (both surfaces): the row's position == which end of the gap it expands. Top row → top of gap → lines appear above. Bottom row → bottom of gap → lines appear below. D1 intentionally diverges from GitHub's chevron-direction convention (where the chevron-glyph points at the half being revealed); D1 is chosen because it's spatially direct and surface-consistent — newly-revealed lines always appear adjacent to the row that revealed them, and the same logical row stream renders cleanly in both OpenTUI's row model and Pierre's DOM grid. Step size N = 20 lines per direction press, matching Pierre's `expansionLineCount: 20` and the TUI's prior symmetric-20 semantics; `Shift+Enter` / shift-click bypasses the step and expands all. Orphan annotations (anchors that fall in Hidden context) are auto-windowed at planner-init time with a ±10 line region around each anchor so the sidebar count and the diff stream stay honest. State is per-renderer-session (in-memory): preserved across watcher-driven bundle reloads (diff is SHA-pinned), reset on tour switch and webapp page reload. Prior versions of this entry claimed the webapp used Pierre's `expandUnchanged: true` — that mechanism auto-renders the whole file and was reverted in `c20a263`; the current mechanism is the Tour-owned gap rows described above.
_Avoid_: load context, show context, unfold

**Line cursor**:
A persistent diff-pane focus position used as the anchor for new top-level Annotations. Walks the full concatenated diff stream across files (cross-file scope, mirroring how the diff pane scrolls), landing on **diff rows and interactive rows** — interactive rows being the `hunk-header` (when its `gapAbove > 0`), `gap-mid-top`, `boundary-bottom`, and `collapsed-file` synthetic rows the cursor walks alongside source lines (PRD #107 / ADR 0013 / ADR 0018). A `hunk-header` with `gapAbove === 0` (first hunk at line 1, or hunks abutting with no hidden gap) stays inert and is skipped by the cursor walker. The previously-emitted `boundary-top` row is gone — its file-top semantics fold into the first hunk's `hunk-header`. Carries an explicit `preferredSide` so split layout's two columns are both addressable. Tracked separately from the **annotation cursor** (`currentAnnotationId`); navigating annotations with `n`/`p` moves both, but moving the line cursor with `j`/`k` does not move the annotation cursor — line motion is for code reading, annotation motion is for engaging with conversation (β-coupling). Both surfaces (TUI + webapp) implement the cursor with progressive disclosure: cursor is null on tour load; the first user interaction (`j`/`k`/`h`/`l`/arrows/`a`/`n`/`p`/click) materializes it at the first top-level Annotation if any exist, else at the first annotatable row of the first non-folded file (initial position never lands on an interactive row by default). On a regular diff row, `a` opens the top-level composer at the cursor's anchor; on an interactive row, `a` is a silent no-op (interactive rows are not annotatable) and `h`/`l` (cursor-side toggle) is also a no-op (no `side` concept there). On an interactive row in the TUI, `Enter` dispatches a row-kind-specific `primary-action` (and `Shift+Enter` a `primary-action-all`); on a diff row `Enter` is a noop reserved for future interactive-row gestures. Visual treatment differs by surface: TUI uses a gutter-confined `▶` glyph + soft bg in the line-number column (matching vim/lazygit/fzf convention); webapp uses a GitHub-style rounded outline (`outline: 2px solid var(--accent)`) on the active row's cell, scoped to `preferredSide`'s column in split layout. See ADR 0011 (TUI), ADR 0012 (webapp), ADR 0013 (interactive-row family + Enter primitive).
_Avoid_: caret, point, position, selection

## Relationships

- A **Tour** has exactly one **Head** and one **Base**, both stored as SHAs.
- A **Tour** has zero or more **Annotations**.
- An **Annotation** belongs to exactly one **Tour** and anchors to one file + line-range inside that Tour's **Diff**.
- A **Reply** is an Annotation that points at another Annotation in the same Tour via `replies_to`.
- A **Thread** is a top-level Annotation plus its chain of Replies; lives entirely inside one Tour.
- A **Working-tree snapshot** acts as a synthetic **Head** when an agent creates a Tour of uncommitted work.

## Example dialogue

> **Dev:** "If the agent amends the commit after creating a **Tour**, do the **Annotations** still line up?"
> **User:** "The **Tour** is pinned to the original SHA. Amending creates a new SHA — the old one is still in git's object store, so the **Diff** and **Annotations** still resolve. If the agent wants to tour the amended version, they create a new **Tour**."

## Resolved decisions

- **Annotation lifetime**: per-tour-pass. Annotations are not re-anchored across rebases or amendments. Stale Tours are abandoned, not migrated.
- **Diff source**: commit-pinned only. Working-tree tours are supported by snapshotting to a synthetic commit at create time, so the rest of the system only ever sees SHAs.
- **Multiplicity**: many Tours coexist, ordered by creation time. Default open behavior surfaces the most recent unfinished one.
- **Tool surface**: CLI-first. Agents drive the system by shelling out (`tour create`, `tour annotate`, …) with `--json` output. MCP is deferred and would be a thin wrapper if added later.
- **Annotation anchor**: `(file, side, line_start, line_end)` with `side ∈ {additions, deletions}` (matching Pierre's `AnnotationSide`) and line numbers being file-line-numbers at the pinned SHA. View-mode independent; both split and unified consume the same shape. No `kind` enum in v1 — the body is plain text.
- **Authoring**: both agents and humans author Annotations. Top-level Annotations and Replies use the same primitive (`tour annotate`, optionally with `--reply-to <ann-id>`). Each Annotation carries `author_kind ∈ {agent, human}`; the free-form `author` string is kept for display. Both surfaces (TUI + webapp) gain a composer for top-level notes and replies. The CLI is canonical and remains the agent path; the UI is the human path. Originally v1 said "agents only, UI is read-only"; that was reversed once Threads landed (see ADR 0010). Resolution state is **not** modeled — readers (the main-agent at `tour pickup` time, or another human) infer outcomes from natural language. No `agree:`/`dismiss:` prefix conventions, no status enum. **Body must be non-empty after trimming**: whitespace-only bodies reject at the Annotation creation seam (`createAnnotation` / `createReply` / `createAnnotations`), uniformly for every surface — CLI, TUI, webapp, reply-runner (PRD #140 / slice 2 #142). For the reply-runner specifically, an empty agent stdout is treated as a normal dispatch completion (lock clears via the standard release path, ADR 0015); the runner records a rejection entry in the per-dispatch log and skips the write so no empty Reply Annotation lands on disk. **Author defaults to the `author_kind` literal**: when a caller omits `author`, the seam falls back to `"human"` if `author_kind === "human"` and `"agent"` if `author_kind === "agent"`. No environment-dependent fallback — no OS username, no per-surface constant. The CLI's `--author "..."` flag (and the webapp's optional `author` body field) lands verbatim when supplied. The TUI's pre-seam `humanAuthor()` helper and the webapp's `DEFAULT_HUMAN_AUTHOR = "you"` constant are gone; both surfaces simply omit `author` and let the seam decide (PRD #140 / slice 3 #143). See ADR 0016 for the no-environment-leak rationale. **Anchor must resolve inside the Tour's Diff at write time**: `createAnnotation` and `createAnnotations` take a required `bundle: TourBundle` argument and reject any request whose `file` is not in `bundle.files` or whose `(line_start, line_end)` falls outside `1 ≤ line_start ≤ line_end ≤ lineCount(file, side)` (head side for `additions`, base side for `deletions`). Hidden-context anchors (between or outside hunks) stay legal — `orphan-window` handles their render-time placement. Unchanged-context-row anchors stay legal under the existing `additions`-by-convention rule. Replies inherit their anchor from the parent and skip validation; `createReply` does not take a bundle. The CLI loads the bundle once per invocation (single or `--batch`); the TUI passes its in-memory bundle through; the webapp loads inline per POST. Pre-existing bad anchors on disk are not retroactively validated — the seam only acts at write time (PRD #140 / slice 4 #144). See ADR 0017 for the rationale.
- **Sidebar (tree)**: folder tree of changed files. Path compression folds any non-leaf node with exactly one folder child into its child (lazygit-style), so sparse tours don't waste rows on empty intermediate folders. Folders sort before files at each level; alphabetical within each group. File rows render an Octicon-based file-status icon (added/modified/deleted/renamed, sourced from `@primer/octicons-react`'s diff icon set via the `web/client/icons.ts` shim and the `web/client/file-icon.ts` `type → { Icon, statusClass }` mapping) and a per-file annotation count badge. Folder rows render a chevron Octicon (`ChevronRightIcon` when collapsed, `ChevronDownIcon` when expanded) followed by `FileDirectoryFillIcon` and the compressed path — GitHub-PR-tree-shaped, with one folder icon in both states (the chevron carries the open/closed signal). Per-depth indent is `16px` for both row kinds, giving the chevron + folder-icon group enough horizontal breathing room. The TUI sidebar mirrors this noise-reduction (no folder rollup, no inline classification-reason tag) and keeps its single-letter status glyph and per-file count. The classification reason still surfaces in the main-pane file header on both surfaces — only the sidebar duplicate is removed. The pure shared module `core/file-tree.ts` is unchanged: `flatten` keeps producing `VisibleRow[]` with `annotationCount` on every row; the surfaces simply ignore it on folder rows. Default state on first render: every folder expanded. Clicking a folder toggles its expand/collapse; clicking a file selects it and scrolls the diff stream to it. `n`/`p` annotation navigation reveals (expands) the ancestor folder chain of the target file. Expand/collapse state is React state, in-memory only, per renderer instance, per tour — page reload resets to "all expanded" (no localStorage, no on-disk state). The tree algorithm lives in the pure shared module `core/file-tree.ts`; both the webapp and TUI consume `VisibleRow[]` from `flatten`. Originally deferred from v1 ("Real folder-tree rendering is deferred until file counts justify it") to ship the smallest viable Tour; reinstated now that the v1 surface is shipped and real tours expose the flat list's limitations on deeply-nested paths. Main pane is unchanged: a single top-to-bottom stream of all file diffs; clicking a sidebar file entry scrolls to that file.
- **Watch behavior**: TUI and webapp watch the open Tour's `.tour/<id>/` folder for changes and re-render. The watcher fires on `*.jsonl` (annotation log changes) and `.reply-lock.json` (in-flight reply state, see Reply-agent dispatch below). They do not watch git refs or working-tree files — the diff is pinned to `head_sha`.

- **Reply-agent dispatch**: when the watcher detects a new Annotation with `author_kind = "human"` AND the renderer was launched with `--reply-agent <name>`, it dispatches via one of the shipped TS adapters in `src/agents/` (claude / codex / gemini / opencode / pi), which spawns the inner CLI in-process with the triggering Annotation's thread context as a JSON envelope passed through the adapter's argv-builder. Agent invocation is single-flight per Tour, gated by `.tour/<id>/.reply-lock.json` (which doubles as the renderer's status-pill source). System prompt is owned by Tour and built in-process; the adapter passes it as a flag value to its inner CLI (e.g. claude's `--system-prompt`) — no external fetch channel. The capability boundary is **zero tools**: the inner CLI runs with no `--allowedTools` / `--disallowedTools` / equivalent allow-list, and the agent's only output is stdout, which Tour captures verbatim as the Reply Annotation body (see ADR 0012). Copilot CLI was cut from MVP (its capability-bounding surface isn't designed for single-tool-only enforcement; see ADR 0010); the registry is fixed and custom out-of-tree adapters are not supported. Without `--reply-agent` the watcher is inert and Tour functions as a static review tool — humans can still author, the conversation just sits until consumed via `tour pickup`. See ADR 0010.

- **Reply streaming**: while a reply-agent is in flight, the renderer paints a status pill below the targeted Annotation ("✏️ `<agent>` is replying… (Ns)") sourced from `.reply-lock.json`. When the agent's `tour annotate --reply-to` write lands and the lock releases, the pill is replaced atomically by the rendered Reply. Token-by-token streaming was rejected in favour of this lockfile-based status because it composes with the single-flight lock we already need (zero new IPC channel) and captures the actual user need ("is the agent on it?") for ~5% of the implementation cost. Orphan locks (renderer crashed mid-flight) auto-clear on read via a PID-liveness check; no user action needed. Stale locks (>2min, agent alive but hung) flip the pill to a single-line warning; recovery is to restart the renderer. See ADR 0015.

- **Conversation pickup**: `tour pickup <id> --json` emits the full conversation tree as JSON for consumption by the user's main-agent session. The schema is `annotations.jsonl` reshaped into a tree (each top-level Annotation carries its `replies` array). No status field, no actionable-only filter, no markdown digest in v1 — Tour emits raw conversation; the main-agent does all reasoning. Triggered by the human (or asked of the main-agent) when they want code changes acted on; Tour itself never moves the SHA mid-conversation.
- **Handoff**: agent prints the tour ID to stdout (and a one-line "Open with: …" hint). No auto-open. Humans open the UI explicitly via `tour` (TUI) or `tour serve` (webapp). `tour create --open` opt-in for users who do want auto-launch.
- **Tour ID**: `YYYY-MM-DD-HHMMSS-xxxx` (UTC timestamp + 4-char random suffix). Sortable by creation time; collision-safe under parallel creates; prefix-matchable (`tour show 2026-05-08-1`).
- **CLI surface**: `tour create --head <ref> [--base <ref>] [--title <s>]` resolves both ends to SHAs at create time. `--head WIP` snapshots uncommitted changes. `--base` default is context-sensitive: `<head>^` for commit tours, `HEAD` for `WIP` tours. Verbs: `create`, `annotate`, `list`, `show`, `close`, `delete`, `prune`, `tui`, `serve`, `pickup`. `tour annotate` accepts `--reply-to <ann-id>` for replies and an `--as-agent` / `--as-human` pair that sets `author_kind` (default: `agent`). `tour tui` and `tour serve` accept `--reply-agent <name>` to enable live agent replies; without it the watcher is inert. Bare `tour` opens TUI on the most recent open Tour.
- **On-disk schema**: `tour.toml` holds `id`, `title?`, `status`, `created_at`, `closed_at?`, `head_sha`, `base_sha`, `head_source`, `base_source`, `wip_snapshot`. `annotations.jsonl` is one annotation per line: `{id, file, side, line_start, line_end, body, author, author_kind, created_at, replies_to?}`. `author` is a free-form display string (e.g. `"agent"`, `"claude-code"`, `"almas"`); `author_kind` is the structured discriminator (`"agent"` | `"human"`) used by the reply-agent dispatch predicate, required on every Annotation. `replies_to` (optional) carries the bare `id` of the parent Annotation when this entry is a Reply. Pre-bidirectional `.tour/` data is not migrated — Tour isn't in production yet, so the schema break is taken cleanly rather than papered over with reader-side defaults. `.reply-lock.json` (transient) holds `{agent, responding_to, started_at, pid}` while a reply-agent is in flight.
- **Webapp lifecycle**: `tour serve` is long-running, single-user, binds to `127.0.0.1:8687` (TOUR on a T9 keypad; configurable via `--port`), no auth. `Ctrl-C` to stop. Browser auto-open only with `--open`. If the default port is busy, the server walks up (8688, 8689, …) until it finds a free port and prints the bound URL; with an explicit `--port <N>` no fallback occurs — busy ports fail loudly.
- **Snapshot loss**: if a Tour's `head_sha` no longer resolves (e.g. user manually ran `git update-ref -d refs/tour/<id>`), the UI shows a banner "Snapshot lost — annotations preserved but diff cannot be displayed." Read-only, can be deleted.
- **Packaging**: single binary, multi-subcommand (same shape as Backlog.md). One `tour` package containing a shared `core/` (git, tour IO, annotations, watcher) and two renderers (`tui/`, `web/`). Stack: Bun + TypeScript + OpenTUI for TUI; Bun HTTP server + React for web.
- **Distribution**: npm package is `tourdiff`; installed binary is `tour`. Per-platform sub-packages `tourdiff-darwin-arm64`, `tourdiff-darwin-x64`, `tourdiff-linux-arm64`, `tourdiff-linux-x64`, `tourdiff-windows-x64` are pulled in via pinned `optionalDependencies` (only the matching one installs on a given OS/arch). The parent ships a `scripts/cli.cjs` shim that resolves the right binary at runtime — exact mirror of `backlog.md`'s install model. GitHub Releases also attach the same 5 binaries as a second distribution path. No telemetry, no update-check, no outbound network calls. Release pipeline is a GitHub Actions workflow on `vX.Y.Z` tag push: matrix-builds 5 binaries with `bun build --production --compile`, smoke-tests `./tour --version` per-platform, then publishes to npm with `--provenance` (granular token, publish-only, scoped to `tourdiff` + `tourdiff-*`). Pre-1.0 semver: minor=breaking, patch=compat. Repo `github.com/a9a4k/tour`, license MIT.
- **Diff engine (v1)**: `@pierre/diffs` parses git diffs in both surfaces. Webapp renders via Pierre's `FileDiff` (uses its built-in annotation framework). TUI walks Pierre's parsed `FileDiffMetadata` row-by-row through a Tour-owned row planner in `core/` (see ADR 0007); `@opentui/core`'s `<diff>` and `hunkdiff/opentui`'s `HunkDiffView` were both rejected because neither exposes a slot for inline annotation content between rows; `anomalyco/opentui#621` (`SlottableDiffRenderable`) does, but was rejected to stay on stock OpenTUI rather than couple to an unmerged fork PR. The TUI render layer is built per-cell with row-flex columns and OpenTUI's `<code>` renderable for `tree-sitter` syntax highlighting (see ADR 0009). Accepted coupling: Pierre is a young package; if it churns, `core/` is insulated and we eat migration cost on the parser only.
- **Lifecycle**: two states, `open` and `closed`. Human transitions via `tour close` or UI button. Closed tours stay in `.tour/<id>/` with `status = "closed"`. Cleanup is explicit (`tour prune --older-than 30d`).
- **Folder layout**: `.tour/<id>/` flat at top level; each contains `tour.toml` (front-matter) and `annotations.jsonl` (append-only annotation log). `.tour/` is gitignored by default (auto-added on first `tour create`); teams can opt-in to commit by removing the line.
- **Working-tree snapshot mechanics**: `git stash create` produces a synthetic commit SHA without touching the working tree; `git update-ref refs/tour/<id> <sha>` keeps it alive past gc. Stored as `head_sha` in `tour.toml`. Released on `tour delete` / `tour prune` via `git update-ref -d refs/tour/<id>`.
- **Annotation body format**: GitHub Flavored Markdown, no raw HTML. The webapp renders it rich (including ` ```mermaid ` fences as SVG diagrams); the TUI shows the raw source unchanged. Single string field — no `kind` enum, no separate `format` field. Existing annotations were already authored with markdown syntax; this decision honours that intent rather than introducing a new format.
- **Layout (v1)**: per-session, in-memory toggle between `split` and `unified`. Both surfaces default to `split` on first paint, matching the project's GitHub-style identity. Pressing `l` flips the layout in either surface (the webapp also exposes a segmented `[ Split | Unified ]` control in the top bar). State lives in `useState` on each renderer's `App`; not persisted (no localStorage, no `tour.toml` field, no on-disk state) and not synced across surfaces — page reload (webapp) or relaunch (TUI) returns to `split`. A single layout value applies to every Tour switched to in the same webapp session; layout is not stored on the Tour. Pierre's `diffStyle` and OpenTUI's `view` props consume the same `"split" | "unified"` vocabulary, so no translation layer.
- **Webapp annotation rendering**: annotations render inline via Pierre's `FileDiff` with `lineAnnotations` + `renderAnnotation`. Each row in `[line_start, line_end]` carries two visual cues: a subtle blue background tint (`rgba(88, 166, 255, 0.12)`) and a 3px accent-coloured gutter stripe at the left edge of the matching column (side-aware via Pierre's `[data-line-type]` selectors — `addition` rows for `additions` annotations, `deletion` rows for `deletions` annotations). The annotation card is bordered in the same `#58a6ff` accent — connecting card and range via shared colour. The card's left margin is `0` so its 3px `border-left` shares an x-column with the gutter stripe painted on the annotated lines above it (per Pierre's grid, `[data-line]` and `[data-annotation-content]` both sit at the start of the matching content column — left margin offsets either away). Both visual rules are injected into Pierre's shadow root via `unsafeCSS`. Mirrors the TUI's two-cue pattern (see ADR 0008); the surfaces share the accent colour and the side-aware placement. The **currently-selected card** (the `n`/`p` target — `isCurrent` true) reads differently from its peers along three redundant axes so the cue survives palette drift, colour blindness, and low-contrast displays: the top/right/bottom borders flip from neutral grey to the accent (left edge was already accent), an `8px` drop shadow lifts the card off the page, and a leading `●` accent glyph appears in the header — matching the TUI's three-axis treatment.
- **TUI annotation rendering**: annotations render inline in the diff stream, directly below their `line_end` row, mirroring the webapp's "below the highlighted area" pattern. Each row in `[line_start, line_end]` carries two visual cues: a subtle blue background tint (matching the webapp's `RANGE_TINT`, applied full-width on context rows and to the line-number column only on `+`/`-` rows so the diff bg survives) and a single accent-colored gutter character (`▎`) in column 0. The annotation card is a bordered `<box>` rendered after the `line_end` row, with `borderColor` set to the same accent — connecting card and range via shared colour rather than literal box-drawing brackets. **Placement** is side-anchored in split layout (the card occupies only the half matching `Annotation.side` — `deletions` → left column, `additions` → right column — with the opposite half empty so the card sits under the range it discusses, mirroring `@pierre/diffs`'s `<FileDiff>` placement on the webapp) and full-width in unified layout. The **currently-selected card** (the `n`/`p` target — `isCurrent` true) reads differently from its peers along three redundant axes so the cue survives palette drift, colour blindness, and low-contrast displays: a `heavy` border style (vs `single` on peers), the brighter `accentCurrent` background tier (vs the subtler `accentSubtle`), and a leading `●` glyph in the header row — a dedicated marker analogous to the line cursor's gutter `▶`. Peers keep their accent border colour so they still read as annotations, but visibly recede. Hunk's docking-bracket apparatus was rejected as bespoke and high-cost; tint-only was rejected as single-cue and accessibility-weak. See ADR 0008.
- **TUI annotation navigation**: `n` / `p` scroll the annotation card itself into view (target id `annotation-${ann.id}`), not the `line_end` row or `line_start`. Mirrors the webapp's `scrollIntoView({ block: "center" })` pattern: the annotation is the navigation focus, and surrounding diff context appears above/below for free because the card is sandwiched between diff rows in the planner's output. The bottom annotation panel from the pre-inline TUI is removed — all annotation reading happens in the diff stream. Files with annotations are auto-expanded (existing rule in `core/file-classifier.ts`); a manually-collapsed file hides its annotations along with its diff body. `n` / `p` also move the **Line cursor** to the target annotation's anchor `(file, side, line_start)` so a subsequent `a` (top-level annotate) / `r` (reply) operates on the annotation the user just navigated to. The reverse coupling is asymmetric: `j` / `k` line motion does not update `currentAnnotationId` — code reading and annotation engagement are two distinct modes (β-coupling).

- **TUI line cursor**: persistent diff-pane focus position used to seed the top-level annotation composer with an arbitrary `(file, side, line_start, line_end)` anchor. Walks the full concatenated diff stream across files (skips folded files, hunk-header rows, annotation cards). Carries an explicit `preferredSide` (`additions` | `deletions`) that survives row moves — so commenting on three deletion-side lines in a row is `h` once + `a` three times, not `h+a` × 3; on rows where the preferred side isn't available, the effective side is forced to the only valid one (e.g. an addition-only row is always `additions` regardless of preference). Initial state is null (cursor is invisible on tour load); first user interaction (`j`/`k`/`h`/`l`/arrows/`a`/`n`/`p`/click) materializes the cursor at the first top-level Annotation if any exist, else at the first annotatable row of the first non-folded file. `a` on a null cursor materializes the cursor at the default target AND opens the composer in one step. Lazy materialization keeps the cursor and `currentAnnotationId` accent from stacking on first paint and matches the webapp cursor's default (ADR 0012). Sidebar file-click moves the cursor to that file's first annotatable row (clicking a file expresses "show me from the top"); bundle reload (watcher fires) preserves cursor when its row still resolves. State shape is a semantic anchor `{ file, lineNumber, side, preferredSide } | null` — same vocabulary as `Annotation`'s anchor, so cursor → composer is a direct copy. The `flatRows` traversal index is derived on demand; the anchor itself is invariant under layout toggle, fold/unfold, and bundle reload. Layout toggle (`L`) preserves cursor anchor and auto-scrolls the new visual position into view; composer cancel/submit doesn't move the cursor. Visual treatment is two-cue and gutter-confined: a `▶` glyph plus a soft bg highlight in the line-number column on the active side; content area visuals (annotation tint, +/- bg, syntax highlight) untouched. Side-aware in split (only the active column carries the cues), full-row in unified. Keys: `j`/`k`/`down`/`up` move; `h`/`l`/`left`/`right` toggle side on paired rows in split layout (no-op elsewhere); click-to-position with side derived from clicked column. The previous global `l` binding for layout toggle is rebound to `L` (Shift-L) to free the lowercase pair for cursor side. See ADR 0011.
- **TUI diff-pane motion**: the line cursor floats in the document; the diff pane is a passive follower (vim / helix / VS Code / IntelliJ / emacs model). `j`/`k` move the cursor only — the pane scrolls one row at a time when the cursor would otherwise cross a 3-row edge margin (vim-style `scrolloff`). `Space`/`Shift-Space` and the hardware `PageDown`/`PageUp` keys scroll one viewport AND advance the cursor so its screen-relative position is preserved, keeping the keyboard's home base reachable (pressing `a` after `Space` annotates a visible line, not a stranded one). `Home`/`End` jump the cursor to the first/last cursor-eligible row in the concatenated diff stream, with the pane scrolling to match. Mouse wheel is pure pane scroll — the cursor stays at its file line; the next `j`/`k` snaps the pane back via `block:nearest`, so keyboard motion remains "magnetic" to the cursor. The keyboard-vs-wheel asymmetry reflects two input modalities preserving their respective home bases (cursor for keyboard, pointer for mouse) and matches every floating-cursor document tool surveyed; list-view tools that pin the cursor to a fixed screen row (lazygit lists, fzf, k9s, htop) don't have this asymmetry, but Tour's diff pane is a document, not a list. `n`/`p` annotation jumps recenter the target card unconditionally (`block:center`, with `block:start` fallback for cards taller than the viewport) — matches the webapp's `scrollIntoView({ block: 'center' })` so both surfaces agree on jump semantics. Implementation: the `focused` prop is removed from the diff `<scrollbox>`; the React keymap is the single source of truth for diff-pane keys, preventing OpenTUI's built-in `ScrollBar.handleKeyPress` from firing in parallel with the keymap (the parallel-handler bug — every `j`/`k` keystroke was moving the cursor **and** scrolling the pane by 1/5 viewport simultaneously — that motivated this contract). See ADR 0011's *Diff-pane motion contract* revision for considered alternatives.
- **Webapp line cursor**: persistent diff-pane focus position with the same data shape and β-coupling as the TUI cursor (cross-file scope, semantic anchor `{ file, lineNumber, side, preferredSide } | null`; `n`/`p` couples cursor + `currentAnnotationId`, `j`/`k`/`h`/`l`/arrows decouple). Materializes lazily on first user interaction (same rule as TUI). Cursor visual is a pixel-thin GitHub-style rounded outline on the active row's cell (`outline: 2px solid var(--accent); border-radius: 4px; outline-offset: -1px;`) — no layout shift, scoped to `preferredSide`'s column in split layout. Hover is a separate transient state on mouse-pointer over an annotatable row: soft bg tint + `+` gutter button via CSS pseudo-element, suppressed while the composer is open. Cursor outline + hover tint coexist visually when both land on the same row. Walks every row Pierre exposes with `[data-line]`, including unchanged context rows; context rows are addressable on **both** columns (matching GitHub's split-view) — `findAnnotatableLine` resolves `side` from the clicked column (`[data-deletions]` / `[data-additions]` ancestor), and keyboard `a` uses the cursor's `preferredSide`. Buffer rows (Pierre's `[data-content-buffer]` / `[data-gutter-buffer]` — the gray spacers where one side has more lines than the other) carry no `data-line` and stay inert, mirroring GitHub. Pierre's `expandUnchanged` (PRD #106) injects new context rows on demand — the cursor's walkable set grows for free as the user expands. Cursor anchor is invariant under bundle reload (watcher), layout toggle, and composer cancel/submit; resets to null on tour switch and when its file is removed from the bundle. Layout toggle key: the previous global `l` binding is rebound to Shift-`L` to free `l` for cursor side = additions (mirrors TUI's prior rebind so both surfaces share one keymap contract). No URL persistence — `?ann=` continues to mirror `currentAnnotationId` only; shareable line-deep-links would be a separate explicit "copy link to line" affordance. Implementation layers `data-tour-cursor` and `data-tour-hover` attributes onto Pierre's row DOM via a wrapper-level effect; CSS injected through Pierre's existing `unsafeCSS` channel — no Pierre fork. Future webapp keyboard expansion of hidden context (chevron, collapsed-file unification, analogous to PRD #107 for the TUI) is out of scope for v1 and would extend the cursor to walk Pierre's interactive rows under a successor of ADR 0013. See ADR 0012.
- **Row planner**: the TUI's diff renderer consumes a pure-data row planner in `core/diff-rows.ts` (no React, no OpenTUI dependency, fully unit-testable). The planner takes Pierre's `FileDiffMetadata` plus the file's annotations plus a `layout` and returns a flat `PlannedRow[]` of three kinds: `diff-row` (one diff line, with optional `tint` and `gutterMark` flags per side), `hunk-header`, and `annotation` (a card to render). Annotation rows are inserted directly after the row whose line number matches `line_end` on the matching `side`; multiple annotations with the same `(side, line_end)` stack in `created_at` ascending order. Side-awareness in split layout: tint paints only the column matching the annotation's `side` (`deletions` → left column, `additions` → right column); the gutter mark sits at the left edge of the matching column (column 0 for `deletions`, just right of the split divider for `additions`). In unified layout, both collapse to column 0 / single-column row tint. The TUI renders each kind with a thin React switch; the planner stays framework-agnostic to keep `core/`'s posture (cf. `file-tree.ts`).
- **Top header**: both surfaces share a single-line GitHub-style header. **Left cluster** (anchored to the left edge): bordered hamburger button (`☰`) that opens the Tour picker, then the title, then `<base_source> ← <head_source>` muted (arrow direction matches `git diff base..head`; refs come from `tour.toml`, not SHAs). **Right cluster** (anchored to the right edge): `[← N/M →]` annotation-navigation pill (hidden when `M === 0`), then `[Split | Unified]` layout toggle. The header has no border; only the hamburger has a border — the title sits naked next to it as a page heading, not inside a chip. The hamburger is icon-only (no inline label; webapp uses `title`/`aria-label`, TUI relies on the existing footer `t: tour picker` hint). When the row can't fit on one line, the right cluster wraps to its own row (still right-aligned), recovering the previous 2-row shape on narrow viewports / 80-col terminals; the title and sources clip with ellipsis when individually too long. Tour short-id (`#abc1`) is **omitted** from the header — disambiguation lives in the Tour picker (status glyph + age + count badge) and the URL / `tour list` carry the full id. Status pill (open/closed) is omitted — lifecycle isn't surfaced in v1's read-only flow. Created-at and total-files counts are omitted — recoverable from the tour picker (relative age) and sidebar (file list). Title is plain text, not clickable; the hamburger is the only picker trigger (mirrors GitHub PR pattern where `All commits ▾` is a separate widget from the PR title). Empty title falls back to `(untitled)` muted, matching the picker. Snapshot-lost banner stays directly below the header (unchanged), not inside it. The webapp's pre-existing floating `SequencePill` is relocated into the header (no longer `position: fixed`). The TUI gains a visible navigation counter for the first time — `n`/`p` keys still work in addition to clicking the pill arrows. Layout toggle in TUI gains a visible state indicator alongside the existing `l` key (previously keyboard-only with no on-screen indicator of current mode).
- **Thread rendering on existing card UI**: top-level Annotations are the unit of the sidebar count badge and the `[← N/M →]` header navigation pill — Replies are *inside* a Thread, not their own navigation targets (`n`/`p` walks topics, not utterances). The TUI's `c` collapse (commit 2a389e5) collapses Replies only, leaving the parent Annotation visible — whole-Thread collapse is reachable via the existing file-level collapse. The TUI's side-anchored card placement (commit c415152) accepts vertical bloat as Threads grow rather than re-flowing across columns; same trade-off the webapp accepts via Pierre's inline placement. Intra-Thread navigation (jumping between Replies inside one Thread) is not bound in v1; revisit if Threads routinely exceed ~5 Replies in real use.

- **Tour picker (v1)**: both surfaces open a Tour-switcher modal on `t` (top-level binding, focus-agnostic; webapp also opens via clicking the hamburger button in the top header; webapp ignores `t` when an editable element has focus). The modal lists every Tour from `.tour/` (open + closed) sorted newest-first; each row shows the status glyph (`●` open / `○` closed), relative age (`just now` / `Nm ago` / `Nh ago` / `Nd ago` / `Nw ago` / `Nmo ago` / `Ny ago`), title (`(untitled)` fallback), and an annotation-count badge when non-zero. Tour ID is hidden from the row. The current Tour is highlighted with a coloured background (webapp: `#1f6feb22`; TUI: `bg="blue"`, since `bg="cyan"` is taken by selection); the initial cursor lands on the first non-current row, or index 0 if the list is all-current. Keys inside the modal: j / ArrowDown / down, k / ArrowUp / up, Enter to commit, click row to commit (webapp), `t` and Esc to close, click scrim to close (webapp). While the picker is open, the App's input dispatch routes everything to the picker first; regular surface keys (`n` / `p` / `l`, j/k for files, Tab, Space, arrows, etc.) are inert (webapp: `n`/`p`/`l` are blocked; TUI: picker absorbs all input, no fall-through to `dispatchKey`). The modal opens even when only one Tour exists. Snapshot-lost Tours appear in the list with no special marker; selecting one falls through to the existing in-Tour banner. **TUI switch**: in-process — `cli/tui.ts` injects `loadTour(id)` and `loadTours()` callbacks; `loadTour` re-runs the existing pipeline (`getTour` + `readAnnotations` + `isShaResolvable` + `getDiff` + `parseDiff` + `classifyFile`) and the App swaps the bundle into local state without exiting; `loadTours` walks `readAnnotations` per Tour for accurate badge counts on first picker open. State reset on TUI switch: `selectedRowIdx` resets to 0, `currentAnnotationId` clears, `collapsedOverrides` clears, `collapsedFolders` clears; `sidebarFocused` and `layout` are preserved. **Webapp URL**: on user-initiated switch, `history.pushState({ tourId }, "", "/?tour=<id>")` (no `ann=` — the cursor restorer writes it once the new Tour anchors); on mount, `?tour=<id>` takes precedence over `window.__INITIAL_TOUR_ID__`; `popstate` syncs `tourId` back to the query param. The current top-level Annotation also persists as `?tour=<id>&ann=<ann-id>` via `replaceState` — chosen over `pushState` so the browser back button steps over Tour switches, not over every `n`/`p` keystroke. On mount/refresh, when the URL `ann=` resolves to a top-level Annotation in the loaded Tour, the surface restores it as if the user had just navigated there: file uncollapsed, folder ancestors expanded, card scrolled into view. A missing `ann=` (deleted, hand-edited, or pointing at a Reply — Replies aren't navigation targets) silently falls back to the first top-level Annotation and the URL is rewritten to drop the stale param, so shared links can't propagate broken state. State reset on webapp switch: `selectedFile`, `currentAnnotationId`, `collapsedOverrides`, and `collapsedFolders` are cleared; `layout` is preserved (per PRD #36). No search / filter v1 — deferred until tour counts justify it (~30+).
