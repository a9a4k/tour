# Tour

A code-walkthrough tool that pairs an ephemeral, GitHub-style diff (split or unified) with persisted AI comments. Drives the same data from a TUI and a webapp, both consuming local files written by agents through a CLI.

## Language

**Tour**:
A single guided traversal of a pinned git diff, with zero or more comments attached. Lives in `<tour-home>/<repo-key>/<id>/` (ADR 0039); the previous in-repo `.tour/<id>/` location is no longer detected or migrated. Carries a `created_in_worktree` stamp so per-worktree filtering survives the storage collapse across worktrees of one clone.
_Avoid_: review, PR, pull request, code review session, changeset

**Tour home**:
The user-scoped root holding every Tour on a machine. `$TOUR_HOME` if set, else `~/.tour/`. Replaces the previous in-repo `.tour/` location (ADR 0039) so coding agents with auto-commit can't sweep Tour internals into the user's commits and `git status` stays clean in every repo Tour has been used in. The retired `ensureTourIgnored` write to `.gitignore` is gone with the move — the repo is never touched on `tour create`.
_Avoid_: tour root, tour store, dotfile, `.tour/`

**Tour config**:
The optional user-scoped TOML file holding Tour's persistent defaults on a machine. Path: `$TOUR_HOME/config.toml`. Currently carries two scalars — `reply_agent` (default for `--reply-agent`) and `editor` (default for `--editor`). Read once at startup in `main.ts`; written by hand, not by Tour itself (no `set` / `get` subcommand). Sits **between** `$TOUR_EDITOR` and `$VISUAL` / `$EDITOR` in the editor resolution chain, and **below** `--reply-agent` in the reply-agent chain. Missing file → no defaults (resolver behaves as if pre-ADR-0042); malformed TOML or unknown keys → loud error with file path; unknown agent name → fails at the existing `assertShippedAgent` chokepoint with the file path appended to the error so the user knows where the bad value came from. Not in the repo (ADR 0039 keeps the repo clean); not per-repo (one machine-wide file). The read-only `tour config show` subcommand renders the resolved value of each key and its source (`config` / `$TOUR_EDITOR` / `$VISUAL` / `$EDITOR` / `default`); `show` survives a malformed file by reporting `(UNREADABLE: <error>)` so the diagnostic surface stays available when every other command hard-fails. See ADR 0042.
_Avoid_: preferences, settings, rc file, dotfile, defaults file, tourrc

**Repo key**:
The folder name under **Tour home** that scopes one repo's Tours. Shape: `<basename>-<short-hash>`, where `basename` is the last segment of the repo's working tree path and `short-hash` is the first 12 chars of `sha1(realpath(git rev-parse --git-common-dir))`. Worktrees of one clone share a Repo key — `git-common-dir` resolves to the same `.git/` for the main checkout and every linked worktree — which is correct because Tours anchor on SHAs reachable from every worktree. Survives `mv` of the working tree (the gitdir reference is stable). Two clones of the same repo at different paths get distinct keys; that's accepted and consistent with the per-pass model. Tour requires a git working tree (ADR 0039 amendment); commands outside one fail at the resolver and the Repo key has no non-git derivation.
_Avoid_: repo id, project key, store key, slug

**Worktree stamp**:
The `created_in_worktree` field on a **Tour**, set at `tour create` time to `realpath(git rev-parse --git-dir)` — `/path/to/repo/.git` for the main worktree, `/path/to/repo/.git/worktrees/<name>` for a linked worktree. Always git-derived; Tour requires a git working tree (ADR 0039 amendment). Drives `tour list`'s default filter (`created_in_worktree === <current worktree stamp>`); `tour list --all` bypasses. Stable across `git worktree move` (git updates the gitdir reference, the on-disk path stays the same); dies cleanly on `git worktree remove` (the gitdir is deleted, the stamp can no longer match — the Tour falls into `--all`, which is the right outcome for "I tidied up that worktree, stop showing those Tours"). Bare `tour` and `tour tui`'s smart-default consume the same stamp; explicit `tour <verb> <id>` ignores it.
_Avoid_: worktree path, created-from, source path, origin

**Diff**:
The set of file changes shown in a Tour, recomputed from git on every open. Never persisted.
_Avoid_: patch, changes, delta

**Head**:
The git ref the Tour's diff ends at. Resolved to a SHA at create time. May be a real commit or a synthetic snapshot of the working tree.
_Avoid_: tip, target

**Base**:
The git ref the Tour's diff starts from. Resolved to a SHA at create time. Defaults to `head^` for single-commit tours.
_Avoid_: parent, ancestor

**Comment**:
A note anchored to `(file, side, line_start, line_end)` inside a Tour's diff. `side` is `additions` or `deletions` (matching Pierre's legacy vocabulary; see ADR 0001); line numbers are file-line-numbers in the file as it exists on that side at the pinned SHA. Authored by agents (via CLI) or humans (via UI or CLI). The body is **GitHub Flavored Markdown** (no raw HTML); the webapp renders it as rich markdown — including ` ```mermaid ` fences as diagrams — while the TUI shows the raw source. No `kind` enum — structure lives in the markdown itself. Carries an `author_kind ∈ {agent, human}` so the system can distinguish the two without parsing the free-form `author` display string. Persisted in the Tour's folder as **Comment events** in `tour-events.jsonl` (ADR 0036); reads project the event log to the current Comment set, including a `deleted` flag carried from `comment.deleted` events and the latest body carried from any `comment.edited` events (latest-edit wins; prior bodies recoverable from the log; no surface-level `(edited)` indicator). The predecessor formats `comments.jsonl` and `annotations.jsonl` are no longer read. See ADR 0043 for the edit verb.
_Avoid_: annotation (legacy Pierre-era term; superseded by ADR 0029), review comment, note

**Reply**:
A Comment whose `thread_id` field points at the top-level Comment of its Thread. By contract `thread_id` always references a top-level Comment — Replies never reference other Replies, so a Thread is structurally flat (one root + a list of Replies, ordered by `created_at`); see ADR 0040. Inherits the root's `(file, side, line_start, line_end)` anchor — replies don't have independent positions. Renders inside the root's card. Deletion cascade unchanged (ADR 0036): a deleted Reply vanishes from the projection; a deleted root with ≥1 live Reply renders as a `[deleted]` stub; a Thread where every node is deleted vanishes entirely.
_Avoid_: comment-on-comment, sub-comment, response, nested reply

**Comment event**:
The unit of persistence in `tour-events.jsonl` (ADR 0036). One of `comment.created`, `reply.created`, `comment.deleted`, or `comment.edited`. The log is append-only; the canonical Comment set for a Tour is the projection produced by folding events at read time. Remaining future verbs (resolve, re-anchor) extend the union without changing the storage seam. Only humans emit `comment.deleted` and `comment.edited` events; agents create only.
_Avoid_: tombstone (the storage shape is uniform — `comment.deleted` is no more a tombstone than `comment.created` is)

**Thread**:
A top-level Comment plus the flat list of Replies attached to it. Structurally flat — Replies always point at the root via `thread_id`, never at other Replies; the data model does not admit nested sub-threads (ADR 0040). Lives entirely inside one Tour — Threads do not carry across Tours in v1 (preserves ADR 0003's "no re-anchoring" stance). When a Tour closes, its Threads close with it. Continuity across Tours is the agent's job (it re-references prior concerns in prose when reviewing the new SHA), not Tour's data model.
_Avoid_: conversation, discussion, exchange, tree, nested

**Reply-agent**:
An ephemeral, capability-bounded agent process spawned by the renderer on explicit user action — pressing `s` in the TUI, or clicking **Send to {agent}** on a human Comment card in the webapp. Both surfaces converge on the shared core seam `requestReply` (`src/core/reply-runner.ts`). The agent has **no tools** — it runs in non-interactive print mode and emits the reply body to stdout; Tour captures stdout and writes the Reply Comment itself. The capability boundary is "no tools," strictly stronger than the prior single-allowed-tool design: the agent literally cannot read, write, edit, or shell out. A Tour-canonical system prompt instructs the model to emit only the reply body — no preamble, no narration. Distinct from the **main-agent** — the user's existing Claude Code / Codex / etc. session — which is unrestricted and consumes the conversation later via `tour pickup`. Invocation was originally implicit (watcher fired on every new human-authored Comment, ADR 0010); reversed in ADR 0021 to eliminate silent over-dispatch on directive / WIP / test Comments the user never intended to send to the agent.
_Avoid_: bot, listener, autoreplier

**Side**:
The diff-half a Comment belongs to: `deletions` for the base-file half, `additions` for the head-file half. Required, not derived. Layout-independent: in `split` layout, `deletions` renders on the left column and `additions` on the right; in `unified` layout, `deletions` attaches to the `-` row and `additions` to the `+` row. Context rows are addressable on **both** columns — `side` follows the column the user clicked (or the cursor's `preferredSide` for keyboard-driven `a`); the prior "context → additions by convention" rule was reversed in ADR 0012's Revisions to match GitHub's split-view comment model. Naming matches Pierre's `CommentSide` so `core/` ↔ Pierre is a no-op.
_Avoid_: LEFT/RIGHT, before/after, old/new, column

**Layout**:
How a Tour's Diff is rendered: `split` (deletions left, additions right) or `unified` (single column with `-`/`+` rows). Per-session UI state in each surface — not persisted on the Tour, not synced between TUI and webapp. Comments are layout-independent: the same `(file, side, line_start, line_end)` anchor resolves in both. Maps to Pierre's `diffStyle` and OpenTUI's `view` props (both already accept `"unified" | "split"`).
_Avoid_: view mode, diff style, view

**Working-tree snapshot**:
A synthetic commit object capturing uncommitted changes at the moment a Tour is created, so the Diff stays pinned even as the working tree keeps moving.
_Avoid_: stash, WIP commit

**Tour bundle**:
Everything required to render a pinned Tour at open time — the Tour TOML, its Comments, the parsed Diff, per-side file contents, file classifications, and orphan-comment auto-windows — packaged as one value computed fresh on every open. Never persisted. Discriminated by `kind`: `ok` carries the full payload; `snapshot-lost` carries just the Tour + Comments so the surface can render the snapshot-lost banner without per-file machinery. The Reply lock is **not** part of the bundle (lock changes are O(read one file) and fetched on their own watcher event without re-running the full hydrate pipeline).
_Avoid_: payload, snapshot (collides with Working-tree snapshot), state

**Tour-session**:
The live state aggregate a single surface drives for one opened Tour. Sits in `core/tour-session.ts` next to **Tour bundle**: the bundle is the *data aggregate* (what the Tour exposes once opened), the session is the *state aggregate* (what the renderer is currently doing with that data). Per-renderer-instance, in-memory only — no on-disk presence, no cross-surface sync, gone on renderer exit / page reload. One per opened Tour; reset on Tour-switch following the per-slice rules CONTEXT.md already pins (Layout preserved, Cursor / Folds / Composer / Picker / Reply-lock cleared).

Carries the slices each App owns as parallel `useState`: `bundle`, `replyLock`, `cursor`, `collapsedFolders`, `collapsedOverrides`, `layout`, `picker`, `composer`, `expansion`. The session is **authoritative** for every slice it carries — cursor, picker, expansion, composer (tagged-union `closed | open | submitting | errored` with target + body + error folded into one slice), folds (`collapsedFolders` + `collapsedOverrides`), and layout all live in the reducer; reply-lock is the in-memory shadow of disk. The "shadow slice" concept introduced in PRD #207 was dropped in slice 2 (PRD #229) — the only candidate was the webapp's `expansion` shadowing Pierre's, and #212 retired Pierre's renderer so both surfaces now consume `core/expansion-state.ts` as the source of truth. If a future surface asymmetry needs a shadow-slice pattern, the concept can be reintroduced then.

Cross-async ordering rules — "watcher reload arrives mid-composer," "cursor re-validates after fold toggle," "URL `popstate` mirrors into cursor," "reply-lock cleared while pill ticker runs" — live in the session's reducer and **Tour-session runtime** instead of duplicating as parallel `useEffect`s in `tui/app.tsx` and `web/client/App.tsx`. The runtime owns the impure realization of every emitted intent; each surface ships a **Tour-session adapter** that exposes the platform-specific bits (OpenTUI vs DOM, in-process vs HTTP, FS watch vs SSE, no-op vs `window.history`) and nothing else. The surface itself shrinks to wiring + JSX: input events translate to `store.dispatch(...)`, the adapter handles the substrate, the runtime drives the rest.
_Avoid_: app state, view state, store, model, context

**Tour-session view**:
The rendered shape of a **Tour-session** — what surfaces draw. Pure projection from `(bundle, state)` via `deriveTourSessionView` in `core/tour-session-view.ts` (PRD #242 / issue #243); not persisted; recomputed per render; layered into `bundle` (tour + comments + files + filesByName + classifications + fileContents), `nav` (topLevel + repliesByRoot + navIndexById + navTotal + currentIdx + sendTarget), `rows` (plannedRowsByFile + flatRowsList + rowCount), `tree` (root + visibleRows + commentCounts), and `cursor` (anchor + onCard + onInteractive + cardId + cardComment + rowIdx) namespaces. Discriminated by `kind` mirroring **Tour bundle**'s `ok` / `snapshot-lost` split so the surface's banner branch stays a single `if (view.kind === "snapshot-lost")`. The view's `cursor.anchor` is the **projection-validated** cursor — `state.cursor` pruned against `flatRowsList` for fold/expansion invisibility. Structural validity is a reducer invariant, so the view never sees a structurally-stale anchor: a `CardAnchor` resolves to a live Comment, or to a deleted parent Comment only while at least one live Reply still makes the C4 `[deleted]` stub renderable. React-side consumers use `useTourSessionView(store, bundle)` which runs one `useMemo` per namespace so granular invalidation survives (cursor moves don't blow away the planner cache, fold toggles don't re-run threads). The `live*` prefix carried by `tui/app.tsx`'s `liveComments` / `liveFiles` / `liveClassifications` / `liveFileContents` / `liveTopLevel` and the webapp's parallel projections carries no semantics and is gone once slices 2 + 3 land.
_Avoid_: render model, view-model, render state, projection

**Tour-session runtime**:
The impure executor of a **Tour-session**, paired with the reducer (`core/tour-session.ts`) and the view (`core/tour-session-view.ts`) to complete the triple: the reducer is the pure state machine, the view is the pure projection, the runtime is the impure half. Lives in `core/tour-session-runtime.ts`. Subscribes to `store.onIntent` and to a **Tour-session adapter**'s tour-events stream; turns intents into adapter calls (fetch / write / scroll / mirror / request-reply) and adapter events into store dispatches (`bundle.refreshed`, `replyLock.loaded`). One per renderer instance; lifetime matches the Tour-session — constructed at App mount, torn down at unmount. The runtime is the single home for every cross-async workflow CONTEXT.md's Tour-session entry pins ("watcher reload arrives mid-composer," "cursor re-validates after fold toggle," "post-submit scroll into view") and for the stale-response guarding that drops a `loadTour` response when the user has already moved on. The previously-parallel `useEffect` chains in `tui/app.tsx` and `web/client/App.tsx` — intent listener, watcher subscription, post-submit eager refetch, inline view re-derivation for cursor validation — collapse into the runtime; the TUI's stale-closure-mitigation `useRef` cluster (`cursorRef` / `flatRowsRef` / `treeRef` / `…`) goes with them because the runtime reads `store.getState()` directly.
_Avoid_: orchestrator, controller, side-effect runner, store middleware, intent dispatcher

**Tour-session adapter**:
The seam between the **Tour-session runtime** and a renderer's substrate (the platform-specific bits: in-process callbacks vs HTTP fetch, OpenTUI vs DOM scroll, FS watcher vs SSE, in-memory history vs `window.history`). Each surface ships one **adapter** implementing `TourSessionAdapter` from `core/tour-session-runtime.ts`. Methods: `fetchBundle`, `fetchReplyLock`, `writeComment`, `requestReply`, `subscribeTourEvents`, `scrollToCard`, `scrollToRow`, `scrollToPickerRow`, `captureAnchor`, `applyAnchor`, `revealFileInSidebar`, `mirrorTourUrl`, `mirrorAnnUrl`. URL-mirror methods are no-ops in the TUI. `captureAnchor` / `applyAnchor` carry an opaque `AnchorToken` so layout-toggle and sidebar-resize reflows can preserve the cursor row's screen-y without `core/` reading DOM or OpenTUI measurements. The runtime depends only on the interface; the platform-specific implementations live next to each surface (`tui/tour-session-adapter.ts`, `web/client/tour-session-adapter.ts`). Same shape as the `ShippedAdapter` pattern in `src/agents/`: one interface, multiple concrete adapters, runtime knows none of them by name.
_Avoid_: substrate, bridge, controller, host, platform layer

**Hidden context**:
The unchanged lines between hunks in a Tour's Diff that aren't part of any hunk's visible context window (git's `-U3` default). Surfaces in three places: between consecutive hunks, before the first hunk (file-top gap, absorbed into the first hunk's `hunk-header row` when there's anything to reveal), after the last hunk (file-bottom gap, addressed via a standalone `expand-down` row with `boundaryRef: "bottom"`). The `hunk-header row` visualises the gap and is the keyboard target for revealing it — emitted **only when the gap has something to expand** (issue #359 / ADR 0025 amendment): a file whose first hunk starts at line 1 has no top-of-file banner, and once a reviewer fully expands a mid-file gap the two adjacent hunks read as one continuous stream of diff/context rows with no banner between them. For mid-file gaps larger than `2N` (N = 20 lines per direction press), a second `gap-mid-top` row appears above the `hunk-header` so each end of the gap has its own affordance. Anchor of a Comment may live inside Hidden context; revealing it is a render-time concern, not a data concern (the SHA, file, and line numbers are unchanged).
_Avoid_: hidden lines, collapsed lines, gap, missing context

**Hunk-header row**:
The synthetic row that carries a hunk's `@@ -a,b +c,d @@` metadata. Emitted by the planner **only when the gap above the hunk has something to expand** (`HunkHeaderRow` in `core/diff-rows.ts`; issue #359 / ADR 0025 amendment). When `gapAbove === 0` (first hunk starts at line 1, or two consecutive hunks abut with no hidden gap between them, or a reviewer has fully expanded the gap between adjacent hunks) the planner emits no banner — adjacent hunks read as one continuous stream and a top-of-file hunk has no banner above its first content row. When a banner is emitted (`gapAbove > 0`) the cursor walks to it and `Enter` expands the gap. Direction semantics derive from `(hunkIndex, gapAbove)`: for the first hunk, `gapAbove` is the file-top gap (lines 1 to first-hunk-start) and expansion is single-direction toward the file's start — the asymmetric merge that replaces the standalone file-top boundary row, structural because git's hunk format has headers but no footers; for mid-file hunks with `0 < gapAbove ≤ 2N` the row carries symmetric expansion (single keystroke / chevron reveals from both ends of the gap); for mid-file hunks with `gapAbove > 2N` the row carries bottom-direction expansion and pairs with a `gap-mid-top` row above. N = 20 lines per direction press across both surfaces.
_Avoid_: hunk separator, hunk-separator row, `@@` row (as a row kind)

**Gap-mid-top row**:
A synthetic interactive row emitted **only** for mid-file gaps where `gapAbove > 2N` (with N = 20, so `gapAbove > 40` lines). Sits immediately above the bottom-direction `hunk-header` of the next hunk. Enter / click expands the **top** end of the gap — newly-revealed lines appear above this row, closest to the previous hunk's end. Paired with its `hunk-header` neighbour the row gives a large mid-file gap two cursor stops and two `Enter` actions, one per end of the gap; small mid-file gaps (`0 < gapAbove ≤ 2N`) need no `gap-mid-top` because the `hunk-header`'s symmetric expand covers the whole gap in one press.
_Avoid_: gap top, top-of-gap row, mid-top boundary

**File-bottom expand-down row**:
The synthetic interactive row emitted after the last hunk's content when the file's hidden tail is non-zero. Carries `subKind: "expand-down"` and `boundaryRef: "bottom"` — the same `expand-down` family the planner uses for mid-file bottom-direction expansion. The legacy `boundary-bottom` variant was retired in issue #295 after issue #280 / ADR 0025 collapsed the file-bottom affordance onto a standalone `expand-down` row; there is no merge target on the file-bottom side (structurally asymmetric with file-top because git's hunk format has hunk-headers but no hunk-footers). Enter / click expands 20 lines toward EOF (matching the `expand-down` step size); suppressed once the gap is fully revealed. Cursor-walkable on both surfaces. No Shift modifier (PRD #270 Slice 5 / issue #275 — the whole-gap escape hatch moved to the per-file Expand-all button in the file header); reveal the entire remaining tail by repeated Enter or via the file-header Expand-all affordance.
_Avoid_: boundary-bottom row, file-bottom row, EOF row, end boundary

**Per-file Expand-all affordance**:
The `↕` affordance that reveals every hidden gap in a single file in one reducer hop. Lives in the file-header chrome on both surfaces (webapp: icon button in `<FileBlock>`'s header, per issue #274; TUI: trailing `↕` cell in the file-header line, per issue #297). Visible iff `fileExpandableGapCount(file, expansion, newContent) >= 2` (issue #298 tightened the gate from "≥ 1 gap" to "≥ 2 gaps" — with exactly one gap the per-hunk banner's `primaryExpand` cell or the file-bottom standalone `expand-down` row already covers it, and a chrome `↕` would just stack a redundant duplicate). Disappears once gap count drops below 2. Dispatch is `expansion.expandFileAll(file, boundaries)` on both surfaces — the reducer walks each boundary's full gap size in one mutation. Webapp: mouse click on the icon button. TUI: mouse click on the file-header `↕` cell, or the `e` keybinding which targets the file the cursor is anchored in (RowAnchor / CardAnchor: cursor.file; sidebar fallback: the selected file row). `e`'s dispatch semantics are unchanged by #298 — `e` stays useful any time there's something to expand (reducer no-ops when no gaps remain); the ≥ 2 gate is scoped to the visible chrome, which is the discoverability surface. Issue #297 retired the planner-emitted standalone `expand-file-all` row that previously occupied the top of each TUI file's diff body — the affordance now lives on one logical location per file on both surfaces, structurally distinct from the per-hunk `↑` / `↓` / `↕` banner buttons. The reducer action signature and semantics are unchanged from PRD #270 / issue #274.
_Avoid_: expand-file-all row, per-file expand banner, expand-all banner row

**Expand context**:
The act of revealing some or all of a stretch of Hidden context. Mechanism is per-surface but the row stream is shared — both surfaces consume the same planner output (`core/diff-rows.ts`) and emit the same gap-row family (`hunk-header` when interactive, `gap-mid-top`, `expand-down`). **Webapp**: Tour-owned gap-row affordances overlay-injected onto Pierre's DOM; click calls Pierre's `expandHunk(hunkIndex, direction, expansionLineCount)` via a per-file `FileDiff` ref. Tour does not mirror Pierre's expansion state — Pierre stays the source of truth for which lines are currently revealed, and Tour re-renders so the planner recomputes gap counts from Pierre's updated `FileDiffMetadata`. **TUI**: cursor walks the same gap-row family; `Enter` routes through `core/expansion-state.ts` reducers to reveal lines, `Shift+Enter` expands the whole gap. The cursor's comment-anchor scope is unchanged — `a` (top-level annotate) is a no-op when the cursor sits on any interactive row. The same `Enter` primitive also expands collapsed-by-classifier files (lockfiles / generated / vendored): the file-collapsed indicator becomes a cursor-walkable interactive row, unifying "expand hidden context" and "expand hidden file" under one gesture. **D1 directional convention** (both surfaces): the row's position == which end of the gap it expands. Top row → top of gap → lines appear above. Bottom row → bottom of gap → lines appear below. D1 intentionally diverges from GitHub's chevron-direction convention (where the chevron-glyph points at the half being revealed); D1 is chosen because it's spatially direct and surface-consistent — newly-revealed lines always appear adjacent to the row that revealed them, and the same logical row stream renders cleanly in both OpenTUI's row model and Pierre's DOM grid. Step size N = 20 lines per direction press, matching Pierre's `expansionLineCount: 20` and the TUI's prior symmetric-20 semantics; `Shift+Enter` / shift-click bypasses the step and expands all. Orphan comments (anchors that fall in Hidden context) are auto-windowed at planner-init time with a ±10 line region around each anchor so the sidebar count and the diff stream stay honest. State is per-renderer-session (in-memory): preserved across watcher-driven bundle reloads (diff is SHA-pinned), reset on tour switch and webapp page reload. Prior versions of this entry claimed the webapp used Pierre's `expandUnchanged: true` — that mechanism auto-renders the whole file and was reverted in `c20a263`; the current mechanism is the Tour-owned gap rows described above.
_Avoid_: load context, show context, unfold

**Cursor**:
A persistent diff-pane focus position used as the anchor for new top-level Comments and as the action target for `r`/`s`/`Enter`. One unified cursor walks the full concatenated diff stream across files (cross-file scope, mirroring how the diff pane scrolls), landing on **diff rows, interactive rows, AND Comment cards** — three first-class cursor stops under one anchor (PRD #192 / ADR 0022, motion model revised by ADR 0023). Two motion gestures, NOT two destination-filtered lanes: `j`/`k` is the **step** gesture — one row per press, no filter on destination kind, so a press from a row whose `line_end` matches a comment lands the cursor ON the card; stacked cards (multiple top-level comments at the same anchor) each count as one step. `n`/`p` is the **jump** gesture — one top-level Comment per press, regardless of intervening rows. Cards are valid stops for both gestures; the two differ in distance per press, not in destination filter (ADR 0023 supersedes ADR 0022's "two-lane" rule per issue #200). Tagged-union state: `RowAnchor { file, lineNumber, side, preferredSide, interactive? }` for diff/interactive rows, `CardAnchor { commentId, preferredSide }` for cards. Both carry `preferredSide` so an `h`/`l` choice survives step-across-card and jump-between-cards. Interactive rows are the `hunk-header` (when its `gapAbove > 0`), `gap-mid-top`, `expand-down` (at the file-bottom position via `boundaryRef: "bottom"`), and `collapsed-file` synthetic rows (PRD #107 / ADR 0013 / ADR 0018); `hunk-header` with `gapAbove === 0` (first hunk at line 1, or hunks abutting with no hidden gap, or a fully-expanded mid-file gap) is not emitted by the planner at all (issue #359 / ADR 0025 amendment), so the walker never sees it. The previously-emitted `boundary-top` row is gone — its file-top semantics fold into the first hunk's `hunk-header`. Action keys dispatch by row kind: `a` opens the top-level composer on a row (no-op on a card/interactive); `r` opens the Reply composer on a card (no-op on a row); `s` sends to the reply-agent on a card (no-op on a row); `Enter` dispatches `primary-action` only on an interactive row. Both surfaces (TUI + webapp) implement the cursor with progressive disclosure: cursor is null on tour load; the first user interaction (`j`/`k`/`h`/`l`/arrows/`a`/`n`/`p`/click) materializes it at the first top-level Comment card if any exist, else at the first annotatable row of the first non-folded file (initial position never lands on an interactive row by default). The **TUI footer always previews the cursor's action target** — e.g. `r: reply to "<truncated title>"` when on a comment card, `r: — (no comment under cursor)` when on a diff row, with a direction hint (`(cursor ↑ above viewport)`) when free-scroll has parked the cursor outside the viewport — so the user reads what an action will do before pressing it. **Scroll on cursor move** has two independent axes — *placement* (where the frame lands: `nearest` vs `center`) and *behavior* (how it gets there: `instant` vs `smooth`) — carried on the `scrollCursorTarget` intent and forwarded to the adapter by `core/tour-session-runtime.ts` (PRD #348). The per-gesture contract is: `n` / `p` comment-jump → `center + smooth` (every jump frames mid-viewport with a perceptible tween so adjacent landings keep a predictable focal point); `j` / `k` step and click → `nearest + smooth` (spatial gestures — the target stays where the user pointed / where keyboard motion was carrying it, animated for travel-distance feedback, with the TUI's 3-row scrolloff margin governing the `nearest` snap); fresh landings (tour-open `cursor.materialize`, URL `?ann=` restore, `r` / `s` auto-recall, post-submit `scrollToComment`) → `center + instant` (no prior frame of reference to preserve, so the target frames immediately). The reducer fills in `behavior` from a `center → instant, nearest → smooth` default when a `cursor.set` dispatch omits the field. **Auto-recall on action** is the at-action mitigation on *both* surfaces: when `r` / `s` fires and the cursor is on a card that's off-screen, the renderer scrolls the card to centre before opening the composer / dispatching the agent (`center + instant` on both surfaces). Visual treatment differs by row kind: rows wear a `❯` glyph + solid-plate bg in the TUI / a GitHub-style rounded outline in the webapp (side-scoped in split); cards wear the three-cue active treatment (heavy border + brighter bg + leading `●` glyph) — same pixels, now meaning "cursor is here". See ADR 0011 (TUI), ADR 0012 (webapp), ADR 0013 (interactive-row family), ADR 0022 (unified cursor), ADR 0023 (step/jump motion).
_Avoid_: caret, point, position, selection, line cursor (the cursor is no longer line-only), annotation cursor (replaced by row-kind dispatch on the single cursor), row lane, card lane, β-coupling, `currentAnnotationId`

**Pane focus**:
Tour-session state slice (`"sidebar" | "diff"`) routing keyboard input between the sidebar tree and the diff pane on both surfaces. Sibling to **Cursor** in `core/tour-session.ts`; the reducer owns transitions (`paneFocus.setSidebar`, `paneFocus.setDiff`, `paneFocus.toggle`). Initial value is conditional and follows the existing seed effect: tours with Comments → `"diff"` (cursor seeds at first Comment via `initialCursor`); tours with no Comments → `"sidebar"` (cursor stays null, sidebar selection lands at the first file row). `Esc` toggles pane focus with modal-unwind precedence — composer open closes composer, picker open closes picker, otherwise Esc flips paneFocus. `Enter` activates the focused row inside the current pane: in diff on an interactive row → `primary-action` (expand hidden context); in sidebar on a file row → select-file + flip to diff; in sidebar on a folder row → toggle fold (no flip, sidebar-internal). Action-target-driven auto-flip rule: `n`/`p` (Comment jump), click on diff row/card → flip to diff; click on sidebar row → flip to sidebar; pane-agnostic actions (`t`/`L`/`e`/`y`/`q`) and sidebar-internal actions (`j`/`k`/`l`/`h`/folder-Enter) don't flip. Diff-pane-only actions (`c`/`r`/`s` — comment / reply / send) are silently gated from sidebar mode; pressing `c` while sidebar-focused is a no-op, the user must Esc to diff first. Cursor and sidebar-row selection are independent slices preserved across flips — Esc-toggle-Esc returns the user to the exact prior state (cursor at its prior file:line, sidebar selection at its prior row). Webapp realises paneFocus via the W3C ARIA tree-widget roving-tabindex pattern — exactly one sidebar row carries `tabindex="0"` at a time, the adapter calls `.focus()` on it when `paneFocus.setSidebar` fires and `.blur()` when it flips back; browser `Tab` walks the sidebar as one tab stop, not N, preserving WCAG 2.1.1 / 2.4.3 keyboard-nav contracts. TUI realises paneFocus via accent-border color flip on the focused pane container; no DOM focus concept applies. Visual treatment on the webapp uses two redundant cues: row-level `:focus-visible` outline (a11y signal, browser-native) plus a 2px accent left-border on the sidebar container (glanceable pane-level cue, mirrors TUI). Replaces the TUI's prior `sidebarFocused` boolean + `Tab`/`Shift-Tab` keymap (ADR 0031 retires both on a pre-1.0 semver license). Footer legend is paneFocus-aware via `composeFooterHints({ surface, paneFocus, ... })` (ADR 0028 amended): sidebar mode emits a shorter sidebar-relevant legend (`j/k: file · h/l: fold · Enter: activate · …`); diff mode emits today's legend minus the retired `Tab: pane` plus `Esc: sidebar`. **Sidebar visibility** is a paired tour-session slice (`sidebarVisible: boolean`, default `true`) governing whether the sidebar column renders at all; invariant `!sidebarVisible ⇒ paneFocus === "diff"`. Toggled by the top-header button (webapp: Primer's `SidebarCollapseIcon`/`SidebarExpandIcon`, far-left, before the hamburger; TUI: bordered `[⇤]`/`[⇥]` cell in the same position) and by capital `B` on both surfaces — both routes have pure-visibility semantics: no focus change, and if hiding while `paneFocus === "sidebar"` the reducer snaps to `"diff"` in the same transition (the invariant). Esc precedence gains one branch (between modal-unwind and pane-flip): when `!sidebarVisible`, Esc unhides AND sets `paneFocus = "sidebar"` in one transition, so Esc remains the unified "give me sidebar focus" command regardless of visibility. Sidebar-row selection and folder fold state survive hide/unhide (they live in independent slices already preserved across paneFocus flips). Preserved across tour-switch (matches Layout per PRD #36); per-renderer-instance, in-memory only, no on-disk presence, no cross-surface sync. When hidden, the sidebar column is removed from the layout (not visibility-hidden in place) and the diff pane reflows to fill the freed width. Footer-hint composer gains a `sidebarVisible` parameter and emits `B: hide sidebar` / `B: show sidebar` accordingly; the diff-mode legend's existing `Esc: sidebar` text remains correct in both states. See ADR 0031, ADR 0038.
_Avoid_: pane, focus (overloaded), active pane, sidebar mode, active region, `sidebarFocused` (TUI implementation predecessor), sidebar collapsed (visibility ≠ folder-collapse)

**Footer**:
One-line muted strip at the bottom of both surfaces. Two jobs: (b) **transient status** — silent-failure surface for `s` no-op reasons (lock held, wrong author kind), comment-create failures, and other actions that don't visibly repaint; auto-dismiss ~2s, last-write-wins (no queue) — and (c) **keybinding legend** — the canonical key surface for that renderer. Status prepends onto the legend (`${status}  ·  ${legend}`) without layout shift. Legend's `s: send to {agent}` is conditional on `--reply-agent` + cursor-on-human-card + lock-free (same predicate on both surfaces). Legend content is per-surface: the TUI ships the full key list (`j/k · h/l · n/p · a · r · s · Enter · e · c · y · Space · L · t · Tab · [/] · q`) and clips at the right on narrow terminals; the webapp ships only the bound-keys subset (`j/k · h/l · n/p · a · r · s · L · t`) and wraps to a second line below ~1100px viewport width. Truthful display beats single-line consistency on the webapp; the TUI's prepended action preview survives any clip so the legend tail is safe to lose. Action-preview content (`r: reply to "<title>"` + `(cursor ↑/↓ off-viewport)` suffix per ADR 0022) is **TUI-only** — the webapp substitutes auto-recall scroll-into-view per `src/web/client/cursor-keymap.ts:79`. Implementation: `composeFooterHints({ surface, replyAgent, showSendHint })` in `core/footer-hints.ts`; surface-only keys live behind the `surface` switch. Transient status timing is owned by `useFlashFooter()` in `src/core/use-flash-footer.ts`, shared by both surfaces. Webapp renders the footer as a `<footer>` flex sibling at the column-flex root (not `position: fixed`); status slot wraps in `<span aria-live="polite" aria-atomic="true">` so screen-reader users hear silent-failure reasons that auto-recall and watcher repaints can't convey. Picker-open and composer-open keep the footer painted and unchanged on both surfaces. See ADR 0028.
_Avoid_: status bar, hint bar, key hints, help strip

**Highlighting**:
Syntax highlighting of Diff content. Tokenisation lives in `core/syntax-highlight.ts` and is engine-shared (Shiki + Oniguruma) across both surfaces; the contract is `tokenize(content, lang) → TokenLine[]` where `TokenLine = { chunks: { text, color?, bold?, italic?, underline? }[] }` (one TokenLine per source line). Surface-specific paint adapters fan out from this shape — the webapp emits inline-styled HTML spans (`src/web/client/syntax-paint.ts`), the TUI emits OpenTUI `StyledText` chunks (`src/tui/syntax-paint.ts`). Grammar coverage is the full Shiki bundled set (~200 languages) via `shiki/bundle/full`; the `EXT_TO_LANG` map in `core/syntax-highlight.ts` is the single curated mapping from file extension to bundled grammar id, and files whose extension isn't in the map fall to plain text. Grammars load lazily per-lang on first view of a file; the per-`(content, lang)` memo + per-lang `subscribe` machinery flips the React tree from plain-text fallback to styled output once the grammar resolves. The italic-comment overlay applies in `core/` (Shiki's `github-dark-default` does not emit italic on comment scopes; `core/syntax-highlight.ts` promotes `italic: true` on tokens whose TextMate scope contains `comment`) so both surfaces gain the same comment treatment. The TUI requires a truecolor terminal (`COLORTERM=truecolor|24bit`) for colour rendering and falls back to plain text otherwise — wrongly-mapped 256-colour rendering is foreclosed (unhighlighted is preferable to wrongly-highlighted). See ADR 0033 (cross-surface tokeniser, alternatives, tradeoffs) and ADR 0009 (per-cell `DiffLine` rendering shape; ADR 0009's `<code>`-renderable consequence is superseded by ADR 0033).
_Avoid_: syntax (overloaded with diff syntax / row syntax), colourisation, theming (theming is the palette concern, separate)

**Text selection**:
Native mouse selection of rendered Tour text for ordinary clipboard copy, distinct from the Tour **Cursor** and from `y`'s structured yank target; Tour preserves platform-native drag / double-click / triple-click semantics instead of reimplementing them.
_Avoid_: selection (ambiguous with Cursor), highlight (ambiguous with syntax Highlighting and Comment range tint), yank (the `y` keybinding is a separate structured copy action)

## Relationships

- A **Tour** has exactly one **Head** and one **Base**, both stored as SHAs.
- A **Tour** has zero or more **Comments**.
- A **Comment** belongs to exactly one **Tour** and anchors to one file + line-range inside that Tour's **Diff**.
- A **Reply** is a Comment that points at another Comment in the same Tour via `replies_to`.
- A **Thread** is a top-level Comment plus its chain of Replies; lives entirely inside one Tour.
- A **Working-tree snapshot** acts as a synthetic **Head** when an agent creates a Tour of uncommitted work.
- **Text selection** is a native copy affordance layered over the rendered surface. The webapp keeps immediate single-click Cursor movement even when the first click becomes part of a later double-click; the second click must not clear or override the browser's native selected text. Drag selection must not activate Tour controls on mouse-up.
- **Text selection** covers content text (Diff code, hunk text, Comment and Reply bodies, location stamps, file paths, sidebar labels) but excludes pure chrome (gutters, diff symbols, cursor glyphs, bullets, chevrons, buttons, badges, layout controls, footer hints).
- In the TUI, finishing a mouse **Text selection** copies the selected text immediately through the existing clipboard transport and flashes a footer status.

## Example dialogue

> **Dev:** "If the agent amends the commit after creating a **Tour**, do the **Comments** still line up?"
> **User:** "The **Tour** is pinned to the original SHA. Amending creates a new SHA — the old one is still in git's object store, so the **Diff** and **Comments** still resolve. If the agent wants to tour the amended version, they create a new **Tour**."

## Resolved decisions

- **Tour storage location**: user-scoped (`<tour-home>/<repo-key>/<id>/`), not repo-scoped (`<repo>/.tour/<id>/`). Driven by (a) `tour create` modifying the user's `.gitignore` on first use, and (b) the eager-gitignore approach being a race against coding agents that auto-commit between edit and tour-create. Moving storage to `~/.tour/` makes it structurally invisible to repo-scoped tools — no gitignore needed, no race possible. Worktrees of one clone now share a Repo key (correct: SHAs reachable from every worktree); per-worktree filtering survives via the `created_in_worktree` stamp on every Tour. `ensureTourIgnored` is retired with the move. See ADR 0039.
- **Comment lifetime**: per-tour-pass. Comments are not re-anchored across rebases or amendments. Stale Tours are abandoned, not migrated.
- **Diff source**: commit-pinned only. Working-tree tours are supported by snapshotting to a synthetic commit at create time, so the rest of the system only ever sees SHAs.
- **Multiplicity**: many Tours coexist, ordered by creation time. Default open behavior surfaces the most recent unfinished one.
- **Tool surface**: CLI-first. Agents drive the system by shelling out (`tour create`, `tour annotate`, …) with `--json` output. MCP is deferred and would be a thin wrapper if added later.
- **Comment anchor**: `(file, side, line_start, line_end)` with `side ∈ {additions, deletions}` (matching Pierre's `CommentSide`) and line numbers being file-line-numbers at the pinned SHA. View-mode independent; both split and unified consume the same shape. No `kind` enum in v1 — the body is plain text.
- **Authoring**: both agents and humans author Comments. Top-level Comments and Replies use the same primitive (`tour annotate`, optionally with `--reply-to <ann-id>`). Each Comment carries `author_kind ∈ {agent, human}`; the free-form `author` string is kept for display. Both surfaces (TUI + webapp) gain a composer for top-level notes and replies. The CLI is canonical and remains the agent path; the UI is the human path. Originally v1 said "agents only, UI is read-only"; that was reversed once Threads landed (see ADR 0010). Resolution state is **not** modeled — readers (the main-agent at `tour pickup` time, or another human) infer outcomes from natural language. No `agree:`/`dismiss:` prefix conventions, no status enum. **Body must be non-empty after trimming**: whitespace-only bodies reject at the Comment creation seam (`createComment` / `createReply` / `createComments`), uniformly for every surface — CLI, TUI, webapp, reply-runner (PRD #140 / slice 2 #142). For the reply-runner specifically, an empty agent stdout is treated as a normal dispatch completion (lock clears via the standard release path, ADR 0015); the runner records a rejection entry in the per-dispatch log and skips the write so no empty Reply Comment lands on disk. **Author defaults to the `author_kind` literal**: when a caller omits `author`, the seam falls back to `"human"` if `author_kind === "human"` and `"agent"` if `author_kind === "agent"`. No environment-dependent fallback — no OS username, no per-surface constant. The CLI's `--author "..."` flag (and the webapp's optional `author` body field) lands verbatim when supplied. The TUI's pre-seam `humanAuthor()` helper and the webapp's `DEFAULT_HUMAN_AUTHOR = "you"` constant are gone; both surfaces simply omit `author` and let the seam decide (PRD #140 / slice 3 #143). See ADR 0016 for the no-environment-leak rationale. **Anchor must resolve inside the Tour's Diff at write time**: `createComment` and `createComments` take a required `bundle: TourBundle` argument and reject any request whose `file` is not in `bundle.files` or whose `(line_start, line_end)` falls outside `1 ≤ line_start ≤ line_end ≤ lineCount(file, side)` (head side for `additions`, base side for `deletions`). Hidden-context anchors (between or outside hunks) stay legal — `orphan-window` handles their render-time placement. Unchanged-context-row anchors stay legal under the existing `additions`-by-convention rule. Replies inherit their anchor from the parent and skip validation; `createReply` does not take a bundle. The CLI loads the bundle once per invocation (single or `--batch`); the TUI passes its in-memory bundle through; the webapp loads inline per POST. Pre-existing bad anchors on disk are not retroactively validated — the seam only acts at write time (PRD #140 / slice 4 #144). See ADR 0017 for the rationale.
- **Comment editing**: humans-only body-only mutation of a Comment via the `comment.edited` event (`{target_id, body, at}`). Anchor edits (file / side / line range) are out of scope and reserved for a future `re-anchor` verb (ADR 0036). The seam refuses agent authors, deleted targets, empty-after-trim bodies, and identical-after-trim bodies — same uniform protocol as the create / delete paths. Targets any author's body (a human can edit an agent-authored Comment, mirroring how a human can delete one). Allowed regardless of Reply state. Projection: the latest `comment.edited` event's body replaces the prior body in the fold; prior bodies are recoverable from `tour-events.jsonl` but never surfaced — no `(edited)` indicator on the cards, no `edits` field in `tour pickup --json`. Cross-surface concurrent edit is last-write-wins at the seam, matching the existing watcher posture; both edits land as events. CLI: `tour comment <tour-id> --edit <comment-id> --body <s>`, mutually exclusive with `--file/--side/--line`, `--reply-to`, `--delete`, and `--batch`; `--as-agent --edit` is refused at parse-time. Surfaces (TUI + webapp) reuse the existing comment composer in an `edit` mode pre-filled with the current body; the cursor stays on the card after submit. TUI: `e` becomes contextual — card → open edit composer, row → expand-all-in-file (existing behaviour); footer preview disambiguates. Webapp: ✏️ button beside the 🗑 in the card header, hidden on `[deleted]` stubs. See ADR 0043.
- **Sidebar (tree)**: folder tree of changed files. Path compression folds any non-leaf node with exactly one folder child into its child (lazygit-style), so sparse tours don't waste rows on empty intermediate folders. Folders sort before files at each level; alphabetical within each group. File rows render an Octicon-based file-status icon (added/modified/deleted/renamed, sourced from `@primer/octicons-react`'s diff icon set via the `web/client/icons.ts` shim and the `web/client/file-icon.ts` `type → { Icon, statusClass }` mapping) and a per-file comment count badge. Folder rows render a chevron Octicon (`ChevronRightIcon` when collapsed, `ChevronDownIcon` when expanded) followed by `FileDirectoryFillIcon` and the compressed path — GitHub-PR-tree-shaped, with one folder icon in both states (the chevron carries the open/closed signal). Per-depth indent is `16px` for both row kinds, giving the chevron + folder-icon group enough horizontal breathing room. The TUI sidebar mirrors this noise-reduction (no folder rollup, no inline classification-reason tag) and keeps its single-letter status glyph and per-file count. The classification reason still surfaces in the main-pane file header on both surfaces — only the sidebar duplicate is removed. The pure shared module `core/file-tree.ts` is unchanged: `flatten` keeps producing `VisibleRow[]` with `commentCount` on every row; the surfaces simply ignore it on folder rows. Default state on first render: every folder expanded. Clicking a folder toggles its expand/collapse; clicking a file selects it, scrolls the diff stream to it, and moves the cursor to its first annotatable row — both surfaces obey this rule (clicking a file expresses "show me from the top"). On the TUI a file-row click also shifts focus from the sidebar to the diff pane (folder-row clicks keep focus on the sidebar — folder clicks are tree-internal navigation, not a "go to this file" gesture). On the webapp the comment-focus bookmark (`?ann=`) is left untouched by a file click — viewing a file is not the same as losing your comment position. `n`/`p` comment navigation reveals (expands) the ancestor folder chain of the target file. Expand/collapse state is React state, in-memory only, per renderer instance, per tour — page reload resets to "all expanded" (no localStorage, no on-disk state). The tree algorithm lives in the pure shared module `core/file-tree.ts`; both the webapp and TUI consume `VisibleRow[]` from `flatten`. Originally deferred from v1 ("Real folder-tree rendering is deferred until file counts justify it") to ship the smallest viable Tour; reinstated now that the v1 surface is shipped and real tours expose the flat list's limitations on deeply-nested paths. Main pane is unchanged: a single top-to-bottom stream of all file diffs; clicking a sidebar file entry scrolls to that file. Cursor-driven scroll-into-view derives both axes from the reducer's `scrollCursorTarget` intent's `placement` discriminator — anchor-kind-agnostic on both surfaces (issue #296): `placement: "nearest"` for in-flight navigation (`n`/`p`, `j`/`k`, click-to-position, bundle-refresh re-snap) so adjacent comments don't re-center jarringly, and `placement: "center"` for fresh landings (initial materialize, URL `?ann=` restore, send-to-agent recall, tour-switch) so the user arriving cold gets the target framed in the middle. The webapp adapter realises placement as the browser's `scrollIntoView({ block, behavior })` — `block: mode` directly, and `behavior` paired (`nearest` → `smooth` to convey travel distance, `center` → `instant` for cold landings). The TUI adapter realises placement by selecting the scroll helper: `center` → `centerChildInView` (instant, content frame math), `nearest` → `animatedScrollChildIntoView` (a 200ms `outQuad` tween via OpenTUI's Timeline engine). The post-create `scrollToComment` intent (webapp + TUI) and the sidebar file-click path stay instant on both surfaces — file-click is a random-access jump that can span the entire diff and would slog under smooth, and the new-comment scroll uses `placement: "center"` by construction. Issue #296 retired the TUI's prior surface-side `[cursor, layout]` useEffect that picked the helper by anchor kind (`CardAnchor` → center, `RowAnchor` → nearest); the layout-flip re-scroll survives as a thin `[layout]`-only effect (`layout.set` doesn't emit `scrollCursorTarget`). Per-surface caveat: the TUI cursor scroll uses a 200ms `outQuad` tween on `nearest` placement, unconditionally — there's no opt-in or opt-out (issues #294 + #299). The whole sidebar column is hide-able via the top-header toggle (capital `B`); when hidden, the diff pane reflows to fill the freed width and the tree's fold/selection state is preserved across reveal — see the **Pane focus** entry above and ADR 0038 for the visibility ↔ focus interaction.
- **Watch behavior**: TUI and webapp watch the open Tour's `<tour-store-root>/<id>/` folder for changes and re-render. The watcher fires on `tour-events.jsonl` (event log writes — comment creates, replies, deletes) and `.reply-lock.json` (in-flight reply state, see Reply-agent dispatch below). They do not watch git refs or working-tree files — the diff is pinned to `head_sha`.

- **Reply-agent dispatch**: explicit, user-triggered. When the user presses `s` on a focused human Comment in the TUI, or clicks the **Send to {agent}** affordance in the webapp, the surface calls the shared core seam `requestReply({ cwd, tourStoreRoot, tourId, commentId, agent? })` (TUI: in-process; webapp: through `POST /api/tours/:id/request-reply`). The seam validates the precondition (human-authored, no existing Reply, atomic `<tour-store-root>/<id>/.reply-lock.json` acquire), spawns the shipped TS adapter (`src/agents/` — claude / codex / gemini / opencode / pi), captures stdout, and writes the Reply Comment. System prompt is owned by Tour and built in-process; the adapter passes it as a flag value to its inner CLI (e.g. claude's `--system-prompt`) — no external fetch channel. The capability boundary is **zero tools**: the inner CLI runs with no `--allowedTools` / `--disallowedTools` / equivalent allow-list, and the agent's only output is stdout, which Tour captures verbatim as the Reply Comment body (see ADR 0012). Copilot CLI was cut from MVP (its capability-bounding surface isn't designed for single-tool-only enforcement; see ADR 0010); the registry is fixed and custom out-of-tree adapters are not supported. The watcher's role narrows to state observation only: `tour-events.jsonl` writes drive bundle re-render; `.reply-lock.json` changes drive the in-flight pill and the disabled state of every Send affordance on the Tour. Without `--reply-agent`, the Send affordance is hidden on every card and Tour functions as a static review tool — humans can still author, the conversation just sits until consumed via `tour pickup`. The auto-dispatch path of ADR 0010 is reversed in ADR 0021; the rest of ADR 0010 (reply-agent vs main-agent split, pinned-SHA invariant, no resolution status, no cross-Tour Threads) stands.

- **Reply streaming**: while a reply-agent is in flight, the renderer paints a status pill below the targeted Comment ("✏️ `<agent>` is replying… (Ns)") sourced from `.reply-lock.json`. When the agent's `tour annotate --reply-to` write lands and the lock releases, the pill is replaced atomically by the rendered Reply. Token-by-token streaming was rejected in favour of this lockfile-based status because it composes with the single-flight lock we already need (zero new IPC channel) and captures the actual user need ("is the agent on it?") for ~5% of the implementation cost. Orphan locks (renderer crashed mid-flight) auto-clear on read via a PID-liveness check; no user action needed. Stale locks (>2min, agent alive but hung) flip the pill to a single-line warning; recovery is to restart the renderer. See ADR 0015.

- **Conversation pickup**: `tour pickup <id> --json` emits the full conversation tree as JSON for consumption by the user's main-agent session. The schema is the projected Comment set (folded from `tour-events.jsonl` per ADR 0036) reshaped into a tree — each top-level Comment carries its `replies` array, and a comment whose `deleted` field is set surfaces as a `[deleted]` stub with the body redacted. No status field, no actionable-only filter, no markdown digest in v1 — Tour emits raw conversation; the main-agent does all reasoning. Triggered by the human (or asked of the main-agent) when they want code changes acted on; Tour itself never moves the SHA mid-conversation.
- **Handoff**: agent prints the tour ID to stdout, and a one-line "Open with: …" hint to **stderr** (so `TOUR_ID=$(tour create --head HEAD)` captures the id cleanly; the hint still reaches an interactive TTY). No auto-open. Humans open the UI via bare `tour` (smart-default: webapp on a graphical desktop, TUI for ssh / non-TTY / win32; ADR 0019), `tour tui` (forces TUI), or `tour serve` (forces webapp). Auto-launching the browser lives on `tour serve --open`; bare `tour` prints the URL but does not auto-open (ADR 0019). `tour create` itself is launch-free.
- **Tour ID**: `YYYY-MM-DD-HHMMSS-xxxx` (UTC timestamp + 4-char random suffix). Sortable by creation time; collision-safe under parallel creates; prefix-matchable (`tour show 2026-05-08-1`).
- **CLI surface**: `tour create --head <ref> [--base <ref>] [--title <s>]` resolves both ends to SHAs at create time. `--head WIP` snapshots uncommitted changes. `--base` default probes `<head>@{upstream}` (HEAD@{upstream} for `WIP`): if found and the branch is ≥2 commits ahead of upstream, the merge-base is used (matches GitHub's PR-diff scope); otherwise falls back to `<head>^` (or `HEAD` for `WIP`) — same shape as before. Detached HEAD, no configured upstream, or any other resolution failure also falls back. Explicit `--base <ref>` always wins. `base_source` records the literal resolution label (`HEAD^`, `HEAD`, `merge-base(<tip>@{upstream})`, or the user's verbatim flag). Issue #201. Verbs: `create`, `annotate`, `list`, `show`, `close`, `delete`, `prune`, `tui`, `serve`, `pickup`. `tour annotate` accepts `--reply-to <ann-id>` for replies, `--edit <comment-id> --body <s>` for body-only edits (humans-only, mutually exclusive with the create / reply / delete / batch flag families; see ADR 0043), and an `--as-agent` / `--as-human` pair that sets `author_kind` (default: `agent`). `tour tui` and `tour serve` accept `--reply-agent <name>` to enable live agent replies; without it the watcher is inert. Bare `tour` opens the best surface for the environment (ADR 0019): webapp when a browser is reachable on a desktop with a TTY, TUI otherwise (ssh, non-TTY, win32, no opener). `tour tui` always forces the TUI.
- **Tour home / Repo key / Worktree stamp**: Tour data lives outside the user's repo at `<tour-home>/<repo-key>/<id>/`. **Tour home** is `$TOUR_HOME` when set, otherwise `~/.tour`. **Repo key** is `<slug>-<short-hash>`: the slug comes from the repo's working-tree basename, and the 12-char hash is `sha1(realpath(git rev-parse --git-common-dir))`. Linked worktrees of one clone therefore share a Repo key. **Worktree stamp** is `realpath(git rev-parse --git-dir)`; new Tours persist it as `created_in_worktree`. Tour requires a git working tree; commands outside one fail at the resolver.
- **On-disk schema**: `tour.toml` holds `id`, `title?`, `status`, `created_at`, `created_in_worktree`, `closed_at?`, `head_sha`, `base_sha`, `head_source`, `base_source`, `wip_snapshot`. `tour-events.jsonl` is the append-only **Comment event** log (ADR 0036) — one event per line, kind-discriminated: `comment.created` carries `{id, file, side, line_start, line_end, body, author, author_kind, at}`; `reply.created` carries `{id, replies_to, body, author, author_kind, at}` (anchor inherited from the parent at fold time); `comment.deleted` carries `{target_id, at}` and is humans-only; `comment.edited` carries `{target_id, body, at}` and is humans-only, rejected at the seam when the target is already deleted, the body is empty-after-trim, or the body is identical-after-trim to the current projected body (ADR 0043). The current Comment set is the projection produced by folding events at read time, with the C4 cascade applied (deleted leaf vanishes, deleted parent with surviving replies projects as a `[deleted]` stub, fully-deleted Thread vanishes). `author` is a free-form display string (e.g. `"agent"`, `"claude-code"`, `"almas"`); `author_kind` is the structured discriminator (`"agent"` | `"human"`) used by the reply-agent dispatch predicate, required on every authoring event. Predecessor on-disk formats (`comments.jsonl`, `annotations.jsonl`) are no longer read. `.reply-lock.json` (transient) holds `{agent, responding_to, started_at, pid}` while a reply-agent is in flight.
- **Webapp lifecycle**: `tour serve` is long-running, single-user, binds to `127.0.0.1:8687` (TOUR on a T9 keypad; configurable via `--port`), no auth. `Ctrl-C` to stop. Browser auto-open only with `--open`. **Single-instance per cwd** (ADR 0020): before binding, `tour serve` probes the preferred port via `GET /__alive`; if a Tour server is already serving the same `cwd`, it prints the existing URL and exits 0 instead of starting a duplicate. If the default port is busy with anything else (a different-cwd Tour, a non-Tour process), the server walks up (8688, 8689, …) until it finds a free port and prints the bound URL; with an explicit `--port <N>` no fallback occurs — busy ports fail loudly (unless they're a same-cwd Tour, in which case reuse still applies).
- **Snapshot loss**: if a Tour's `head_sha` no longer resolves (e.g. user manually ran `git update-ref -d refs/tour/<id>`), the UI shows a banner "Snapshot lost — comments preserved but diff cannot be displayed." Read-only, can be deleted.
- **Packaging**: single binary, multi-subcommand (same shape as Backlog.md). One `tour` package containing a shared `core/` (git, tour IO, comments, watcher) and two renderers (`tui/`, `web/`). Stack: Bun + TypeScript + OpenTUI for TUI; Bun HTTP server + React for web.
- **Distribution**: npm package is `tourdiff`; installed binary is `tour`. Per-platform sub-packages `tourdiff-darwin-arm64`, `tourdiff-darwin-x64`, `tourdiff-linux-arm64`, `tourdiff-linux-x64`, `tourdiff-windows-x64` are pulled in via pinned `optionalDependencies` (only the matching one installs on a given OS/arch). The parent ships a `scripts/cli.cjs` shim that resolves the right binary at runtime — exact mirror of `backlog.md`'s install model. GitHub Releases also attach the same 5 binaries as a second distribution path. No telemetry, no update-check, no outbound network calls. Release pipeline is a GitHub Actions workflow on `vX.Y.Z` tag push: matrix-builds 5 binaries with `bun build --production --compile`, smoke-tests `./tour --version` per-platform, then publishes to npm with `--provenance` (granular token, publish-only, scoped to `tourdiff` + `tourdiff-*`). Pre-1.0 semver: minor=breaking, patch=compat. Repo `github.com/a9a4k/tour`, license MIT.
- **Diff engine (v2)**: `@pierre/diffs` parses git diffs in both surfaces via `parsePatchFiles` only; its React renderer, worker pool, embedded stylesheet, and comment-slot framework are no longer used (ADR 0024 supersedes ADR 0001's renderer claim and ADR 0013's webapp chevron-UI claim). Both surfaces walk Pierre's parsed `FileDiffMetadata` row-by-row through a Tour-owned **row planner** in `core/diff-rows.ts`; comment cards are first-class entries in the row stream (`kind: "card"`), not slot-filled by a callback. The **TUI row renderer** is built per-cell with row-flex columns and OpenTUI's `<code>` renderable for `tree-sitter` syntax highlighting (ADRs 0007, 0009); `@opentui/core`'s `<diff>` and `hunkdiff/opentui`'s `HunkDiffView` were rejected because neither exposes a slot for inline comment content between rows. The **web row renderer** (ADR 0024) is a per-row `<div>` using CSS subgrid (each row inherits column widths from the file-level grid), with comment cards as plain `<div>` siblings between rows; syntax highlighting is main-thread Shiki, tokenized lazily per file via `IntersectionObserver` (`rootMargin: 200px`) and memoized per session; a fixed common-language grammar set is bundled eagerly. Both surfaces converge on the same `core/expansion-state.ts` reducer for hidden-context expansion — click (web) and `Enter` (web + TUI) dispatch the same action. Cursor walks `core/flat-rows.ts` directly on both surfaces; the previous web DOM-walking cursor was a Pierre-renderer artifact and is gone. Accepted coupling: Pierre's parser is the only Pierre surface left; if it churns or archives, a parser swap is bounded to `core/diff-model.ts` per ADR 0001's promise.
- **Lifecycle**: two states, `open` and `closed`. Human transitions via `tour close` or UI button. Closed tours stay in `<tour-store-root>/<id>/` with `status = "closed"`. Cleanup is explicit (`tour prune --older-than 30d`).
- **Folder layout**: `<tour-store-root>/<id>/` flat under the repo key; each Tour contains `tour.toml` (front-matter) and `tour-events.jsonl` (append-only Comment event log per ADR 0036; predecessor `comments.jsonl` / `annotations.jsonl` formats are no longer read). `tour create` does not write `.tour/` or `.gitignore` in the user's repo. A repo-root `.tour/` directory is treated as opaque user content.
- **Working-tree snapshot mechanics**: `git stash create` produces a synthetic commit SHA without touching the working tree; `git update-ref refs/tour/<id> <sha>` keeps it alive past gc. Stored as `head_sha` in `tour.toml`. Released on `tour delete` / `tour prune` via `git update-ref -d refs/tour/<id>`.
- **Comment body format**: GitHub Flavored Markdown, no raw HTML. The webapp renders it rich (including ` ```mermaid ` fences as SVG diagrams); the TUI shows the raw source unchanged. Single string field — no `kind` enum, no separate `format` field. Existing comments were already authored with markdown syntax; this decision honours that intent rather than introducing a new format.
- **Layout (v1)**: per-session, in-memory toggle between `split` and `unified`. Both surfaces default to `split` on first paint, matching the project's GitHub-style identity. Pressing `l` flips the layout in either surface (the webapp also exposes a segmented `[ Split | Unified ]` control in the top bar). State lives in `useState` on each renderer's `App`; not persisted (no localStorage, no `tour.toml` field, no on-disk state) and not synced across surfaces — page reload (webapp) or relaunch (TUI) returns to `split`. A single layout value applies to every Tour switched to in the same webapp session; layout is not stored on the Tour. Pierre's `diffStyle` and OpenTUI's `view` props consume the same `"split" | "unified"` vocabulary, so no translation layer.
- **Webapp comment rendering**: comments render inline as first-class entries in the row stream emitted by the row planner (`kind: "card"`); each card is a plain `<div>` sibling between row `<div>`s in the file block, sized either full-width (unified layout) or side-anchored to the column matching `Comment.side` (split layout — `deletions` left, `additions` right) via `grid-column` on the file-level CSS subgrid. Each row in `[line_start, line_end]` carries two visual cues: a subtle blue background tint (`theme.bg.accentRange.web`) painted on the row's subgrid container, and a 3px accent-coloured gutter stripe at the left edge of the matching column (side-aware — `addition` rows for `additions` comments, `deletion` rows for `deletions` comments). The comment card is bordered in the same `#58a6ff` accent — connecting card and range via shared colour. Mirrors the TUI's two-cue pattern (see ADR 0008); the surfaces share the accent colour and the side-aware placement. The **currently-selected card** (the `n`/`p` target — `isCurrent` true) reads differently from its peers along three redundant axes so the cue survives palette drift, colour blindness, and low-contrast displays: the top/right/bottom borders flip from neutral grey to the accent (left edge was already accent), an `8px` drop shadow lifts the card off the page, and a leading `●` accent glyph appears in the header — matching the TUI's three-axis treatment. Replaces the previous Pierre-`FileDiff`-with-`lineComments`-and-`renderComment`-callback shape (ADR 0024 supersedes ADR 0001's webapp rendering claim).
- **TUI comment rendering**: comments render inline in the diff stream, directly below their `line_end` row, mirroring the webapp's "below the highlighted area" pattern. Each row in `[line_start, line_end]` carries two visual cues: a subtle blue background tint (matching the webapp's `RANGE_TINT`, applied full-width on context rows and to the line-number column only on `+`/`-` rows so the diff bg survives) and a single accent-colored gutter character (`▎`) in column 0. The comment card is a bordered `<box>` rendered after the `line_end` row, with `borderColor` set to the same accent — connecting card and range via shared colour rather than literal box-drawing brackets. **Placement** is side-anchored in split layout (the card occupies only the half matching `Comment.side` — `deletions` → left column, `additions` → right column — with the opposite half empty so the card sits under the range it discusses, mirroring `@pierre/diffs`'s `<FileDiff>` placement on the webapp) and full-width in unified layout. The **currently-selected card** (the `n`/`p` target — `isCurrent` true) reads differently from its peers along three redundant axes so the cue survives palette drift, colour blindness, and low-contrast displays: a `heavy` border style (vs `single` on peers), the brighter `accentCurrent` background tier (vs the subtler `accentSubtle`), and a leading `●` glyph in the header row — a dedicated marker analogous to the line cursor's gutter `❯`. Peers keep their accent border colour so they still read as comments, but visibly recede. Hunk's docking-bracket apparatus was rejected as bespoke and high-cost; tint-only was rejected as single-cue and accessibility-weak. See ADR 0008.
- **TUI comment navigation**: `n` / `p` is the **jump** gesture (ADR 0023 / PRD #192): walks the canonical top-level Comment list — the same order the `[N/M]` pill counter reads — one entry per press, regardless of intervening rows. The cursor lands ON the target card as a `CardAnchor`, the card scrolls into view (`scrollIntoView({ block: "center" })` on `comment-${ann.id}`), and the active-card visual treatment lights up so a subsequent `r` (reply) or `s` (send) operates on the card the user just navigated to. Files with comments are auto-expanded (existing rule in `core/file-classifier.ts`); a manually-collapsed file hides its comments along with its diff body. The bottom comment panel from the pre-inline TUI is removed — all comment reading happens in the diff stream. Complement to the **step** gesture (`j`/`k`): cards are valid stops there too, so walking onto a card by `j` or by `n` produces the same CardAnchor — `r` / `s` work either way.

- **TUI Cursor** (renamed from "TUI line cursor" under ADR 0022): persistent diff-pane focus position used to seed the top-level comment composer with an arbitrary `(file, side, line_start, line_end)` anchor and to drive `r` / `s` against the focused card. Walks the full concatenated diff stream across files — diff rows, interactive rows, and Comment cards are all valid stops (`j`/`k` step / `n`/`p` jump; ADR 0023). Carries an explicit `preferredSide` on both `RowAnchor` and `CardAnchor` so an `h`/`l` choice survives row moves AND step-across-card / jump-between-cards — so commenting on three deletion-side lines in a row is `h` once + `a` three times, not `h+a` × 3; on rows where the preferred side isn't available, the effective side is forced to the only valid one (e.g. an addition-only row is always `additions` regardless of preference). Initial state is null (cursor is invisible on tour load); first user interaction (`j`/`k`/`h`/`l`/arrows/`a`/`n`/`p`/click) materializes the cursor at the first top-level Comment card if any exist, else at the first annotatable row of the first non-folded file. `a` on a null cursor materializes the cursor at the default row target AND opens the composer in one step; `r` on a null cursor is a no-op with the footer hint. Lazy materialization keeps the cursor accent from competing with the diff content on first paint and matches the webapp cursor's default (ADR 0012). Sidebar file-click moves the cursor to that file's first annotatable row (clicking a file expresses "show me from the top"); bundle reload (watcher fires) preserves cursor when its anchor still resolves (the card's `commentId` survives `tour-events.jsonl` re-reads / re-folds; the row's anchor survives re-parses). State shape is a tagged union: `RowAnchor { file, lineNumber, side, preferredSide, interactive? }` (same vocabulary as `Comment`'s anchor, so cursor → composer is a direct copy) or `CardAnchor { commentId, preferredSide }` (PRD #192 / ADR 0022). The `flatRows` traversal index is derived on demand; the anchor itself is invariant under layout toggle, fold/unfold, and bundle reload. Layout toggle (`L`) preserves cursor anchor and auto-scrolls the new visual position into view; composer cancel/submit doesn't move the cursor. Visual treatment is row-kind-aware: rows wear a leading `❯` glyph + solid-plate bg (`theme.bg.cursorRow.tui`, ≈30% accent) painted across both the gutter and content cells of the active-side `DiffLine`; cards wear the three-cue active treatment (heavy border + brighter bg + leading `●` glyph). Composition rule on rows: cursor bg > comment tint > +/- bg > syntax bg, so the cursor row reads as a single saturated plate even when it sits inside a comment range or on a `+`/`-` row. Side-scoped in split (each side is its own 50%-width `DiffLine`, so `cursorActive` paints only the active column), full-row in unified. Keys: `j`/`k`/`down`/`up` step one row (including cards); `n`/`p` jump one top-level Comment; `h`/`l`/`left`/`right` toggle side on paired diff rows in split layout (no-op on cards / interactive / single-side rows — preferredSide preserved); click-to-position with side derived from clicked column on diff rows, click-on-card lands the cursor on the card. The previous global `l` binding for layout toggle is rebound to `L` (Shift-L) to free the lowercase pair for cursor side. **Footer-preview rule**: the bottom hint line always renders the cursor's action target — `r: reply to "<truncated title>"` on a card, `r: — (no comment under cursor)` on a row, with `(cursor ↑ above viewport)` / `(cursor ↓ below viewport)` appended when free-scroll has parked the cursor outside the visible area. The footer is the off-screen affordance; no separate banner. See ADR 0011 (TUI), ADR 0022 (unified cursor + footer preview), ADR 0023 (step/jump motion).
- **TUI diff-pane motion**: the cursor floats in the document; the diff pane is a passive follower (vim / helix / VS Code / IntelliJ / emacs model). `j`/`k` step the cursor one row at a time through the flat stream (diff, interactive, and card stops alike per ADR 0023); the pane scrolls one row when the cursor would otherwise cross a 3-row edge margin (vim-style `scrolloff`). `n`/`p` jumps follow the unified placement rule (issue #293 / #296): in-flight gestures emit `scrollCursorTarget` with `placement: "nearest"`, so an adjacent card already in the viewport doesn't trigger a jolt-recenter; an off-viewport card scrolls into view via the adapter's `scrollChildIntoView` (TUI) / `scrollIntoView({ block: "nearest" })` (webapp). Fresh landings (initial materialize, send-to-agent recall, URL `?ann=` restore on the webapp) emit `placement: "center"` and frame the target mid-viewport on both surfaces. `Space`/`Shift-Space` and the hardware `PageDown`/`PageUp` keys scroll one viewport AND advance the cursor so its screen-relative position is preserved, keeping the keyboard's home base reachable (pressing `r`/`a` after `Space` acts on a visible row/card, not a stranded one). `Home`/`End` jump the cursor to the first/last cursor-eligible row in the concatenated diff stream, with the pane scrolling to match. Mouse wheel is pure pane scroll — the cursor stays at its anchor; the next `j`/`k`/`n`/`p` snaps the pane back via `block:nearest`, so keyboard motion remains "magnetic" to the cursor. The keyboard-vs-wheel asymmetry reflects two input modalities preserving their respective home bases (cursor for keyboard, pointer for mouse) and matches every floating-cursor document tool surveyed; list-view tools that pin the cursor to a fixed screen row (lazygit lists, fzf, k9s, htop) don't have this asymmetry, but Tour's diff pane is a code-reading document, not a list. When wheel-scroll parks the cursor off-screen, the **footer-preview rule** of the Cursor entry surfaces the action target so `r` / `s` remain legible without recall — no sticky off-screen banner. Implementation: the `focused` prop is removed from the diff `<scrollbox>`; the React keymap is the single source of truth for diff-pane keys, preventing OpenTUI's built-in `ScrollBar.handleKeyPress` from firing in parallel with the keymap (the parallel-handler bug — every `j`/`k` keystroke was moving the cursor **and** scrolling the pane by 1/5 viewport simultaneously — that motivated this contract). See ADR 0011's *Diff-pane motion contract* revision for considered alternatives, and ADR 0022 for the wheel-vs-action-target reconciliation.
- **Webapp Cursor** (renamed from "Webapp line cursor" under ADR 0022): persistent diff-pane focus position with the same data shape as the TUI cursor — a tagged anchor `RowAnchor { file, lineNumber, side, preferredSide, interactive? } | CardAnchor { commentId, preferredSide } | null` — cross-file scope. The prior `currentCommentId` is gone; the cursor itself walks comment cards via the same `n`/`p` jump as the TUI (ADR 0023). `j`/`k`/arrows step every cursor-eligible row including cards; `n`/`p` jumps between top-level Comments regardless of intervening rows; `h`/`l` toggle side on paired diff rows. Materializes lazily on first user interaction (same rule as TUI). Cursor visual on diff rows is a pixel-thin GitHub-style rounded outline on the active row's cell (`outline: 2px solid var(--accent); border-radius: 4px; outline-offset: -1px;`) — no layout shift, scoped to `preferredSide`'s column in split layout; on cards the three-cue active treatment (accent borders + drop shadow + leading `●` glyph). Hover is a separate transient state on mouse-pointer over an annotatable row: soft bg tint + `+` gutter button via CSS pseudo-element, suppressed while the composer is open. Cursor outline + hover tint coexist visually when both land on the same row. Walks every row Pierre exposes with `[data-line]`, including unchanged context rows; context rows are addressable on **both** columns (matching GitHub's split-view) — `findAnnotatableLine` resolves `side` from the clicked column (`[data-deletions]` / `[data-additions]` ancestor), and keyboard `a` uses the cursor's `preferredSide`. Buffer rows (Pierre's `[data-content-buffer]` / `[data-gutter-buffer]` — the gray spacers where one side has more lines than the other) carry no `data-line` and stay inert, mirroring GitHub. Pierre's `expandUnchanged` (PRD #106) injects new context rows on demand — the cursor's walkable set grows for free as the user expands. Cursor anchor is invariant under bundle reload (watcher), layout toggle, and composer cancel/submit; resets to null on tour switch and when its file is removed from the bundle. Layout toggle key: the previous global `l` binding is rebound to Shift-`L` to free `l` for cursor side = additions (mirrors TUI's prior rebind so both surfaces share one keymap contract). URL persistence — `?ann=` mirrors the cursor's `commentId` when the cursor is a `CardAnchor` and is absent otherwise (the row-anchored cursor has no URL form in v1); shareable line-deep-links would be a separate explicit "copy link to line" affordance. Implementation layers `data-tour-cursor` and `data-tour-hover` attributes onto Pierre's row DOM via a wrapper-level effect; CSS injected through Pierre's existing `unsafeCSS` channel — no Pierre fork. Future webapp keyboard expansion of hidden context (chevron, collapsed-file unification, analogous to PRD #107 for the TUI) is out of scope for v1 and would extend the cursor to walk Pierre's interactive rows under a successor of ADR 0013. See ADR 0012 (webapp), ADR 0022 (unified cursor), ADR 0023 (step/jump motion).
- **Row planner**: the TUI's diff renderer consumes a pure-data row planner in `core/diff-rows.ts` (no React, no OpenTUI dependency, fully unit-testable). The planner takes Pierre's `FileDiffMetadata` plus the file's comments plus a `layout` and returns a flat `PlannedRow[]` of three kinds: `diff-row` (one diff line, with optional `tint` and `gutterMark` flags per side), `hunk-header`, and `comment` (a card to render). Comment rows are inserted directly after the row whose line number matches `line_end` on the matching `side`; multiple comments with the same `(side, line_end)` stack in `created_at` ascending order. Side-awareness in split layout: tint paints only the column matching the comment's `side` (`deletions` → left column, `additions` → right column); the gutter mark sits at the left edge of the matching column (column 0 for `deletions`, just right of the split divider for `additions`). In unified layout, both collapse to column 0 / single-column row tint. The TUI renders each kind with a thin React switch; the planner stays framework-agnostic to keep `core/`'s posture (cf. `file-tree.ts`).
- **Top header**: both surfaces share a single-line GitHub-style header. **Left cluster** (anchored to the left edge): bordered hamburger button (`☰`) that opens the Tour picker, then the title, then `<base_source> ← <head_source>` muted (arrow direction matches `git diff base..head`; refs come from `tour.toml`, not SHAs). **Right cluster** (anchored to the right edge): `[← N/M →]` comment-navigation pill (hidden when `M === 0`), then `[Split | Unified]` layout toggle. The header has no border; only the hamburger has a border — the title sits naked next to it as a page heading, not inside a chip. The hamburger is icon-only (no inline label; webapp uses `title`/`aria-label`, TUI relies on the existing footer `t: tour picker` hint). When the row can't fit on one line, the right cluster wraps to its own row (still right-aligned), recovering the previous 2-row shape on narrow viewports / 80-col terminals; the title and sources clip with ellipsis when individually too long. Tour short-id (`#abc1`) is **omitted** from the header — disambiguation lives in the Tour picker (status glyph + age + count badge) and the URL / `tour list` carry the full id. Status pill (open/closed) is omitted — lifecycle isn't surfaced in v1's read-only flow. Created-at and total-files counts are omitted — recoverable from the tour picker (relative age) and sidebar (file list). Title is plain text, not clickable; the hamburger is the only picker trigger (mirrors GitHub PR pattern where `All commits ▾` is a separate widget from the PR title). The webapp's pre-existing floating `SequencePill` is relocated into the header (no longer `position: fixed`). The TUI gains a visible navigation counter for the first time — `n`/`p` keys still work in addition to clicking the pill arrows. Layout toggle in TUI gains a visible state indicator alongside the existing `l` key (previously keyboard-only with no on-screen indicator of current mode).
- **Thread rendering on existing card UI**: top-level Comments are the unit of the sidebar count badge and the `[← N/M →]` header navigation pill — Replies are *inside* a Thread, not their own navigation targets (`n`/`p` walks topics, not utterances). - **Tour picker (v1)**: both surfaces open a Tour-switcher modal on `t` (top-level binding, focus-agnostic; webapp also opens via clicking the hamburger button in the top header; webapp ignores `t` when an editable element has focus). The modal lists every Tour from the current `<tour-store-root>/` (open + closed) sorted newest-first; each row shows the status glyph (`●` open / `○` closed), relative age (`just now` / `Nm ago` / `Nh ago` / `Nd ago` / `Nw ago` / `Nmo ago` / `Ny ago`), title (`(untitled)` fallback), and a comment-count badge when non-zero. Tour ID is hidden from the row. The current Tour is highlighted with a coloured background (webapp: `#1f6feb22`; TUI: `bg="blue"`, since `bg="cyan"` is taken by selection); the initial cursor lands on the first non-current row, or index 0 if the list is all-current. Keys inside the modal: j / ArrowDown / down, k / ArrowUp / up, Enter to commit, click row to commit (webapp), `t` and Esc to close, click scrim to close (webapp). While the picker is open, the App's input dispatch routes everything to the picker first; regular surface keys (`n` / `p` / `l`, j/k for files, Tab, Space, arrows, etc.) are inert (webapp: `n`/`p`/`l` are blocked; TUI: picker absorbs all input, no fall-through to `dispatchKey`). The modal opens even when only one Tour exists. Snapshot-lost Tours appear in the list with no special marker; selecting one falls through to the existing in-Tour banner. **TUI switch**: in-process — `cli/tui.ts` injects `loadTour(id)` and `loadTours()` callbacks; `loadTour` re-runs the existing pipeline (`getTour` + `readComments` + `isShaResolvable` + `getDiff` + `parseDiff` + `classifyFile`) and the App swaps the bundle into local state without exiting; `loadTours` walks `readComments` per Tour for accurate badge counts on first picker open. On mount/refresh, when the URL `ann=` resolves to a top-level Comment in the loaded Tour, the surface restores it as if the user had just navigated there: file uncollapsed, folder ancestors expanded, card scrolled into view. A missing `ann=` (deleted, hand-edited, or pointing at a Reply — Replies aren't navigation targets) silently falls back to the first top-level Comment and the URL is rewritten to drop the stale param, so shared links can't propagate broken state. State reset on webapp switch: `selectedFile`, the cursor (both row- and card-kind anchors), `collapsedOverrides`, and `collapsedFolders` are cleared; `layout` is preserved (per PRD #36).
