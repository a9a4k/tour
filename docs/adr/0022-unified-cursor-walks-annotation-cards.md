# Unified cursor walks annotation cards

The TUI and webapp collapse their two-cursor model — line cursor + `currentAnnotationId` — into a **single Cursor** that walks diff rows, interactive rows, **and annotation cards** as first-class stops. Action keys are dispatched by the cursor's row kind: `a` annotates on a diff row, `r` / `s` reply / send on an annotation card, `Enter` expands on an interactive row. The previous **β-coupling rule** (`n`/`p` moves both cursors; `j`/`k` moves only the line cursor) is gone; with a single cursor and two navigation lanes (`j`/`k` for rows, `n`/`p` for cards), there is no coupling left to be asymmetric about. The free-scroll hazard — wheel-scrolling the cursor off-screen and then pressing `r` to reply to a now-invisible target — is mitigated by a **footer-preview rule**: the bottom hint line always renders the cursor's current action target, including an off-screen direction hint when free-scroll has parked the cursor outside the viewport. This supersedes ADR 0011's β-coupling and "line cursor skips annotation cards" rules; the rest of ADR 0011 (lazy materialization, `preferredSide`, the magnetic-snap motion contract, the `❯` glyph + bg plate visual) stands unchanged.

Field evidence: with `currentAnnotationId` stored separately from the line cursor, a wheel-scroll that left `currentAnnotationId` parked on an off-screen annotation while a different annotation card sat centred in the viewport made `r` reply to the wrong thread. The two-cursor model's "code reading vs annotation engagement are two distinct modes" framing was defensible on paper but produced a WYSIWYG violation in practice: the user looked at thread A and replied to thread B with no on-screen warning before the keystroke committed.

## Considered options

- **Status quo: two cursors with β-coupling.** Rejected. The split between line cursor and `currentAnnotationId` is invisible to the user — both render with strong active treatments (`❯` glyph and heavy-bordered card) — so the model conflates two pieces of state under one apparent affordance. The desync hazard is silent: nothing on screen tells the user that the wheel-scroll just decoupled the card their eye is on from the card their `r` will hit.

- **`r` reads a derived "centred annotation" instead of stored state.** Rejected, after grilling. The derived value (closest-to-viewport-midpoint with hysteresis) would have fixed `r`/`s` semantics without touching the motion contract, but it introduces a second concept ("the centred thing") alongside the cursor — and the cursor concept already wants to be the action target. Two concepts where the user pictures one. Rejected in favour of widening the cursor's domain to include cards.

- **Cursor walks cards + cursor follows wheel-scroll** (lazygit-mode). Rejected. Would couple cursor and viewport on every scroll event, eliminating the desync structurally — but at the cost of the motion contract Tour deliberately picked from Vim / Helix / VS Code / tig / delta. Tour is a code-reading tool that happens to host threads, not a list-of-threads tool that happens to show code; the document-editor lineage is the right alignment for the diff pane. Free-scroll-without-moving-the-cursor is a feature for code reading and is preserved.

- **Cursor walks cards + footer-preview of action target** (selected). The single cursor model the user expects ("`r` replies to where my cursor is") combined with the motion contract Tour already has ("wheel = pure pane scroll, cursor stays at its anchor"). When the wheel parks the cursor off-screen, the footer renders `r: reply to "<title>" (cursor ↑ above viewport)` so the action target is legible without a recall keystroke. Validated against the prototype at `src/tui/__prototype-r-target__.tsx` on 2026-05-12 — footer-only preview was reported sufficient; no sticky off-screen banner needed.

- **Sticky off-screen banner near viewport edge** (e.g. `↑ Cursor at annotation 19/19 (above)`). Rejected as the primary affordance. Visually intrusive in a TUI where every line of chrome matters, and redundant once the footer already names the target. Kept on the bench as a future option if real use shows the footer line is too peripheral.

## Consequences

- **Single state shape, two-case anchor.** The cursor's state shape becomes a tagged union — `{ kind: "row", file, lineNumber, side, preferredSide } | { kind: "card", annotationId } | null` — replacing the prior pair `(LineCursor | null, currentAnnotationId: string | null)`. `n`/`p` writes the `card` case; `j`/`k`/`h`/`l`/click-on-row writes the `row` case; click-on-card writes the `card` case. The row case carries `preferredSide` (meaningful only on paired diff rows); the card case carries only the annotation id (a card has no side ambiguity — it renders on the side dictated by its anchoring annotation).

- **Two navigation lanes, one cursor.** `j`/`k` step among diff/interactive rows (skip cards); `n`/`p` step among annotation cards (skip rows). Both write the same cursor, so a `j` from a card lands on the next diff/interactive row in the stream, and an `n` from a diff row lands on the next card. No lane-mode state, no `Tab`-to-switch — the keymap is the disambiguator.

- **Action keys are row-kind-dispatched, not anchor-shape-coerced.** On a diff row, `a` opens the top-level composer at the cursor's anchor and `r` / `s` are no-ops (footer hint reads `r: — (no annotation under cursor)`). On a card, `r` opens the reply composer for that card, `s` sends that card to the reply-agent, and `a` is a no-op (no diff anchor). On an interactive row, `Enter` / `Shift+Enter` dispatch the existing `primary-action` / `primary-action-all` per ADR 0013. No keystroke fires a misdirected action; the worst case is a no-op with a hint.

- **Footer-preview is the TUI's off-screen *pre-action* affordance.** The bottom hint line always renders the cursor's action target. Concretely: `r: reply to "<truncated title>"` when the cursor is on a card, `r: — (no annotation under cursor)` when on a diff row, plus a parenthesised direction (`(cursor ↑ above viewport)` / `(cursor ↓ below viewport)`) when free-scroll has parked the cursor outside the viewport. The TUI uses this rule; the webapp has no equivalent footer line and relies on the at-action affordance below.

- **Auto-recall is the *at-action* affordance on both surfaces.** When the user presses `r` or `s` and the cursor (kind = card) is off-screen, the renderer scrolls the cursor's card into view *before* opening the composer / dispatching the agent. Webapp: `scrollIntoView({ block: "center", behavior: "smooth" })` on the cursor's card element; the composer mounts after the scroll settles (`scrollend` event, with a 250 ms timeout fallback for browsers where `scrollend` is unsupported). TUI: instant scroll-to-row, since there is no smooth-scroll primitive — by the time the composer mounts the cursor's card is on-screen. The user's mental model is "I press `r`, the card I'm replying to comes into view, then the composer opens" — the action *reveals* its target as it commits. No surprise replies to off-screen cards on either surface. Sticky banners and modal "are you sure?" prompts are explicitly rejected — auto-recall + footer-preview (TUI) cover the visibility gap without new chrome.

- **β-coupling is removed.** With a single cursor, the asymmetric rule "`n`/`p` moves both, `j`/`k` moves only the line cursor" has no referent. The TUI annotation navigation entry in `CONTEXT.md` is rewritten accordingly: `n`/`p` simply move the cursor to the next/prev card. `j`/`k` simply move the cursor to the next/prev row. The motion contract's wheel rule is preserved verbatim.

- **The card-active visual treatment moves under the Cursor umbrella.** The three-cue language that previously marked "this card is `currentAnnotationId`" (heavy border + brighter bg + leading `●`) is now reused as "the cursor is on this card." Same pixels, different referent. The `❯` glyph + bg plate remains the on-row cursor cue. The two visual languages don't co-occur on a single row (the cursor is on exactly one row kind at a time).

- **URL persistence in the webapp**: `?ann=<id>` now mirrors the cursor's `annotationId` when the cursor is a `kind: "card"` anchor and is absent when the cursor is a `kind: "row"` anchor. The row-anchored cursor has no URL form in v1; deep-linking to a specific line is a separate "copy link to line" affordance kept on the bench. On mount, when a URL `ann=` resolves to a top-level Annotation in the loaded Tour, the cursor materializes at that card (kind = card). A missing / stale `ann=` falls back to the existing rule (first top-level Annotation, URL rewritten to drop the param).

- **State reset on Tour switch.** Both surfaces clear the cursor on Tour switch (re-seeds lazily on first interaction with the new Tour). Replaces the prior "clear `currentAnnotationId`" rule. Sidebar / layout state is preserved as before.

- **No data-model changes.** `Annotation` and `tour.toml` are untouched. `replies_to` continues to identify Reply Annotations. The change is entirely renderer-side state.

- **Migration is trivial.** Both surfaces fold `currentAnnotationId` into the cursor state in a single commit per surface: when a card-anchored cursor would have been set, write the `card` anchor; when a row-anchored cursor would have been set, write the `row` anchor. The TUI's `jumpToAnnotation` and the webapp's annotation-jump handlers become single setters on the unified cursor. The composer-seed paths (TUI: `openReplyComposer`, `openTopLevelComposer`; webapp: equivalents) read the cursor and branch on its `kind`.

- **Reversibility.** Reverting to the two-cursor model is a state-shape unsplit and a re-introduction of `currentAnnotationId` alongside the line cursor — no on-disk changes. The prototype at `src/tui/__prototype-r-target__.tsx` records the design rationale and can be referenced if the reversal is ever considered.
