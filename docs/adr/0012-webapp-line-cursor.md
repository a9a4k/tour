# Webapp line cursor for arbitrary-line annotation + GitHub-style focus

> **Scope:** Webapp-only — companion to ADR 0011 (TUI line cursor). ADR 0011's Decisions section is amended in lockstep (lazy materialization rule, see ADR 0011 Revisions).

ADR 0011 explicitly deferred this surface: *"The webapp does not gain a cursor. Its Composer continues to seed from Pierre's click event. Adding a cursor on the webapp later would need a separate ADR — the keyboard-first vs mouse-first division is a feature, not a temporary asymmetry."* This ADR closes that deferral. Reviewers using the webapp on a keyboard-only laptop, or who navigate primarily with `n`/`p`, have no way to position an arbitrary-line top-level annotation without reaching for the mouse. The mouse-first path itself is also under-affordanced — annotatable rows announce themselves with a 1-pixel `cursor: pointer` and nothing more. We add a keyboard cursor for the keyboard-first path AND upgrade the mouse-hover affordance for the mouse-first path, both targeting the same `[data-tour-focus]`-style row primitive.

## Decisions

The cursor is **persistent within the session** when the diff pane has scope, **lazily materialized** (null on tour load; first interaction triggers materialization), **cross-file** (walks the entire concatenated diff stream, mirroring ADR 0011), and carries an explicit **`preferredSide`** toggled by `h`/`l`/`←`/`→`. `j`/`k`/`↓`/`↑` move row-wise; `a` opens the top-level composer at the cursor anchor (materializing the cursor first if null).

`n`/`p` annotation navigation moves both cursor and `currentAnnotationId` to the target Annotation's anchor; `j`/`k`/`h`/`l`/arrows do not move `currentAnnotationId` — same **β-coupling** ADR 0011 locked in for the TUI. Lazy materialization is **back-ported to the TUI** in lockstep (ADR 0011 Revisions): both surfaces start with cursor null, both materialize at the same target on first interaction (first top-level Annotation if any exist, else first annotatable row of first non-collapsed file).

**Cursor visual** is a pixel-thin GitHub-style rounded outline on the active row's cell (`outline: 2px solid var(--accent); border-radius: 4px; outline-offset: -1px;`), scoped to `preferredSide`'s column in split layout. **Hover state** is a separate transient layer: soft bg tint + `+` gutter button rendered via CSS pseudo-element on hover, suppressed while the composer is open. Cursor outline and hover tint coexist visually when both land on the same row (outline reads as primary; tint reads underneath).

**Side selection** uses `h` / `l` / `←` / `→`, identical to TUI. The webapp's pre-existing global `l` binding for layout toggle is rebound to Shift-`L`, mirroring ADR 0011's TUI rebind so both surfaces share one keymap contract.

The cursor walks every row Pierre exposes with `[data-line]`, **including unchanged context rows**. `a` on a context row annotates with side `additions` (CONTEXT.md convention). Today's `findAnnotatableLine` silently dropped context rows; this ADR extends it to handle them so cursor and click resolve identically. State shape is the same semantic anchor ADR 0011 uses: `{ file, lineNumber, side, preferredSide } | null`. The Pierre `flatRows` traversal index is derived on demand; the anchor itself is invariant under layout toggle, fold/unfold, bundle reload, and Pierre's `expandUnchanged` chevron-expansion (PRD #106) — newly-revealed context rows carry `[data-line]` and join the walkable set automatically.

**No URL persistence.** `?ann=` continues to mirror `currentAnnotationId` only; the cursor is session-only state. Reload resets the cursor to null (consistent with progressive disclosure). A future explicit "copy link to line" affordance is a separate decision and would not necessarily live in the URL on every keystroke.

**Implementation**: a wrapper-level React effect layers `data-tour-cursor="true"` / `data-tour-hover="true"` (with `data-tour-cursor-side` for split-layout column scoping) onto Pierre's row DOM. CSS targets are injected via Pierre's existing `unsafeCSS` channel — no Pierre fork.

## Considered Options

- **Hover-only (no keyboard cursor)** — rejected. The user explicitly cited GitHub's keyboard-navigation box as the reference; halving the scope would lose keyboard parity with the TUI without simplifying the visual language (the hover state is the same kind of CSS overlay either way). Every future keyboard-first feature on the webapp would then pay the cursor cost again.
- **Cursor-only (no hover upgrade)** — rejected. Mouse users see no improvement; the existing 1-pixel `cursor: pointer` is the entire affordance. Asymmetric work for asymmetric users when the implementation surface is the same `[data-tour-focus]` data attribute either way.
- **Per-file cursor scope (revdiff's model)** — rejected. Same reasoning as ADR 0011: the webapp diff pane is a continuous scrollbox of all `<FileDiff>` blocks, so the renderer's scope is already cross-file. Per-file would create a permanent mismatch between what the user sees (continuous stream) and what the cursor can address (one file). Sidebar selection in Tour today only acts as a jump-to-file trigger (not a viewport filter), so cross-file is also the simpler IA contract.
- **Auto-derive side from row type (drop `preferredSide`)** — rejected. Cannot target the deletions side of a `change` row in split layout — same reasoning as ADR 0011. The webapp's existing click-to-annotate already auto-derives side from `data-line-type` (and that's fine because click is per-cell), but the keyboard cursor lacks a column-level click; without `preferredSide`, paired rows default to additions and the deletions side becomes keyboard-unreachable.
- **Tab to swap side** — rejected. Avoids the `l → L` rebind, but loses TUI muscle-memory transfer and breaks the universal vim / less / man / gh pair `h`/`l` for horizontal motion. The one-time rebind cost is small pre-1.0.
- **`[` / `]` for side toggle** — rejected for the same `h`/`l` muscle-memory reasons.
- **Always-on cursor on tour load (ADR 0011's original TUI rule)** — rejected. The cursor and `currentAnnotationId`'s accent border would stack on the same row on every page load, creating two competing focus signals when the cursor wasn't asked for. Lazy materialization (cursor null until first j/k/h/l/arrows/a/n/p/click) keeps the first paint's focus on the existing `currentAnnotationId` accent and reveals the cursor only when the user invokes keyboard nav. The TUI ADR is amended in lockstep so both surfaces share one rule.
- **Symmetric coupling: `j`/`k` also moves `currentAnnotationId` (γ-coupling)** — rejected. Same reasoning as ADR 0011: accent flicker during code reading. On webapp, `currentAnnotationId` also drives a card border accent and the URL `?ann=` mirror — the URL would churn on every keystroke.
- **Symmetric coupling: `n`/`p` decouples (α-coupling)** — rejected. After `n` to a different file's annotation, pressing `a` would annotate the wrong file (cursor stayed in the previous file). Same break ADR 0011 documented.
- **TUI parity visual: full-row bg + `❯` gutter glyph** — rejected on web. ADR 0011 chose the bg+marker pattern after rejecting per-row borders because terminal cell-grid borders cause vertical layout shift (`<box borderStyle="single">` per row consumes 3 rows of vertical space for 1 row of content). On web, CSS `outline` is pixel-thin and consumes zero layout space, so the constraint that drove the TUI choice doesn't apply here. Tour's webapp also uses GitHub's syntax theme (`github-dark-default`); a GitHub-style outline preserves the established visual language. Each surface adopts its native idiom — same principle as keyboard-first vs mouse-first.
- **Pierre `lineAnnotations` as the cursor carrier** — rejected. Renders an "invisible focus marker" through Pierre's annotation pipeline. Reuses existing infra but couples cursor to annotation rendering, would break if Pierre's API churns, and pollutes the annotation count / sequence pill with synthetic entries.
- **URL persistence (`?cursor=<file>:<side>:<line>`)** — rejected as v1. Adds a second URL parameter to keep in sync, an interaction with `?ann=` (which wins on load if both are set?), and URL noise on every keystroke. Shareable deep-links to a specific line are real but better served by a future explicit "copy link to line" affordance triggered by the user, not by every cursor move. Reversible — can layer on later.
- **Cursor walks `+`/`-`/`change-*` only (skip context rows)** — rejected. Today's webapp click-to-annotate has this scope, but the TUI cursor walks all rows with line numbers. Mirroring the TUI scope keeps the surfaces consistent and removes a pre-existing webapp gap (clicking a context row today is silent — no error, no composer). Extending `findAnnotatableLine` to context rows in the same patch keeps cursor and click resolution identical.

## Consequences

- The webapp gains a first-class keyboard navigation primitive. Future features that need to address an arbitrary line (range select, hunk jumps, vim-motion-style counts, line-link sharing) can layer on without rebuilding their own targeting flow.
- TUI / webapp parity now goes through *behavior* and *data*, not just *mechanism*. Both surfaces share the cursor's anchor shape, β-coupling, lazy materialization, and side-selection keys. They differ only in visual idiom (TUI bg+marker, webapp GitHub outline) — by design, because each idiom is native to its surface.
- The `l → L` rebind on webapp is a one-time muscle-memory cost. Pre-1.0, small; post-1.0, part of the public webapp keymap contract. The same rebind landed on the TUI via ADR 0011.
- Click-to-annotate semantics expand to unchanged context rows. A reviewer who clicked a context line and saw nothing now opens the composer at that line. Side defaults to `additions` per CONTEXT.md convention. Aligns the webapp with the TUI cursor's row scope; surfaces if any downstream consumer relied on context-row clicks being silent (none known).
- The cursor composes for free with PRD #106 (Pierre `expandUnchanged: true`). Newly-revealed context rows from chevron expansion carry `[data-line]`, so the cursor walks them as soon as they appear. The semantic anchor `(file, lineNumber, side)` is invariant under DOM insertions; only the derived row index changes.
- Future webapp keyboard expansion of hidden context (chevron, file-collapse, analogous to PRD #107 for the TUI) is out of scope for v1. It would extend the cursor to walk Pierre's interactive rows, wire `Enter` / `Shift+Enter` to per-row-kind handlers, and inherit ADR 0013's cross-surface conventions. Tracked as a successor ADR; not implied by this one.
- Hover affordance and Pierre's own `:hover` styling can collide. Pierre may already paint a default `:hover` state on rows we target; implementation must verify the `[data-tour-hover]`-targeted CSS doesn't fight Pierre's defaults. Mitigation is straightforward (higher selector specificity, or use a CSS variable Pierre exposes); flagged here so the implementer isn't surprised.
- No on-disk artifacts. Cursor state is React `useState`; reload resets to null. Reversibility cost is the keymap rebind (`l → L`), identical to ADR 0011's reversibility posture.
- The webapp continues to rely on `<FileDiff>`'s click event for the mouse-first path. The cursor doesn't replace mouse interactions; it adds a parallel keyboard surface and an upgraded hover affordance, both backed by the same `[data-tour-focus]`-style row primitive.

## Revisions

### Context rows are addressable on both columns (reverses the "additions by convention" rule)

The original Decisions section forced `side = additions` for any cursor or click that landed on a context row. In split layout this produced an observable bug: a user clicks the **left** column of a context row, the cursor outline appears on the **right** column (the canonical side), and the click feedback lands far from the click point. Same line, same code, but the visual anchor jumped across the diff pane.

The "additions by convention" rule was a simplification, not a technical constraint. GitHub's split-view comment model treats the two columns of a context row as independent comment targets — click left → comment lives on `deletions`; click right → comment lives on `additions`. Users carry that muscle memory in.

**Revised rule:** `side` follows the column. For mouse: `findAnnotatableLine` walks the click path for the nearest `[data-deletions]` / `[data-additions]` ancestor and uses that column as `side`, including for context rows. For keyboard: the cursor's existing `preferredSide` (toggled by `h` / `l` / `←` / `→`) decides — `a` on a context row opens the composer on whichever column the cursor sits in. Buffer rows (Pierre's `[data-content-buffer]` / `[data-gutter-buffer]` spacers, where one side has more lines than the other) carry no `data-line` and remain inert — matching GitHub's gray non-clickable filler.

**Consequences of the revision:**
- One concept, not two: `side` is the column, period. The earlier proposal of splitting "visual column" from "anchor side" is unnecessary once the canonicalization goes away.
- `h` / `l` on a context row is no longer a silent no-op — it visibly moves the outline between columns, and a subsequent `a` follows.
- Existing context-row annotations on disk are all on `additions`; they keep rendering on the right column. New ones can land on either side. No migration.
- TUI parity: no TUI change. `cursorFromRow` in `core/cursor-state.ts` already resolves paired rows (which include context) by `preferredSide`, not by a forced `additions`. The webapp was the asymmetric one — the canonicalization lived in `findAnnotatableLine`'s click resolution, not in the shared cursor reducer. This revision pulls the webapp's click resolution into line with the TUI's already-correct behaviour.
- CONTEXT.md's **Side** glossary entry is updated to drop the "additions by convention" line.

**Considered alternatives for the fix (rejected):**
- *Outline jumps to the canonical side when `a` is pressed* — every `a` on a context row produces a surprising visual jump; the jump stops reading as information after the first few times and starts reading as twitch.
- *Outline stays on the clicked column, composer opens on the canonical side* — outline and composer end up in different columns; the cursor outline lies about where the comment will land.
- *Full-row outline + full-width composer for context rows* — visually communicates "context is one logical row," but doesn't match GitHub's mental model and introduces visual-rhythm changes as the keyboard cursor crosses between context and change rows.
