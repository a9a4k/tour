# Hidden context expansion

> **Status:** Webapp half ("Pierre owns the chevron UI, click handlers, expansion state; Tour writes no expansion code on the webapp") is superseded by ADR 0024. The webapp now renders gap rows natively as planner-emitted interactive rows and dispatches expansion to the same `core/expansion-state.ts` reducer the TUI uses — both surfaces converge on Tour-owned expansion state. The orphan-auto-window decision (±10 lines around hidden anchors) and the TUI cursor + `Enter` primitive are unchanged.

> **Scope:** webapp + TUI. The two surfaces converge on the same conceptual feature (reveal unchanged lines outside hunks, GitHub-style); historically via asymmetric mechanisms (Pierre's mouse-driven chevrons on the webapp, the cursor + `Enter` primitive on the TUI), now via a symmetric mechanism — gap rows are planner-emitted on both surfaces, click (web) and `Enter` (TUI + web cursor) dispatch the same action to the same reducer.

A reviewer in either surface can only see ±3 lines of unchanged context per hunk (git's `-U3` default). Code immediately around a change — function signatures, surrounding logic, file imports — is invisible without leaving Tour. Annotations whose anchor falls in **Hidden context** are silently dropped by the planner, breaking the sidebar count ↔ rendered diff agreement.

## Decisions

### Webapp: lean on Pierre

The webapp turns on `@pierre/diffs`'s built-in `expandUnchanged: true` with `expansionLineCount: 20` and ships per-side full file content (`oldContent` / `newContent`) eagerly in the tour bundle so Pierre can resolve unchanged lines synchronously on click. Pierre owns the chevron UI, the click handlers, and the per-instance expansion state. Tour writes no expansion code on the webapp.

### TUI: cursor + `Enter` is the unified primitive

The TUI's existing **Line cursor** is extended to walk a new family of **interactive rows** — hunk-separators, synthetic file-top / file-bottom boundaries, and classifier-collapsed file indicators. Pressing `Enter` while the cursor sits on any of these dispatches a row-kind-specific action: hunk-separator + boundary rows reveal `10` lines above + `10` lines below the boundary (`20` total, symmetric); collapsed-file rows expand the file's diff body. `Shift+Enter` reveals the entire gap. The cursor's annotation-anchor scope is unchanged — `a` (top-level annotate) is a no-op when the cursor sits on an interactive row.

### Orphan annotations: auto-windowed at planner-init

Annotations whose anchor falls outside any hunk get a `±10` line region around their anchor pre-expanded at planner-init time. Overlapping windows union. This holds on both surfaces — the same pure module (`core/orphan-window.ts`) computes the regions; the webapp passes them to Pierre via `expandedHunks`, the TUI seeds them into its `core/expansion-state.ts` reducer.

### State is per-renderer-session

Expansion state is in-memory only. Watcher-driven bundle reloads preserve it (the diff is SHA-pinned; gaps are unchanged). Tour switch and webapp page reload reset it. No localStorage, no on-disk persistence — same posture as **Layout** and the existing `collapsedOverrides`.

### TUI delivery sequenced behind cursor v2

The TUI cursor (#100, broken into #101–#105) is in flight when this ADR is written. The interactive-row extension (#107) is a strictly additive successor that the TUI hidden-context implementation (#108) depends on. The webapp implementation (#106) ships independently — Pierre's expansion has zero TUI/cursor coupling.

## Considered Options

- **Path B: own the affordance on the webapp** — emit our own chevron rows from `core/diff-rows.ts`'s planner, render them via custom DOM injection, hold expansion state in Tour, render Pierre as a dumb `FileDiff` over a pre-augmented `FileDiffMetadata`. Rejected: duplicates work Pierre already does, fights Pierre's shadow DOM, and gives near-zero payoff in cross-surface code sharing — the only state that would be "shared" is a per-hunk boolean, and the surfaces already diverge on layout state and cursor scope. ADR 0007's accepted Pierre coupling makes Path A the on-strategy choice.
- **TUI: cursor lands on hunk-headers** — extend the cursor's walkable scope to cover hunk-headers and use them as the single anchor for both navigation and side toggle (h/l) and expansion ([/]). Rejected: directly contradicts #100's design (story 32 — "cursor skips hunk-header rows"). The cursor's role in #100 is "where my next Annotation lands"; overloading it with multi-purpose actions modalizes the cursor (different keys do different things on different rows), which #100 explicitly rejected. The compromise — extend cursor scope to interactive rows but keep `a` no-op there — preserves #100's posture while gaining the navigation benefit. The interactive-row extension lives in a successor PRD (#107) so #100 itself isn't perturbed.
- **TUI: cursor-anywhere `[`/`]`/`{`/`}` keys** — four keymap entries: `[`/`]` for 20-line expansion at the gap closest to cursor (top vs bottom of gap), `{`/`}` for "expand all" variants. Rejected: four keys for one feature; "top vs bottom of gap" is a UX distinction users almost never make (~95% of expansions go cursor-outward); "nearest gap" tiebreak heuristics are fuzzy. The unified cursor + `Enter` model uses one key (Enter) for three row kinds, gives discoverable affordances on the rows themselves, and maps mouse click to the same action when mouse arrives later.
- **TUI: asymmetric `Enter` with motionDirection** — cursor accumulates a `motionDirection: 'up' | 'down'` from the last `j`/`k` motion; pressing `Enter` on a hunk-separator expands toward where the cursor came from (e.g. walked up to a header → expand toward the upper hunk's end; walked down → expand toward the lower hunk's start). Rejected in favour of symmetric 10+10. Reasons: motionDirection is stateful in a non-obvious way (users won't realize their walk direction matters); the field would need to be tracked, validated, reset on tour switch / fold / mouse-click; the asymmetric outcome only matters when a gap is larger than 20 lines AND the user has a directional preference — both rare. Symmetric expansion is "never wrong" and removes the field entirely, keeping the `Cursor` type smaller.
- **TUI: `Enter` expands the entire gap (no chunking)** — single-keystroke maximalism. Rejected: floods the screen on big gaps. The reviewer almost always wants iterative reveal; `Shift+Enter` covers the "give me everything" case at the cost of one modifier.
- **TUI: collapsed-file affordance on the file header (β)** — the file header itself becomes cursor-walkable when the file is classifier-collapsed; no synthetic indicator row. Rejected in favour of α (synthetic indicator row beneath the existing header). Reasons: keeps the file header's existing role (display + classification reason badge) intact; the synthetic indicator row is the same row shape as synthetic file-top / file-bottom boundary rows, so all interactive-row visuals share one renderer path; visual hierarchy is cleaner — header = identity, synthetic row = action.
- **Orphan annotations: pure user-driven (i)** — orphan annotations stay invisible until the user manually expands the right gap. Rejected: the sidebar count badge says "3 annotations" while the diff renders 2 — a silent lie. The reviewer would have to know to go hunting for the missing annotation by expanding gaps speculatively.
- **Orphan annotations: auto-expand on `n`/`p` (iii)** — when the reviewer navigates to an annotation that's currently hidden, the gap auto-expands. Rejected: surprise expansion at navigation time; the diff jumps as a gap opens; disorienting and breaks the predictable navigation contract from #100 (where `n`/`p` move the cursor to a specific row).
- **Orphan annotations: auto-expand the whole containing gap (ii-naive)** — at load time, find each orphan annotation and expand the entire gap that contains it. Rejected in favour of `±10` window: a 200-line gap auto-expanded for one annotation wastes screen real estate. The `±10` window is the smallest region that gives the annotation surrounding context without flooding.
- **Webapp keyboard parity** — add keyboard shortcuts in the webapp that mirror the TUI's `Enter` model. Rejected: the webapp has no cursor (per #100's deferral note), and adding one is a separate effort with its own ADR. The intentional cross-surface asymmetry — webapp mouse-driven, TUI keyboard-driven — is consistent with how `Layout` and `Line cursor` already differ between surfaces.
- **Lazy fetch of `oldContent`/`newContent` on click** — bundle stays small; webapp fetches per-file content on chevron click. Rejected: Pierre's `expandHunk` is a synchronous click handler; making it async would require intercepting the click, fetching ourselves, then forwarding. This throws away the simplicity that justified Path A. Bundle size is bounded — typical 10-file tours add 200–500 KB; classifier collapses the genuinely-huge files (lockfiles, generated, vendored).
- **Increase `git diff -U` to a giant number** — eliminate Hidden context entirely by including all unchanged lines in the original diff. Rejected: every file renders as one giant hunk; the reviewer can't see what's a change vs context at a glance. Defeats the "show only what changed by default" point that motivates GitHub's diff UI.

## Consequences

- The webapp gains GitHub-equivalent expansion UX with near-zero custom UI work. Pierre carries the visual + interaction load.
- The TUI gains a single `Enter` primitive that unifies "expand Hidden context" and "expand classifier-collapsed file" — two features under one gesture. Future row kinds (e.g. "load more annotations") can join via a new `subKind` without re-shaping the cursor or the keymap.
- The cross-surface asymmetry is now established as a **posture**, not an interim state: webapp uses mouse-on-Pierre, TUI uses keyboard-on-cursor. The same posture applies to **Line cursor** (#100) and to **Layout** (per-session, not synced). Reviewers switching surfaces don't lose the capability — only the gesture.
- The `Cursor` type gains an optional `interactive` discriminator (and no `motionDirection`). The cursor's primary role remains "where my next Annotation lands"; the interactive-row extension is strictly additive.
- Bundle JSON shape gains `oldContent`, `newContent`, and `orphanWindows` per file. Bundle size grows by the per-side file content × file count for non-binary, non-classifier-collapsed files. Tested under the assumption of typical local-127.0.0.1 single-user delivery; size cap deferred until shown to bite.
- Annotations on lines beyond a file's actual line count remain silently dropped (today's behaviour). The agent CLI (`tour annotate`) does not validate line bounds at submit time — flagged for a future hardening pass, not blocked here.
- Reversibility: removing webapp expansion is a 1-line change (drop `expandUnchanged: true`); the bundle's extra fields would become unused but harmless. Removing TUI expansion would unwind `core/expansion-state.ts`, the planner's three new row kinds, and the cursor's interactive-row extension — non-trivial but bounded. The most painful piece to revert would be the cursor's interactive-row scope, since downstream surfaces (mouse click handler, footer hint) carry its semantics.
- The CLI is unchanged. Agents writing annotations don't need to know whether their target line is in or out of any hunk — Tour's renderer handles visibility, the data model is unchanged. ADR 0003's "no re-anchoring" stance survives intact: Hidden context is a render-time concern, not a data-model concern.
- The TUI delivery sequence depends on cursor v1 (#100 / #101–#105) landing first, then the cursor-extension successor (#107), then the actual expansion + collapsed-file unification (#108). The webapp delivery (#106) is independent and ships first.
