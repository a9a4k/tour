# TUI per-cell row architecture with syntax highlighting

ADR 0007 chose a row-walking renderer on `@pierre/diffs` to get a slot for inline annotation cards between rows, accepting "loss of `<diff>`'s built-in syntax highlighting in v1." That row renderer rendered each diff row as a single `<text>` element holding both halves of a split-layout row in concatenated string form. With long context lines, the right-side line numbers drifted with `leftText` length and the right half wrapped under the left half — visible in real tours and incompatible with `wrapMode="word"`. Independently, the absence of syntax highlighting was the most-noticed UX gap vs the webapp.

We split each diff row into one `<box flexDirection="row">` containing per-cell `DiffLine` components — `width="50%"` for split, `width="100%"` for unified — and each cell's content goes through OpenTUI's `<code>` renderable, which feeds `tree-sitter` for highlighting. The architecture mirrors `DiffLineRenderable` from `anomalyco/opentui#621` (`SlottableDiff`) without depending on it: we stay on stock `@opentui/react` and use React composition to interleave `AnnotationCard` siblings between rows, replacing `SlottableDiff`'s `insertSlot()` API. The row planner in `core/diff-rows.ts` is unchanged — this ADR is scoped to the TUI render layer.

## Considered Options

- **`@opentui/core`'s stock `<diff>`.** Already rejected in ADR 0007 (no inline annotation slot between rows). The annotation-positioning constraint hasn't changed; cross-reference only.
- **Install PR #621 prebuilt artifacts via `pkg.pr.new`.** `npm i https://pkg.pr.new/anomalyco/opentui/@opentui/core@621` would give us `SlottableDiffRenderable` and per-line `DiffLineRenderable` immediately, with full syntax highlighting + virtualization + slot insertion. Rejected: the PR is `CONFLICTING` against upstream main, last commit Feb 2026, no review activity. Pinning to it couples our release cadence to a fork's PR lifecycle and blocks routine `@opentui/core` upgrades.
- **Vendor `DiffLineRenderable` + `SlottableDiffRenderable` + Diff.ts `private→protected` patch.** ~1.7k LoC of upstream-internal code under our control. Rejected: ongoing rebase burden as upstream OpenTUI churns; we'd diverge from canonical OpenTUI and own the maintenance. Worth revisiting only if PR #621 stalls long-term and we need its features (virtualization, click-to-comment).
- **Switch engine to Ink + Shiki.** Mature React-TUI ecosystem with an established syntax highlighter. Rejected: no native scrollbox, no first-class mouse-wheel, no built-in code/diff renderable. Re-architecting on Ink would regress strictly more than it gains.
- **Custom row-flex layout with per-cell `<code>` on stock OpenTUI.** Chosen. Same architectural shape as PR #621 (per-line cell renderable, row-flex container, `CodeRenderable` per cell), implemented as ~150 LoC in our codebase. Stock `@opentui/react@0.2.5` works unchanged; future upstream releases drop in cleanly.

## Consequences

- **Column drift and wrap stacking are gone.** Right-side line numbers anchor at the start of the right column box, independent of `leftText` length. Each cell's `wrapMode="word"` wraps inside its own column; row-flex sizes the row to `max(left, right)` height, so the next logical row starts on a clean break for both sides without padding tricks.
- **Syntax highlighting on diff content** for the languages whose `tree-sitter` grammars `@opentui/core` ships in `assets/`: ts, tsx, js, jsx, markdown. `src/tui/syntax.ts` whitelists this set and degrades gracefully to plain `<text>` for unsupported filetypes (no missing-grammar errors). The palette is a singleton `SyntaxStyle` (GitHub Dark) seeded once via `getSyntaxStyle()`.
- **Per-line parse, not per-file.** Each cell's `<code>` parses its single line in isolation, so multi-line constructs (template literals, JSX blocks, multi-line comments) lose cross-line context. This matches the trade-off `delta` operates under and is acceptable for a diff viewer where each visible line is already a leaf in the user's mental model.
- **Trailing-`\n` hygiene at the planner.** `@pierre/diffs` retains the line terminator on every entry in `additionLines` / `deletionLines`. OpenTUI's `<text>` and `<code>` (both extending `TextBufferRenderable`) honour an embedded `\n` as a hard line break and emit a phantom empty visual row per logical row — doubling the diff's vertical footprint. The fix is a single `stripTrailingNewline` pass in `core/diff-rows.ts`'s `walkHunks` so every `PlannedRow.leftText` / `rightText` is single-line. Centralized so future TUI consumers of the planner get clean text by default; the webapp consumes Pierre's `FileDiff` directly and is unaffected.
- **`<code>` and `<text>` are not direct flex children.** Both extend `TextBufferRenderable` and misreport their measured height when used as direct flex children — `<code>` inflates by a phantom row in row-flex; multiple `<text>` siblings in a column-flex can collapse onto the same buffer row, so a longer second sibling paints over the first and any spaces in the new content fail to clear the underlying chars (visible as the previous text leaking through space columns — first noticed as garbled "annotation header bytes showing through the body's first line" in `AnnotationCard`). Always wrap in `<box>` so flex sizing is owned by the box and the text-buffer renderable just fills it. Same pattern as anomalyco/opentui#621's `DiffLineRenderable._contentBox`. `DiffLine`'s `<code>` branch, `AnnotationCard`'s header/body, and any future text-buffer consumer must follow this.
- **Single-line gutter siblings need `alignItems="flex-start"` + explicit `height={1}`.** When a row's content cell wraps to multiple visual lines, the accent and line-number `<text>` elements (each one line tall) need to anchor at the top of the row. Yoga's default `alignItems="stretch"` makes them grow to the row's full height, and the rendered text drifts off the first visual line — line number ends up next to a wrap continuation instead of the row's start. Setting `alignItems="flex-start"` on `DiffLine`'s outer `<box>` plus `height={1}` (and `flexShrink={0}` for narrow-width safety) on the gutter `<text>`s pins them to the top regardless of how tall the row gets due to wrapping.
- **`DiffLine` honours ADR 0008's gutter-vs-content tint distinction.** The component takes `gutterTinted` and `contentTinted` separately; `DiffRows` sets `contentTinted` only when the row has line numbers on both sides (split, paired) or `type === "context"` (unified). The `+/−` structural signal survives on annotated change rows.
- **No virtualization.** Each visible row is a real renderable tree (3 boxes + 1 text + 1 code = 5 nodes per cell). At tour scale (dozens of files, ≲1k rows per file), this is fine; OpenTUI's `kommander` flagged the same concern on PR #621 and the response there was to add virtualization. Revisit if real tours expose layout perf as the dominant cost.
- **No word-level intra-line diff.** Pure layout change; orthogonal to delta-style intra-line emphasis. Adding it later is additive — a per-row pass on `change`-typed rows that diffs `leftText` vs `rightText` and applies span colors before `<code>` highlighting.
- **Reversibility is high, in both directions.** The renderer is ~150 LoC localized to `src/tui/{DiffLine,DiffRows,syntax}.tsx`. If PR #621 merges upstream and we want its features (virtualization, `insertSlot`, click handlers), swapping `DiffRows` for a `<slottable_diff>` invocation is a contained refactor — the row planner stays canonical and nothing in `core/` moves.
- **ADR 0007's "loss of syntax highlighting in v1" consequence is recovered**; that line is amended to cross-reference this ADR.
