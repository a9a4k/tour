# TUI line cursor for arbitrary-line annotation

> **Scope:** TUI-only. The webapp has no equivalent — Pierre's `FileDiff` already accepts click-to-position events that carry the side, so the webapp gets arbitrary-line targeting for free from a different mechanism. The TUI needs a keyboard-first path.

The TUI's top-level annotate flow (`a`) couldn't target arbitrary lines. Its anchor was either the currently-selected Annotation (if any) or `firstAnnotatableAnchor` — the first row with a line number in the selected file. Reviewers who wanted to comment on a specific line had to drop to the CLI (`tour annotate --file ... --line ...`). We add a persistent diff-pane line cursor to seed the composer.

## Decisions

The cursor is **persistent and always-on** when the diff pane is focused (not a modal "pick a line" mode that opens with `a`), **cross-file** (walks the entire concatenated diff stream rather than scoping to the sidebar-selected file), and carries an **explicit `preferredSide`** that the user toggles with `h`/`l` (rather than auto-deriving side from row type).

`n`/`p` annotation navigation moves the cursor to the target Annotation's anchor; `j`/`k` line motion does not move `currentAnnotationId` — annotation engagement and code reading are two distinct modes that shouldn't feed back into each other. Initial position seeds from the first top-level Annotation if any exist (preserving today's "land on first annotation" UX), else the first annotatable row of the first non-folded file. Bundle reloads from the watcher preserve cursor position. The visual treatment follows the **universal TUI focus pattern** — strong accent bg + leading `❯` glyph + bold line number, all confined to the active-side column — rather than literally porting GitHub's CSS overlay outline; this is the same pattern vim's `CursorLine`, lazygit, fzf, tig, gitui, and effectively every interactive TUI uses. Three cues, zero layout shift. The pre-existing global `l` binding for layout toggle is rebound to Shift-`L` to free the lowercase pair for cursor side selection.

## Considered Options

- **Modal / operator-pending cursor (vim's `c{motion}`)** — `a` opens a transient line-picker, j/k moves it, Enter commits. Rejected: smaller now (~150 LOC vs ~300) but doesn't reuse for any future diff-pane interaction (`d` to delete an annotation, range select, hunk jump), so any second feature pays the cursor cost again. Reviewer's mental model also asked for "a line cursor" as a thing that exists, not a mode that appears.
- **Per-file cursor (revdiff's model)** — cursor lives in the sidebar-selected file; `j` at file boundary stops. Rejected: Tour's diff pane is a single continuous scrollbox of all file cards, so the renderer's scope is already cross-file. A per-file cursor would create a permanent mismatch between what the user sees (continuous stream) and what the cursor can address (one file). It also forces a "current file" trilemma between sidebar selection, scroll position, and cursor position that only collapses if cursor is the single source of truth (the cross-file model).
- **Row-scoped cursor with auto-derived side** — cursor has only `(file, rowIdx)`; side is derived at annotate time per a fallback rule (additions on paired rows). Rejected: cannot target the deletions side of a `change` row in split layout, which is a legitimate review use case ("the old timeout value was tuned for case X — losing it will regress"). Constrained interpretation of "arbitrary line." Also breaks WYSIWYG: split layout renders two columns of equal weight, so the cursor should be able to address either; auto-derive masks half of what looks targetable.
- **Bidirectional cursor ↔ `currentAnnotationId` coupling (γ)** — `j`/`k` recomputes `currentAnnotationId` on every move so the accent border tracks cursor position. Rejected: makes the accent flicker during code reading (j/k is the dominant motion when reading code, and the user doesn't want their reading focus reset on every line move). Also breaks "I'm reading annotation A but positioning cursor on line X for an unrelated note" — pinning is the value.
- **Independent cursor and `currentAnnotationId` (α)** — neither moves the other. Rejected: after `n` to a different file's annotation, pressing `a` annotates the wrong file because cursor stayed put. `n` is inherently both annotation-nav AND code-nav; the cursor must follow it.
- **Tint + gutter-character overlay using the annotation accent** — reuse the annotation visual treatment for the cursor. Rejected: cursor often sits on annotated rows (initial position seeds from annotation 1), so the two would visually merge.
- **Caret glyph (`▶`) + soft gutter bg, gutter-confined** — two-cue overlay scoped to the line-number column. Rejected on the path to picking a richer visual: side selection wasn't visible enough (a 1-cell glyph swap is easy to miss when h/l is pressed), and a caret-shaped glyph reads as "typing here" rather than "focused here."
- **`<box borderStyle="single">` per-row outline wrap** — literal port of GitHub's CSS focus outline: wrap the active-column cell of the cursor row in a single-line border. Rejected after auditing how ~25 TUIs (vim, lazygit, fzf, tig, gitui, htop, k9s, ranger, vifm, mc, ncmpcpp, gh dash, Charm bubbletea apps, neovim, helix, revdiff, dialog/whiptail) handle per-row focus — **none of them** use box-drawing borders around individual rows. The reasons cohere: (1) CSS overlay borders are pixel-thin and don't consume layout space, but TUI cell-grid borders are minimum 1-cell-thick glyphs; (2) a 3-line outline around 1 line of content reads as chunky/heavy at terminal cell density; (3) cursor moves cause localised vertical shift (cursor row temporarily becomes 3 lines tall, old row collapses by 2 when moved away from). The semantic concern that motivated the outline ("focus, not typing-cursor") is fully solved by the universal TUI bg+marker convention — vim's `CursorLine`, lazygit's selected row, fzf's match indicator all read as "focus" without an outline. The literal-vs-semantic conflation was the bug in the original plan.
- **Half-block left/right edge characters (`▏`/`▕`) + bg fill** — closest TUI approximation to GitHub's thin-line aesthetic without vertical layout shift. Half-block characters render as one-eighth-cell-wide vertical bars on column edges, looking thinner than `│`. Rejected as v1 because the universal bg+glyph pattern is sufficient and discoverable; kept as a reachable v2 refinement (~10 LOC delta over the chosen approach) if user feedback shows the bg-only treatment doesn't read as "outlined" enough. Top/bottom edges via `▔`/`▁` are not practical — they'd require modifying adjacent rows' content and break when adjacent rows are themselves special (annotation cards, hunk-headers).
- **`[`/`]` for cursor side, keep `l` as layout** — non-breaking. Rejected: `h`/`l` is the universally-known horizontal pair (vim, less, man, gh) and the muscle-memory cost of `[`/`]` over a Tour-pre-1.0 lifetime is higher than the one-time disruption of moving layout to Shift-`L`.

## Consequences

- The TUI gains a first-class navigation primitive that future features (delete-annotation-under-cursor, range select, hunk jumps, vim-motion-style counts) can layer on without rebuilding their own targeting flow.
- TUI/webapp parity now goes through *behavior*, not *mechanism*: both surfaces support arbitrary-line annotation, but the TUI uses a keyboard cursor and the webapp uses Pierre's click-to-position. Reviewers switching surfaces don't lose the capability, only the gesture.
- The β-coupling (n/p couples both, j/k decouples) introduces a deliberate asymmetry between the two cursor-like things in the TUI. Documented in CONTEXT.md and in the in-code keymap so future contributors don't "fix" it.
- The `l` → `L` rebind is a one-time UX disruption for current users. Pre-1.0, the cost is small; post-1.0, this becomes part of the public keymap contract.
- Visual treatment is conventional, not aspirational. The TUI looks and behaves like any other interactive terminal review tool — that's a feature, not a limitation, because the convention is universal enough that reviewers from vim/lazygit/fzf will read the cursor correctly on first sight without learning a Tour-specific idiom.
- Cursor bg replaces annotation tint on the cursor row when the cursor sits on an annotated line. The annotation `▎` gutter mark continues to render so the "this row is annotated" signal isn't lost; only the bg layer is overridden. Composition rule: `cursor bg > annotation tint > +/- bg > syntax bg`.
- The webapp does not gain a cursor. Its Composer continues to seed from Pierre's click event. Adding a cursor on the webapp later would need a separate ADR — the keyboard-first vs mouse-first division is a feature, not a temporary asymmetry.
- Reversibility: removing the cursor would regress the arbitrary-line capability and restore the CLI-only fallback. The `l`/`L` rebind would be the most painful piece to revert (muscle memory), but the cursor's data model leaves no on-disk artifacts — purely UI state.

## Revisions

**2026-05-10 — Lazy materialization (during ADR 0012 design grilling).** Initial-position rule changed: cursor is null on tour load; first user interaction (`j` / `k` / `h` / `l` / arrows / `a` / `n` / `p` / click) materializes the cursor at the prior default target — first top-level Annotation if any exist, else first annotatable row of the first non-folded file. The materialization target is unchanged; only the trigger moves from "tour load" to "first interaction." `a` on a null cursor materializes at the default target AND opens the composer in one step (preserves today's "annotate first row of first file" fallback). Rationale: reduces first-load visual noise (the cursor's bg highlight no longer stacks on the same row as `currentAnnotationId`'s accent border on first paint), preserves the "land on first annotation" UX (the annotation accent border is still the eye-catcher on load), and unifies the surface-parity story with ADR 0012's webapp cursor — both surfaces start null and materialize at the same target on first interaction. PRD #100 and slice #101 to be amended in lockstep with this revision.

**2026-05-10 — Diff-pane motion contract.** The cursor floats in the document (vim / helix / VS Code / IntelliJ / emacs model — *Equilibrium 2*); the diff pane is a passive follower of cursor motion, not a co-actor. `j` / `k` move the cursor only — the pane scrolls **one row at a time** iff the cursor would otherwise cross a **3-row edge margin** (vim's `scrolloff` default). `Space` / `Shift-Space` and the hardware `PageDown` / `PageUp` keys move both: pane scrolls one viewport AND the cursor advances so its **screen-relative position is preserved**, so the keyboard's home base stays usable (pressing `a` after `Space` annotates a visible line, not a stranded one). `Home` / `End` jump the cursor to the first / last cursor-eligible row in the concatenated diff stream and the pane scrolls to match. Mouse wheel is pure pane motion — the cursor stays anchored to its file line; if it leaves the viewport, the next `j` / `k` snaps the pane back to it via `block:nearest`, keeping keyboard motion "magnetic" to the cursor. The keyboard-vs-wheel asymmetry is intentional and reflects two input modalities preserving their respective home bases (cursor for keyboard, pointer for mouse); every floating-cursor document tool surveyed (vim, helix, VS Code, IntelliJ, emacs, less) makes the same choice. List-view tools that pin the cursor to a fixed screen row (lazygit lists, fzf, k9s, htop) don't have the asymmetry by construction, but Tour's diff pane is a document, not a list. `n` / `p` annotation jumps recenter the target card unconditionally (`block:center` with `block:start` fallback for cards taller than the viewport), aligning the TUI with the webapp's `scrollIntoView({ block: 'center' })` — `CONTEXT.md`'s prior *"mirrors the webapp"* claim becomes accurate rather than aspirational. Implementation: the `focused={…}` prop is removed from the diff `<scrollbox>`; the React keymap is the single source of truth for diff-pane keys. OpenTUI's built-in `ScrollBar.handleKeyPress` had been firing in parallel with the React keymap (because the scrollbox was `focused`), producing the parallel-handler bug where each `j` / `k` keystroke moved the cursor **and** scrolled the pane by 1/5 viewport simultaneously — same posture as the existing `autoFocus: false` decision in `startTui`, extended one boundary further. Rejected: **cursor-pinned-to-screen** (Equilibrium 1 — code-review needs context, treadmill scrolling destroys it); **pure decouple with no auto-scroll** (cursor disappears off-screen, user must manually find it); **selective `preventDefault`** (brittle — couples Tour's behavior to OpenTUI's internal key list, requires updates whenever a new ScrollBar key is added upstream); **`Ctrl-D` / `Ctrl-U` half-page chord keys** (breaks the no-modifier single-key contract for marginal benefit; reachable as a 1-line follow-up if vim users push back). The 3-row scrolloff value and the full-page step on `Space` are UX tuning knobs, not structural commitments — half-page on `Space` (a 1-line change) and adding `Ctrl-D` / `Ctrl-U` are both compatible follow-ups if user feedback demands them.

**2026-05-10 — Half-page Space / `b`, full-page hardware PageUp / PageDown (PRD #138, issue #139).** Exercising the "half-page on Space" tuning knob the prior entry called out as a compatible follow-up. `Space` and `Shift+Space` now step a **half** viewport (`Math.max(1, Math.floor(viewportHeight / 2))`); `b` (no modifier) is added as the portable page-up alias paired with Space — the Unix mainline pairing from `more(1)` (1978) onward, used by `less` / `man` / `vim` (pager) / `tig` / `delta`. Hardware `PageUp` / `PageDown` keep the **full** viewport step (their cross-application "jump a screen" expectation; opt-in escape hatch when the user wants the bigger jump). The 1-row clamp preserves the "Space always moves" invariant from the prior entry on tiny viewports. Two motivations: (1) Shift+Space arrives as plain `0x20` on macOS Terminal.app and default-config iTerm2 (no shift bit because neither negotiates the kitty CSI-u or `xterm modifyOtherKeys` protocols), so the documented page-up keystroke silently fell through to plain Space — `b` is the portable fix that works on every terminal because it's a single ASCII byte; (2) full-viewport jumps destroy visual context every press in a code-review setting where the reviewer is tracking commentary near specific lines. Half-page paging preserves ~half a screen of overlap, pairing with the cursor screen-Y preservation invariant to give a vim-`Ctrl+D`/`Ctrl+U`-style review flow. Modal precedence and sidebar parity rules are unchanged: composer / picker focus short-circuit before `dispatchKey` is consulted (literal `b` reaches a composer input), and `b` works whether the sidebar or the diff pane is focused — same rule that already protects `Space`. `Ctrl+b` and `Shift+B` (capital) remain unbound by paging so future Ctrl-chord bindings stay available without collision. The `Shift+Space` binding is intentionally kept (harmless on legacy terminals, free on protocol-aware ones); removing it would only reduce capability where the protocol works. Implementation shape: `dispatchKey`'s `KeyAction` union splits the 2 paging variants into 4 (`page-diff-{up,down}` for full / hardware keys, `half-page-diff-{up,down}` for Space / `b` / Shift+Space) — flat, no payload on actions, matching the existing discriminated-union shape. `pageMove` takes `step: "half" | "full"`; all other invariants (doc-fits-viewport no-op, screen-Y preservation, bound-snap to first/last cursor-eligible row, `cursor === null` short-circuit, reference equality preserved on no-op) are unchanged. Webapp paging is out of scope (browser native scroll handles `Space` / `Shift+Space` at the OS level; ADR 0012 governs web cursor independently).

**2026-05-11 — Cursor pop: full-row bg, glyph `▶`→`❯`, bold cue dropped, token renamed.** The shipped cursor visual (`▶` glyph + soft gutter-confined bg at `#112441`) read as too subtle next to diff +/- backgrounds — eye-tracking against the web's outlined-row equivalent showed the TUI cursor was hard to locate. Three changes resolve the gap. (1) **Full-row bg.** `cursorActive` now paints the cursor bg on the content cell as well as the gutter cell, making the active row a single solid plate. This is the terminal-native analogue of the web's outlined focus row: terminals can't render outlines without taking extra rows or competing with box-drawing characters, but a saturated solid fill carries the same "this row is the subject" semantics. In split layout the fill is naturally side-scoped (each side is its own `DiffLine` at width=50%, and `cursorActive` is per-side), preserving the ADR's "active-side column" intent; in unified the row has one column so it fills the whole row. The composition rule from the original Decisions section is unchanged: `cursor bg > annotation tint > +/- bg > syntax bg`. (2) **Brighter token.** `theme.bg.cursorRow.tui` bumped from `#112441` (≈20% accent on canvas) to `#1a3566` (≈30%) so the fill reads as a clear blue plate rather than a faint wash; web alpha tracks at `rgba(31, 111, 235, 0.30)` even though no web selector currently reads `--bg-cursor-row` (parity preserved for any future consumer). The token was also renamed `bg.cursorGutter` → `bg.cursorRow` (and `--bg-cursor-gutter` → `--bg-cursor-row`) because the name "gutter" no longer describes what it paints — `cursorRow` is now the honest name. (3) **Glyph `▶`→`❯`.** The chevron form reads as "focus / pointing at" rather than "play / typing-cursor" — same connotation distinction that motivated rejecting the caret-shaped glyph in the original Considered Options. (4) **Bold cue dropped.** The original Decisions text listed three cues — strong bg + leading glyph + bold line number — but the full-row bg at `#1a3566` is doing enough work that bold became visual noise rather than emphasis; bold on the tree-sitter-coloured `<code>` body would also have required rebuilding `SyntaxStyle` per cursor row, which is not worth the implementation cost for a redundant cue. The cursor now reads as two channels (bg + glyph), still independent enough to survive colour-blindness and muted themes. Rejected alternatives during the tuning: left-edge accent stripe (collides with the annotation-range `gutterAccent` stripe — same column, same colour, ambiguous when cursor sits in or near a range); pushing bg past `accentEmphasis` (`#1f6feb` solid hurts text legibility against `fg.default`); reintroducing bold on plain text only (gutter-only bold is a minor cue overshadowed by the bg and creates an inconsistent "bold here but not in code" visual). Test impact: `tests/tui/diff-line.test.ts`'s "cursor visual is gutter-confined" assertion is flipped to assert `contentBg === CURSOR_ROW_BG` and overrides `diffBg` / `contentTinted`; theme test updated to the new colour + alpha + var name. ADR 0018's `▶` glyph mention in TUI rendering text is updated; the `▶` reference in this ADR's Considered Options stays because it describes the *rejected* prior visual.

**2026-05-13 — On-load materialization restored for non-empty tours (issue #256).** Reverts the 2026-05-10 "Lazy materialization" revision for the non-empty-tour path: on tour load, when at least one top-level Annotation exists, the cursor materialises on `topLevel[0]` as a `CardAnchor` and the cursor-follow effect scrolls the card into view — same surface contract the webapp gets from ADR 0022's URL-anchored mount. The lazy-materialization rule survives for the empty-tour and snapshot-lost paths (no target to seed; cursor stays null and the next user interaction promotes it via `initialCursor`'s row-anchor fallback). Two of the original revision's grounds no longer hold: ground 1 ("surface parity — both surfaces start null and materialise at the same target on first interaction") was broken when ADR 0022 introduced URL-anchored mount on the webapp (the webapp now materialises the cursor at `?ann=` or `topLevel[0]` on mount unconditionally); ground 2 ("preserved 'land on first annotation' UX — the annotation accent border is still the eye-catcher on load") only delivered when the first annotation sat inside the initial viewport — for any tour where the first annotation sits later in the first file or in a later file, the eye-catcher was below the fold and the user saw the top of the first file with no annotation in view. Ground 3 ("reduced first-load visual noise — the cursor row plate would otherwise stack on the annotation accent border") applied to *row* cursors; the cursor that lands on a top-level annotation under ADR 0022 is a `CardAnchor`, not a `RowAnchor`, and card cursors don't paint a row-spanning bg plate, so the stacking concern doesn't apply to the actual landing state. Implementation: a `cursor.materialize` dispatch in the App-shell's existing tour-open `useEffect` (the same first-paint-per-tour guard that already drops sidebar focus and reveals the first annotation's file in the tree) with `initialCursor({ topLevelAnnotations, flatRows })` as the seed; same-tour `bundle.refreshed` is suppressed by the `seededTourIdRef` guard so user motion taken before a watcher reload survives, and the reducer's `cursor.materialize` is a strict no-op on a non-null cursor as a belt-and-suspenders fallback. PRD #192 US 25 is amended in lockstep.
